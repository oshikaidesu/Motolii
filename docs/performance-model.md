# 性能モデル: 何が重いのか

作成日: 2026-07-07

「AEはなぜ重いのか/このツールはなぜ軽くできるのか」を推測ではなく処理コストの構造で説明する。
実装上の性能判断(プレビュー品質、パス設計、精度選択)はこのモデルを前提にする。

## 結論の先出し

コンポジットの支配的コストは**演算ではなくメモリ帯域**であり、「解像度」そのものではなく
**「1フレームぶんのピクセルを、何回、どの速さのメモリで往復させるか」**が重さを決める。
同じ解像度でもAEと同じにはならない — AEはその往復を最も遅い経路(CPU RAM、しかもCPU⇔GPU転送込み)で
何度も払っており、うちはそこを構造的に払わない設計にしている。

## コストの内訳(重い順)

### 1. メモリ帯域(最重要)

- 1080p RGBA8 = 約8.3MB/フレーム、4K = 約33MB。エフェクトを1つ通すたびに全ピクセルの読み+書きが発生する
- エフェクト10個をナイーブに直列適用すると、フレームデータがメモリを10往復する
- CPU RAMの帯域は約50GB/s、GPU VRAMは400〜1000GB/s。**同じ処理でもピクセルの置き場所で1桁以上違う**
- AE(伝統的パイプライン)はCPU RAM上のバッファで、しかも16/32bpc(2〜4倍のバイト数)で往復する
- → 対策(採用済み): ピクセルは常にwgpuテクスチャとして**VRAMに常駐**させ、CPUに下ろさない

### 2. CPU⇔GPU転送(PCIe)

- PCIe帯域は10〜25GB/s程度で、しかも転送はパイプラインを止めやすい
- AEの本当の泥沼はここ: プラグイン資産がCPU前提のため、GPU対応エフェクトの後にCPUエフェクトが
  挟まると**フレーム全体がVRAM→RAM→VRAMとバウンス**する。エフェクトスタックの並び次第で転送が量産される
- → 対策(採用済み): エフェクトプラグインの境界を最初から「wgpuテクスチャ入力→wgpuテクスチャ出力」に
  固定した(concept.md)。CPUバウンスするエフェクトという概念自体が存在しない

### 3. 中間バッファの確保とコピー

- エフェクトごとに新規バッファ確保+コピーをすると帯域に加えてアロケーションコストも払う
- → 対策(M1-T8実装方針): ピンポンバッファ(2枚を交互に使い回す)。将来の最適化として
  単純なエフェクト連結のパス融合(1シェーダに複数エフェクトを合成)の余地を残す

### 4. 動画デコード

- H.264/HEVCのlong-GOP素材はシークのたびに「キーフレームから読み捨てデコード」が走る(B-2)
- → 対策(M4計画済み): インポート時プロキシ/全Iフレーム中間コーデック化

### 5. ピクセル精度(ビット深度)

- fp32中間バッファはRGBA8の4倍のバイト数=4倍の帯域を食う
- → 方針: 中間はfp16を既定(リニア合成に十分)、最終レンダのみ必要に応じてfp32。
  8bitで済む箇所(ソース、最終出力)は8bitのまま

### 6. 並列性の歴史的制約(AE固有)

- AEのレンダパイプラインは長らくフレーム内実質シングルスレッドで、マルチフレームレンダリングは
  後付け(メモリ大食い)。GPUコンピュートはこの問題クラス自体を持たない

## 「解像度だったらAEと変わらない」への答え

解像度は**仕事量の下限**を決めるだけで、重さは「下限×往復回数×メモリ経路の遅さ」で決まる。
AEはこの乗数部分が大きい。同じ4Kでも、VRAM常駐+fp16+転送ゼロのパイプラインは
CPU往復+fp32+PCIeバウンスのパイプラインより**1〜2桁速い**。うちが縮められるのは乗数であり、
そこがこのプロジェクトの軽さの根拠。逆に言えば、**VRAM常駐を破る設計変更(安易なCPU処理の混入)を
1箇所入れるだけで優位性は消える**。レビュー時はそこを最重視すること。

## プレビュー品質モード(コンセプト決定 2026-07-07)

「多少の色の違いには目をつぶり、最終レンダリングだけ厳密なら良い」を仕様化する。
プレビューと最終レンダは同一の`render_frame(t, Quality)`を通る(B-4)ため、差は`Quality`の値だけ:

| 項目 | Draft(既定プレビュー) | Final(書き出し/最終品質プレビュー) |
|---|---|---|
| 解像度スケール | 1/2(重い時1/4に自動段階降格) | 1/1 |
| 中間精度 | fp16固定 | fp16(必要ノードのみfp32) |
| 色変換 | 近似可(sRGB空間ブレンド等のショートカット許容) | 厳密(リニア合成) |
| エフェクトサンプル数 | 各エフェクトのdraft値 | full値 |
| グレイン等の最終質感 | スキップ可 | 必須 |

- UIにモード切替を置き、「最終品質でプレビュー」も選べる(重くなるのは承知の上で確認用)
- ゴールデンイメージテストはFinal品質に対してのみ厳密比較を行う。Draftは「クラッシュせず出る」ことのみ保証

## GPUを効率よく使うための原則

「GPUを使う」だけでは速くならない。GPUが速いのは**大量の独立したデータに同じ処理を並列適用する**時だけで、逆にそこを外すと簡単にCPUより遅くなる。うちが守る原則:

1. **往復を減らす > 演算を減らす**: 前述の通り支配コストはメモリ帯域。エフェクトを個別パスで順に適用するより、単純な連鎖は1シェーダに融合(パスフュージョン)して中間書き出しを消す方が効く。中間バッファはピンポン2枚を使い回す(確保・解放を毎フレームやらない)
2. **CPU⇔GPUの同期点を作らない**: `download`(GPU→CPU読み戻し)やバッファmapはGPUパイプラインを止める(ストール)。プレビュー表示はGPUテクスチャを直接画面に出し、読み戻しは書き出しとテストだけに限る。「1フレームごとに結果をCPUで確認」は最悪パターン
3. **ディスパッチ回数を減らす(バッチング)**: 小さな描画を大量に発行するとGPUが遊ぶ。同種オーバーレイ(大量の図形)はインスタンシングで1ディスパッチにまとめる。ドローコール数はフレーム時間に直結する
4. **分岐と可変ループを避ける**: GPUは32/64スレッドが同じ命令を歩調を合わせて実行する(SIMT)。シェーダ内の`if`でスレッドごとに道が分かれると両方の道を実行して捨てる(ダイバージェンス)。エフェクトのパラメータ分岐はできるだけシェーダ生成時(コンパイル時)に解決し、実行時分岐にしない
5. **精度は必要最小限**: fp16はfp32の半分の帯域で、GPUによっては2倍のスループット。中間はfp16既定、8bitで足りる入出力は8bitのまま(前述)
6. **非同期に走らせCPUを遊ばせない**: デコード(ffmpegサイドカー)・解析・レンダを別ステージにしてパイプライン化。CPUが次フレームをデコードする間にGPUが現フレームを描く

### 3Dに固有の効率化(M5の動的シミュレーション不採用と接続)

- 3Dを「時刻tの純関数」に限定した(M5決定)ことで、**フレームは互いに独立** = 先読み(prefetch)・並列ベイク・区間キャッシュが全部効く。前フレーム依存のシミュレーションを持ち込むと、この独立性が壊れてキャッシュもパイプライン化も無効になる — だから持ち込まない
- 静的/決定論的メッシュは**頂点バッファをGPUに置きっぱなし**にできる(毎フレーム転送しない)。動くのはカメラ・トランスフォーム行列(数十バイトのuniform)だけ。これがシミュレーション系との決定的な差で、うちの3Dが軽い理由
- プレビューでカクついてよい部分は、Draft品質(低ポリLOD・低解像度レンダターゲット)で回し、確定後にM4の区間ベイクで焼く。3D専用のキャッシュ機構は作らず、2D/エフェクトと同じキャッシュに相乗りする
