# 性能モデル: 何が重いのか

作成日: 2026-07-07

「AEはなぜ重いのか/このツールはなぜ軽くできるのか」を推測ではなく処理コストの構造で説明する。
実装上の性能判断(プレビュー品質、パス設計、精度選択)はこのモデルを前提にする。

## 結論の先出し

コンポジットの支配的コストは**演算ではなくメモリ帯域**であり、「解像度」そのものではなく
**「1フレームぶんのピクセルを、何回、どの速さのメモリで往復させるか」**が重さを決める。
同じ解像度でもAEと同じにはならない — AEはその往復を最も遅い経路(CPU RAM、しかもCPU⇔GPU転送込み)で
何度も払っており、うちはそこを構造的に払わない設計にしている。

## コストの内訳(重い順)

### 1. メモリ帯域(最重要)

- 1080p RGBA8 = 約8.3MB/フレーム、4K = 約33MB。エフェクトを1つ通すたびに全ピクセルの読み+書きが発生する
- エフェクト10個をナイーブに直列適用すると、フレームデータがメモリを10往復する
- CPU RAMの帯域は約50GB/s、GPU VRAMは400〜1000GB/s。**同じ処理でもピクセルの置き場所で1桁以上違う**
- AE(伝統的パイプライン)はCPU RAM上のバッファで、しかも16/32bpc(2〜4倍のバイト数)で往復する
- → 対策(採用済み): ピクセルは常にwgpuテクスチャとして**VRAMに常駐**させ、CPUに下ろさない

### 2. CPU⇔GPU転送(PCIe)

- PCIe帯域は10〜25GB/s程度で、しかも転送はパイプラインを止めやすい
- AEの本当の泥沼はここ: プラグイン資産がCPU前提のため、GPU対応エフェクトの後にCPUエフェクトが
  挟まると**フレーム全体がVRAM→RAM→VRAMとバウンス**する。エフェクトスタックの並び次第で転送が量産される
- → 対策(採用済み): エフェクトプラグインの境界を最初から「wgpuテクスチャ入力→wgpuテクスチャ出力」に
  固定した(concept.md)。CPUバウンスするエフェクトという概念自体が存在しない
- → 実装メモ(2026-07-08): `motolii-plugin`のFilter/Composite契約は`wgpu::Texture` + `wgpu::CommandEncoder`
  だけを受け取る。CPU参照実装はゴールデンテストの正しさ基準に限定し、製品プラグイン境界には入れない

### 3. 中間バッファの確保とコピー

- エフェクトごとに新規バッファ確保+コピーをすると帯域に加えてアロケーションコストも払う
- → 対策(M1-T8実装方針): ピンポンバッファ(2枚を交互に使い回す)。将来の最適化として
  単純なエフェクト連結のパス融合(1シェーダに複数エフェクトを合成)の余地を残す

### 4. 動画デコード

- H.264/HEVCのlong-GOP素材はシークのたびに「キーフレームから読み捨てデコード」が走る(B-2)
- → 対策(M4計画済み): インポート時プロキシ/全Iフレーム中間コーデック化

### 5. ピクセル精度(ビット深度)

- fp32中間バッファはRGBA8の4倍のバイト数=4倍の帯域を食う
- → 方針: 中間はfp16を既定(リニア合成に十分)、最終レンダのみ必要に応じてfp32。
  8bitで済む箇所(ソース、最終出力)は8bitのまま

### 6. 並列性の歴史的制約(AE固有)

- AEのレンダパイプラインは長らくフレーム内実質シングルスレッドで、マルチフレームレンダリングは
  後付け(メモリ大食い)。GPUコンピュートはこの問題クラス自体を持たない
- **AEが30年マルチコア化に苦しんだ根因と、うちの答え(2026-07-08)**: AEのエフェクトAPIは
  グローバル可変状態と逐次実行を前提に固まっており、後からフレーム並列を入れると
  race/メモリ複製爆発になる(MFRが後付けでメモリ大食いなのはこのため)。
  このプロジェクトでは並列性は「後で足す機能」ではなく、**既存の3決定の帰結**として最初から成立する:
  1. **フレーム=時刻tの純関数**(B-4/M5決定) → フレーム間に依存が無く、複数フレームを
     複数スレッドで同時にレンダしてよい。先読み・並列ベイク・書き出しのパイプライン化が全部この上に乗る
  2. **単一writer+不変スナップショット**(F-2) → 読み手(レンダ・解析・書き出し)は`Arc<Document>`を
     持つだけでロック無しに並走できる。Natron型のrace/deadlockが構造的に起きない
  3. **役割分離**: フレーム内の画素並列はGPUの仕事、CPUコアは「フレーム間並列」
     (デコード・先読み・ベイク・エンコード)に使う
- この帰結を守る条件はただ1つ: **プラグインに隠れた可変状態を持たせない**(出力が時刻tと入力だけで
  決まる純関数契約)。逐次前提のプラグインAPIを1つ通した瞬間、AEと同じ「後付け不能」に戻る。
  プラグインABI凍結時のレビュー最重視項目とする(VRAM常駐・色変換一元化と並ぶ第3の規律)
- **補足(2026-07-10)**: この規律が掛かるのは**レンダ経路**であり、物理シミュレーションの禁止ではない。
  逐次状態が要る表現(布・液体・パーティクル)は、レンダ経路の外のホスト管理ベイク境界
  (`SimulationPlugin`+StateTrack)で走らせ、`render_frame(t)`はベイク結果を読む純関数のままにする —
  フレーム並列はそのまま守られる。設計は[simulation-model.md](simulation-model.md)

### 7. 同時レイヤー数の設計目標(2026-07-08)

AEクローンとして**1シーン40本の動画レイヤー(1080p想定)が同時にアクティブでも破綻しない**ことを設計目標とする。

- VRAMは問題にならない: ソース40×8.3MB≈330MB + fp16中間ピンポン2枚(**レイヤー数に依存しない**)で計~400MB
- 合成帯域も成立: 40層fp16で約1.7GB/フレーム、30fpsで~50GB/s。ディスクリートGPUで1割程度、
  iGPUはDraft 1/2で1/4になり収まる
- PCIe転送はYUVのままアップロード(YuvToRgba方式)で40本×3.1MB×30fps≈3.7GB/s。PCIe 3.0の1/4で可。
  RGBAでアップロードする設計に変えると2.7倍に膨らむ — YUVアップロードは性能規律の一部
- 支配コストは**デコード**(40本同時デコードが壁)。対策はB-2追記の3点:
  全Iフレームプロキシ / アクティブレイヤーのみデコード / デコーダプール(上限+先読み)
- **この試算は下限(floor)として読む**(2026-07-09): 先読みバッファ・エフェクトスタック帯域・キャッシュ占有は未計上。
  注記と、容量・メモリ階層(VRAM/RAM/ディスクの役割分担)の議論は[memory-model.md](memory-model.md)に一元化する

## 「解像度だったらAEと変わらない」への答え

解像度は**仕事量の下限**を決めるだけで、重さは「下限×往復回数×メモリ経路の遅さ」で決まる。
AEはこの乗数部分が大きい。同じ4Kでも、VRAM常駐+fp16+転送ゼロのパイプラインは
CPU往復+fp32+PCIeバウンスのパイプラインより**1〜2桁速い**。うちが縮められるのは乗数であり、
そこがこのプロジェクトの軽さの根拠。逆に言えば、**VRAM常駐を破る設計変更(安易なCPU処理の混入)を
1箇所入れるだけで優位性は消える**。レビュー時はそこを最重視すること。

## プレビュー品質モード(コンセプト決定 2026-07-07)

「多少の色の違いには目をつぶり、最終レンダリングだけ厳密なら良い」を仕様化する。
プレビューと最終レンダは同一の`render_frame(t, Quality)`を通る(B-4)ため、差は`Quality`の値だけ:

| 項目 | Draft(既定プレビュー) | Final(書き出し/最終品質プレビュー) |
|---|---|---|
| 解像度スケール | 1/2(重い時1/4に自動段階降格) | 1/1 |
| 中間精度 | fp16固定 | fp16(必要ノードのみfp32) |
| 色変換 | 近似可(sRGB空間ブレンド等のショートカット許容) | 厳密(リニア合成) |
| エフェクトサンプル数 | 各エフェクトのdraft値 | full値 |
| グレイン等の最終質感 | スキップ可 | 必須 |

- UIにモード切替を置き、「最終品質でプレビュー」も選べる(重くなるのは承知の上で確認用)
- ゴールデンイメージテストはFinal品質に対してのみ厳密比較を行う。Draftは「クラッシュせず出る」ことのみ保証
- **前提条件(2026-07-08、落とし穴F-1)**: 「Draft=半解像度でも見た目の分岐なし」の約束は、**全空間パラメータ(ブラー半径・位置・サイズ等)が解像度相対(正準座標系)で持たれている場合にのみ成立する**。ブラー半径を絶対pxで持つとDraftとFinalで効き方が2倍変わり約束が破綻する(Blender compositorが「relative space」改修で苦しんでいる前例)。px値をパラメータに持つノードはレビューで禁止

## 伝達関数(ガンマ)のv1ポリシー(2026-07-08 決定、レビュー指摘#4)

「リニア化は後段の責務」だけでは、**どの曲線でリニア化するか**が未規定だった。v1は以下に固定する:

- **内部の8bit「ガンマ保持RGBA」はsRGB曲線として解釈する**。リニア化(Final品質の合成時)はsRGB EOTF、8bitへ戻す時はその逆関数
- BT.709タグ付き素材の伝達関数(BT.709 OETF/BT.1886)とsRGB曲線は中間調で厳密には異なるが、**v1はsRGB近似で統一する**ことを明示的に容認する(AE等の実務ツールも表示参照のsRGB前提運用が一般的)。原色(primaries)はsRGB=BT.709で同一なので色相はズレない
- 書き出しタグは`bt709`(trc含む)を付ける。「内部sRGB近似 vs タグbt709」の食い違いは既知の近似として受け入れ、厳密な変換管理が必要になったらOCIO導入(v2)で解決する
- この近似はプロジェクト全体で**一貫している限り**視覚的問題を生まない。危険なのは箇所ごとに解釈が揺れること — リニア化・逆変換はmotolii-gpuの共通関数のみを使い、各エフェクトで勝手に実装しない
- **このガードの重み(2026-07-08、落とし穴F-5)**: Oliveは「OCIOを後から入れ直すため」に0.2で全書き直しした。色変換が1箇所に閉じている限り、v2でのOCIO導入は「その1箇所の中身を差し替える」だけで済む(=パイプラインがOCIO-shaped)。散らばった瞬間にOliveの二の舞になるため、**この規律はVRAM常駐と並ぶレビュー最重視項目**とする

## GPUを効率よく使うための原則

「GPUを使う」だけでは速くならない。GPUが速いのは**大量の独立したデータに同じ処理を並列適用する**時だけで、逆にそこを外すと簡単にCPUより遅くなる。うちが守る原則:

1. **往復を減らす > 演算を減らす**: 前述の通り支配コストはメモリ帯域。エフェクトを個別パスで順に適用するより、単純な連鎖は1シェーダに融合(パスフュージョン)して中間書き出しを消す方が効く。中間バッファはピンポン2枚を使い回す(確保・解放を毎フレームやらない)
2. **CPU⇔GPUの同期点を作らない**: `download`(GPU→CPU読み戻し)やバッファmapはGPUパイプラインを止める(ストール)。プレビュー表示はGPUテクスチャを直接画面に出し、読み戻しは書き出しとテストだけに限る。「1フレームごとに結果をCPUで確認」は最悪パターン。禁止の正確な範囲と例外(**確定出力の非同期コピーアウト**によるRAM/ディスクキャッシュ充填は許可)は[memory-model.md](memory-model.md) P1(2026-07-09)
3. **ディスパッチ回数を減らす(バッチング)**: 小さな描画を大量に発行するとGPUが遊ぶ。同種オーバーレイ(大量の図形)はインスタンシングで1ディスパッチにまとめる。ドローコール数はフレーム時間に直結する
4. **分岐と可変ループを避ける**: GPUは32/64スレッドが同じ命令を歩調を合わせて実行する(SIMT)。シェーダ内の`if`でスレッドごとに道が分かれると両方の道を実行して捨てる(ダイバージェンス)。エフェクトのパラメータ分岐はできるだけシェーダ生成時(コンパイル時)に解決し、実行時分岐にしない
5. **精度は必要最小限**: fp16はfp32の半分の帯域で、GPUによっては2倍のスループット。中間はfp16既定、8bitで足りる入出力は8bitのまま(前述)
6. **非同期に走らせCPUを遊ばせない**: デコード(ffmpegサイドカー)・解析・レンダを別ステージにしてパイプライン化。CPUが次フレームをデコードする間にGPUが現フレームを描く

### 逐次依存シミュレーションをレンダ経路に持たないことの効率化([concept.md](concept.md)横断決定)

この項の判断は2026-07-10に**「馬鹿正直にシミュレートしない」として3Dに限らないツール全体の根本コンセプトへ一般化**した(イージング・パーティクル・将来の物理風機能すべてに適用。[concept.md](concept.md)決定事項の筆頭)。

- 全レイヤー/ノードを「時刻tの純関数」に限定することで、**フレームは互いに独立** = 先読み(prefetch)・並列ベイク・区間キャッシュが全部効く。布/液体/パーティクル/2D剛体/Rapier等の**前フレーム依存**を持ち込むと独立性が壊れ、キャッシュもパイプライン化も無効になる — 2D/3Dの区別なく**レンダ経路には**持ち込まない(B-5)。逐次状態が本当に要る表現はレンダ外のベイク相(SimulationPlugin+StateTrack)で走らせ、レンダは結果を読むだけにする(2026-07-10、[simulation-model.md](simulation-model.md))— フレーム並列はそのまま守られる
- **3Dレイヤー固有の軽さ**: 静的/決定論的メッシュは**頂点バッファをGPUに置きっぱなし**にできる(毎フレーム転送しない)。動くのはカメラ・トランスフォーム行列(数十バイトのuniform)だけ。シミュレーション系(毎フレーム頂点を再計算)との差
- プレビューでカクついてよい部分は、Draft品質(低ポリLOD・低解像度レンダターゲット)で回し、確定後にM4の区間ベイクで焼く。3D専用のキャッシュ機構は作らず、2D/エフェクトと同じキャッシュに相乗りする
