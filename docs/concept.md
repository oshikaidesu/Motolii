# コンセプト定義

作成日: 2026-07-07(セッション内での確認に基づく)

## 一文で

**MV(ミュージックビデオ)制作のための、モーショングラフィック指向のコンポジットツール。**
3〜5分程度の動画を最終成果物として書き出すことがゴール。

## 解決したい問題(既存ツールの限界)

- **Cavalry**: ベクター/パラメトリックに寄りすぎており、ピクセル(ラスター)処理や3D適用が重い
- **After Effects**: 全体的に重い(旧世代アーキテクチャ)
- **Alight Motion**: 軽いが、映像解析→ジェネレーティブ生成(Traceryライク)と3Dメッシュ合成を持たない

→ 「ベクターにもラスターにも偏らず、**GPUネイティブでモーショングラフィックと3Dメッシュ・動画素材を1つのコンポジットに束ねられる**軽いツール」が存在しない、というのが出発点。

## このツールが「である」もの

1. **モーショングラフィックツール**: キーフレーム・イージング・プロシージャル生成(パラメータ駆動の図形/オーバーレイ)が主役。評価エンジン(oc-eval)がコア
2. **解析駆動のジェネレーティブ合成**: 映像の色解析・トラッキング結果(DataTrack)がパラメータを駆動し、その“解析→生成”はプラグイン効果として差し込める(Traceryライクな挙動)
3. **2.5D/3D合成**: 動画テクスチャ平面とglTFメッシュを同一シーンで合成(合成順序はレイヤー順、Zはカメラ投影のみ)  
   - 点群(Cloud Points)も同じ「レイヤー」の一種として扱う。プラグインがXYZ点群をカメラ投影し、最終的にpremultiplied RGBAの2Dテクスチャへラスタライズして出力するため、以降の重ね合わせは通常の合成(Composite)で行える。  
   - **カメラ文脈はコンポ全体で共有する**: v1では「レイヤごとにカメラを持つ」ことはしない。コア(コンポ/Document)側に `CompCamera`(position/target/fov/roll等)としてカメラ文脈を1つ保持し、3D系レイヤソースプラグイン(点群/メッシュ等)はその `CompCamera` を参照して投影し、2Dテクスチャを返す。  
   - **ユーザー体験の制約**: レイヤ側は“3D素材としての配置/2D-ishなtransform”だけを持ち、カメラ追加・切替・グループ内カメラの制御といったBlender/Nuke的な沼に入らない。  
   - したがって「点群レイヤが2Dレイヤと重なると面白い」は設計上そのまま成立する。一方で、レイヤ間で“深度による相互遮蔽”を厳密にやりたい場合は、v1の範囲を超える(深度バッファ/順序付きラスタライズ等が必要)ため、後続の拡張領域として扱う。
4. **MVを1本書き出せる**: 3〜5分・音楽同期の最終書き出し(音声mux込み)が完成条件

## このツールが「ではない」もの

- **カラーグレーディングツールではない**: 色調整はM5の最小ノード(リフト/ガンマ/ゲイン等)で足りる範囲に留める。スコープ拡大しない
  - ※内部の色空間正確性(落とし穴B-3)はグレーディング機能とは別物で、引き続き必須(合成結果が正しく見えるための土台)
- **汎用NLEではない**: カット編集主体の長尺編集(Vlog/実写編集)は対象外
- **VFXコンポジタではない**: 深度合成・ロトスコープ・キーイング等のNuke領域はやらない

## 競合比較: 現時点で「足りない」もの(2026-07-07 整理)

### 意図的にやらない(コンセプト上の非目標)

| 欠けている機能 | 競合 | 理由 |
|---|---|---|
| カラーグレーディングの深度 | AE/DaVinci | 「ではないもの」に明記済み。最小の色調整ノードのみ |
| モバイル/タッチUI | Alight Motion | OSネイティブデスクトップが対象 |
| アプリ内フリーハンド描画 | Alight Motion/Cavalry | コアは標準シェイプ+SVG読み込み+パス変形まで。直接描画はプラグイン領域(決定済み) |
| パスブーリアン等の高度なベクター演算 | Cavalry | ベクターはSVG素材+プロシージャル生成の範囲に留める |
| ロトスコープ・キーイング・深度合成 | AE/Nuke | VFXコンポジタではない |
| 複数音声トラック・ミキシング | AE/NLE全般 | 音声=楽曲1本の決定による |

### プラグイン(古典的エフェクトスタック)で賄う

| 欠けている機能 | 競合 | 方針 |
|---|---|---|
| 歌詞テキスト/タイポグラフィ | AE/AM | プラグイン領域(決定済み) |
| エフェクトライブラリの厚み(グロー、歪み、スタイライズ等) | AE/AM | コアは最小ノードのみ。「入力テクスチャ+パラメータ→出力」の単純なプラグイン境界を用意し、LLMで書き足していく |
| Cavalry的なジェネレータ/デュプリケータ/配列 | Cavalry | プロシージャルオーバーレイ生成の拡張として同様にプラグインで |
| エクスプレッション(パラメータ間スクリプト) | AE | ParamSourceのDataTrack参照が部分代替。汎用スクリプトはWASMパラメータプラグイン(v2)の領域 |

### 構造に関わる3項目(2026-07-07 決定済み → 上の決定事項参照)

| 機能 | 決定 |
|---|---|
| マスク/トラックマット | **クリッピングマスク方式でv1コアに入れる**(下レイヤーへのクリップ、モード選択式)。マスク形状=通常のシェイプ/SVGレイヤーなので専用マスクパス編集は不要 |
| プリコンポ | **作らない。グループ化(再帰可)+仮出力(ベイク)で置換**。ドキュメントモデルは初めから「トラック項目=クリップ or グループ」の再帰スキーマにする(後付け回避) |
| モーションブラー | **プラグイン領域**。Quality型にサンプル数の口だけ確保 |

### 逆に、競合に無いこの ツールの強み(再確認)

映像解析→ジェネレーティブ生成(Traceryライク)、3Dメッシュ(glTF)と動画の統一シーン合成、ネイティブ軽量、楽曲1本前提のMV特化ワークフロー(波形+BPMグリッド基準線)。

## 3〜5分書き出しが設計に課すこと

- 1080p30で約5,400〜9,000フレーム。書き出しスループットが実用性に直結(レンダ速度の実測をM1-T11の完了条件に含める)
- **書き出しには音声muxが必須**(映像だけのmp4はMVとして成立しない)→ M2に書き出し音声muxタスクを追加
- 解析データ(DataTrack)は5分@30fpsで約9,000サンプル/系列 — 現行設計で問題ないが、区間キャッシュ(M4)の対象になる長さ

## 決定事項(2026-07-07 確認)

- **エフェクトモデルはAE式の積み重ねスタック**: レイヤー(クリップ)に順序付きでエフェクトを積む古典的モデル。ノードグラフをユーザーに見せるUIは作らない(「ノードベースで無限の拡張」はやらない)。内部エンジンの依存グラフ+ダーティフラグは実装詳細として維持し、レイヤーのエフェクトスタックはレンダ時に線形チェーンとして展開される
- **エフェクトは古典的プラグイン形式**: 「入力テクスチャ+パラメータ→出力テクスチャ」の単純な境界。拡張性はノードの組み合わせではなく、プラグインを書くことで得る(LLMでプラグインを自作できる時代なので、境界のシンプルさ=書きやすさを最優先)
- **ベクター描画(直接描画)はプラグイン領域**: コアは「標準シェイプ+SVG読み込み+パスの変形(デフォーマ。テキスト由来のパスも同様に歪ませられる)」まで。フリーハンドの直接描画ツールはUIをごちゃつかせるためコアに入れない
- **マスクはクリッピングマスク方式(お絵描きソフト・クリスタをリスペクト)**: AEのマスクパス方式ではなく「下のレイヤーにクリップする」方式を採用。視覚的な分かりやすさ優先。マスクモード(アルファ/ルミナンス/反転)は選択式。マスク形状が欲しければシェイプ/SVGレイヤーを下に置いてクリップすればよい(専用マスクパス編集機能が不要になる)
- **プリコンポは作らない。「グループ化+仮出力」で置き換える**: AEのプリコンポ(別タイムラインへの分離)はやらず、レイヤーの単純なグループ化(再帰可)にする。重さ対策は「グループを選択して仮出力(ベイク)」— グループの出力を時間範囲でキャッシュし、編集したら自動で無効化。プリレンダの直観的な代替
- **モーションブラーはプラグイン領域**: コアのレンダラには入れない(使用頻度が低い)。品質パラメータにサンプル数の口だけ確保しておく
- **UI基盤はSlint**(2026-07-08、S1スパイク合格を条件): wgpuテクスチャのゼロコピー埋め込み(A-1ブリッジ問題の消滅)と日本語IME実績を両立する唯一のRustネイティブ候補のため。WebView/Tauri案は廃止。日本語UIを一級の判断基準とする。詳細はM3仕様
- **プレビューは品質を落とし、最終レンダのみ厳密**: プレビューでは多少の色の違い・解像度低下に目をつぶる(Draft品質: 半解像度・fp16・色変換ショートカット・エフェクトのdraftサンプル数)。最終レンダリング(Final品質)だけが正しさの基準。UIのモード切替で「最終品質プレビュー」も可能。両者は同一の`render_frame(t, Quality)`を通るため見た目の分岐は起きない。根拠と詳細は[performance-model.md](performance-model.md)(重さの正体はメモリ帯域×往復回数であり、解像度そのものではない)

- **歌詞テキスト/タイポグラフィ**: プラグイン領域。v1コアには入れない(最小コア+プラグインベースの方針通り)。ただしレビュー指摘の通り**日本語MVでは歌詞組版は主用途に近く、サードパーティ任せでは取りこぼす**ため、v1完成後の**ファーストパーティプラグイン第1号**として自分たちで作る計画とする(縦書き・ルビ等の日本語組版は非自明なため、プラグインAPIの実力テストを兼ねる)
  - **分界の追記(2026-07-08、落とし穴F-6)**: 「シェーピングまではコア、組版はプラグイン」。フォント読み込み・フォールバック・日本語シェーピング(rustybuzz)・グリフ→パス変換は共有基盤としてコア(oc-text、M5-P6)に置く。縦書き・ルビ・行組・歌詞タイミングがプラグインの領分。コアにテキスト基盤が無いと第1号プラグインがテキストスタックまるごと自作になるため
- **正準座標系(2026-07-08、落とし穴F-1)**: 合成空間は**単位なし・原点=コンポ中央・Y-up・高さ基準正規化(高さ=1.0)**。全空間パラメータ(位置・アンカー・ブラー半径・マスク位置)は正準空間の値で持ち、px変換はレンダ直前の1箇所のみ。これがDraft(半解像度)とFinalの見た目一致の前提であり、glTF(Y-up)との2.5D同居の前提。グループは変形コンテナ、ペアレンティング(親参照)はスキーマ予約
- **並行性・所有権(2026-07-08、落とし穴F-2)**: 単一writer(コマンド適用の編集スレッド)+不変スナップショット(`Arc<Document>`)。レンダ・書き出し・解析・プロキシ生成は読み手。Natronの死因(race/deadlock/キャッシュデッドロック)の構造的排除
- **クリップは時間写像(TimeMap)を持つ(2026-07-08、落とし穴F-4)**: v1は恒等+定数速度のみ実装だが、ソース時刻アクセスは初日からTimeMapを通す。速度ランプ・逆再生(MV必須)へスキーマ互換のまま拡張可能にする
- **SVG読み込み**: コア機能。モーショングラフィック素材(ロゴ・図形)の入口として根本に関わるため、ベクター描画基盤の選定(M1-T7)はSVGレンダリングを前提に行う(Vello + usvgを第一候補として評価必須)
- **アプリ内エクスプローラー(アセットブラウザ)**: コアUI。素材(動画・SVG・音声)の管理はドキュメントモデルのAsset実体(M2)+ブラウザUI(M3)として設計する
- **ビート検出**: やらない。作り手はBPMを既に知っているため、プロジェクト設定にBPMを手動入力し、タイムラインにビートグリッド/スナップを出す方が正しい(M2/M3)
- **音声は1プロジェクトに1本(楽曲)のみ**: MV制作では音声=完成済みの楽曲1本であり、多トラック音声・ミキシング・音声編集は作らない。動画クリップ内蔵の音声は使わない(常にミュート)。タイムラインは楽曲を基準線(波形+BPMグリッド)として編集する。書き出し時は楽曲をそのままmux(可能ならストリームコピーで無劣化)
- **音声の根本設計**: 音ズレ・途切れ・「重い時に映像だけ遅れる」問題は後付け修正ではなく、トランスポート(再生ヘッド)のクロック設計で構造的に排除する。方式は[M2仕様の「音声トランスポート設計」](specs/M2-document-model.md)に規定(要旨: 再生ヘッドの所有者は常に1つ。通常時は音声デバイスクロックが主で映像が追従(フレームドロップ)、レンダが実時間に届かない時はレンダ進捗がクロックになり音声が適応リサンプリングで追従=AEのような「映像だけ遅れて音が先行する」状態を構造的に不可能にする)
