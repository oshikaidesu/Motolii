# コンセプト定義

作成日: 2026-07-07(セッション内での確認に基づく)

## 一文で

**MV(ミュージックビデオ)制作のための、モーショングラフィック指向のコンポジットツール。**
3〜5分程度の動画を最終成果物として書き出すことがゴール。

## 解決したい問題(既存ツールの限界)

- **Cavalry**: ベクター/パラメトリックに寄りすぎており、ピクセル(ラスター)処理や3D適用が重い
- **After Effects**: 全体的に重い(旧世代アーキテクチャ)
- **Alight Motion**: 軽いが、映像解析→ジェネレーティブ生成(Traceryライク)と3Dメッシュ合成を持たない

→ 「ベクターにもラスターにも偏らず、**GPUネイティブでモーショングラフィックと3Dメッシュ・動画素材を1つのコンポジットに束ねられる**軽いツール」が存在しない、というのが出発点。

## このツールが「である」もの

1. **モーショングラフィックツール**: キーフレーム・イージング・プロシージャル生成(パラメータ駆動の図形/オーバーレイ)が主役。評価エンジン(oc-eval)がコア
2. **解析駆動のジェネレーティブ合成**: 映像の色解析・トラッキング結果がパラメータを駆動する(Traceryライクな挙動)
3. **2.5D/3D合成**: 動画テクスチャ平面とglTFメッシュを同一シーンで合成(合成順序はレイヤー順、Zはカメラ投影のみ)
4. **MVを1本書き出せる**: 3〜5分・音楽同期の最終書き出し(音声mux込み)が完成条件

## このツールが「ではない」もの

- **カラーグレーディングツールではない**: 色調整はM5の最小ノード(リフト/ガンマ/ゲイン等)で足りる範囲に留める。スコープ拡大しない
  - ※内部の色空間正確性(落とし穴B-3)はグレーディング機能とは別物で、引き続き必須(合成結果が正しく見えるための土台)
- **汎用NLEではない**: カット編集主体の長尺編集(Vlog/実写編集)は対象外
- **VFXコンポジタではない**: 深度合成・ロトスコープ・キーイング等のNuke領域はやらない

## 3〜5分書き出しが設計に課すこと

- 1080p30で約5,400〜9,000フレーム。書き出しスループットが実用性に直結(レンダ速度の実測をM1-T11の完了条件に含める)
- **書き出しには音声muxが必須**(映像だけのmp4はMVとして成立しない)→ M2に書き出し音声muxタスクを追加
- 解析データ(DataTrack)は5分@30fpsで約9,000サンプル/系列 — 現行設計で問題ないが、区間キャッシュ(M4)の対象になる長さ

## 決定事項(2026-07-07 確認)

- **歌詞テキスト/タイポグラフィ**: プラグイン領域。v1コアには入れない(最小コア+プラグインベースの方針通り)
- **SVG読み込み**: コア機能。モーショングラフィック素材(ロゴ・図形)の入口として根本に関わるため、ベクター描画基盤の選定(M1-T7)はSVGレンダリングを前提に行う(Vello + usvgを第一候補として評価必須)
- **アプリ内エクスプローラー(アセットブラウザ)**: コアUI。素材(動画・SVG・音声)の管理はドキュメントモデルのAsset実体(M2)+ブラウザUI(M3)として設計する
- **ビート検出**: やらない。作り手はBPMを既に知っているため、プロジェクト設定にBPMを手動入力し、タイムラインにビートグリッド/スナップを出す方が正しい(M2/M3)
- **音声の根本設計**: 音ズレ・途切れ・「重い時に映像だけ遅れる」問題は後付け修正ではなく、トランスポート(再生ヘッド)のクロック設計で構造的に排除する。方式は[M2仕様の「音声トランスポート設計」](specs/M2-document-model.md)に規定(要旨: 再生ヘッドの所有者は常に1つ。通常時は音声デバイスクロックが主で映像が追従(フレームドロップ)、レンダが実時間に届かない時はレンダ進捗がクロックになり音声が適応リサンプリングで追従=AEのような「映像だけ遅れて音が先行する」状態を構造的に不可能にする)
