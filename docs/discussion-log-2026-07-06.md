# 設計ディスカッション ログ (2026-07-06)

`design-memo.md` (2026-07-05作成) をベースに行った追加検討のまとめ。

## 1. UIをゼロから作るか、既存を参考にするか

- **OpenCut** (MITライセンス) はUI流用元として有力。特に注目すべき点:
  - 現在OpenCut自体が「Web(Next.js) + デスクトップ(GPUI, Rustネイティブ) + Rustコア(GPU compositor/effects/masks/WASM bindings)」という、本メモの方向性とほぼ同じ構成にフルリライト中。
  - デスクトップ版はTauri+WebViewではなく**GPUI**(Zedチーム製、Rustネイティブ・GPU描画のGUIフレームワーク、WebView不使用)を採用 — WebViewさえ捨てるという、より徹底した選択肢の存在を確認できた。
  - ただしRustコア部分は現在進行形でリファクタリング中(プレビュー/エクスポートは「バイナリレンダリング方式へ移行中」)であり、まだ依存先にするには不安定。
  - **結論**: Web版のReactタイムラインUI(MIT)はコード流用の価値あり。Rustコアは設計思想の参考に留める。GPUIはWebView代替候補として要検討。

## 2. Alight Motion / 既存ツールで代替できないか

- 機能性だけで見るとAlight Motionは軽量AE代替として優秀だが、以下2点は持っていない(と推測):
  1. 映像解析(色トラッキング/オプティカルフロー)からのジェネレーティブ生成(Traceryライクな挙動)
  2. 3Dメッシュ(OBJ/glTF)の読み込みと合成
- 「軽さ」だけが目的ならAMで足りるが、上記2機能が本題である以上、自作する理由は残る。

## 3. ノードベース内部設計 と 簡易UI は両立するか

- ノードベースが業界主流なのは技術的必然ではなく、Nuke/Houdini等が「精密な合成順序制御」をユーザーに見せたいプロダクト判断によるもの。
- 内部エンジン(依存関係グラフ+ダーティフラグキャッシュ)とUI表層(ノードグラフを見せるか、レイヤー/タイムラインだけ見せるか)は分離可能。
- 単独用途(自分のMV制作)なら、ノードグラフはユーザーに見せず、Alight Motion的な簡易UIだけを表層に出す設計で問題ない。

## 4. AviUtl2(ExEdit2)を魔改造するルートの検討

調査の結果:

- **現状**: 2025年7月にオリジナル作者がゼロから再設計しベータ公開(2026年2月時点でベータ33〜34、週次更新)。拡張編集(タイムライン・レイヤー・キーフレーム)が本体に統合済み。64bit対応、DirectX 11.3 + AVX2必須(GPUアクセラレーション前提の新エンジンと推測)。UIも単一ウィンドウに統合されモダン化。
- **プラグインSDK**: テーブル登録方式のC ABI(DLLが構造体をexport)。C++縛りではなくRustの`cdylib`で実装可能 — Rust方針と衝突しない。
- **キーフレームUI**: 本体機能として既にAE的な体験にかなり近い。
- **UIのAE風モダン化**: 配色レベルのテーマ/MOD(`style.conf`配布、`al2_jd`によるDLL injection+Direct2D/DirectWriteでのダークモード強制)は存在するが、**AEのような可変パネルドッキング構造そのものを実現するMODは見当たらない**(単一ウィンドウ統合済みのため需要自体が薄いと推測)。
- **Tracery相当の機能**: 存在しない。近いものは「Motion Tracking MK-Ⅱ (Plus)」のみだが、これは点/領域ベースの物体追跡(モザイク追従等)であり、「フレーム全体の色解析→ジェネレーティブ生成」というTraceryの設計思想とは別物。
- **キャッシュ機構**: 画像/音声キャッシュサイズ、キャッシュフレーム数、VRAMキャッシュ(静止画のみ)を設定可能。ただし本質的にはAEのRAMプレビューと同じ「フレームまるごとキャッシュ」方式で、ノード依存関係ベースの選択的キャッシュではない。上限を明示的に設定できる点は旧AviUtl1(32bit・2GB天井)より改善。
- **安定性**: 64bit化により旧来の「メモリ不足です」クラッシュのクラスは構造的に起きにくくなっているはずだが、まだベータであり大量/大容量素材関連の不具合も実際に報告・修正されている。

**結論**: キーフレームUI・タイムラインは魔改造せずとも本体機能でほぼ解決している。しかしTracery相当の解析→生成パイプライン、3D合成、ノードベースキャッシュ設計はAviUtl2側に存在しないため、結局自作が必要。ベータの不安定さ・API変動リスクも踏まえ、**土台として依存するのはリスクが高い**という判断。

## 5. コンセプト決定

- **最小コア + 全機能プラグインベース**: 基本機能を作り込まず、拡張性を基礎とする。実装する粒を最小限にする。
- **開発体制**: 並列(AI)エージェントによる開発を前提に設計する。
- **プラットフォーム**: OSネイティブで考える。
- **WebGPUへの不安**: ブラウザ実装間のばらつき(Chrome/Safari/Firefoxの差異)への不安であり、OSネイティブ(wgpu Rustクレート)を使う時点でスコープ外になる。wgpu自体はVulkan/Metal/DirectX12に直接マッピングするネイティブライブラリでブラウザ非経由。ただしwgpuクレート自体はまだ活発に開発中でAPI破壊的変更やバックエンド固有バグのリスクは残る。

### 5-1. プラグイン隔離方式

- ネイティブ dylib(高速、隔離弱い) vs WASMサンドボックス(安全、隔離コスト大)の二択ではなく、**「生のピクセル/GPUバッファに直接触るか否か」で切り分ける**。
  - ピクセル/GPUバッファに触るもの(デコード・トラッキング解析・3D合成・コンポジット): ネイティブ(第一級市民)
  - 構造化データ(数値・パラメータ)だけをやり取りする生成ルール/スクリプト的拡張(Traceryライクな部分): WASMサンドボックス
- 理由: WASM線形メモリは隔離されておりゼロコピーでGPUテクスチャを渡せないため、ピクセルの大容量データを毎フレームやり取りする用途には不向き。パラメータ授受だけなら隔離コストはほぼ無視できる。

### 5-2. フレーム/バッファの共通フォーマット

他プロジェクトの前例:
- **three.js**: r152(2023年)で`texture.encoding`という曖昧な指定から`texture.colorSpace`(`SRGBColorSpace`/`LinearSRGBColorSpace`等)を明示必須にするAPIへ刷新。色空間を暗黙にすると事故る、という教訓。
- **FFmpeg (AVFrame)**: pixel format(`yuv420p`等) + `linesize`(strideパディング) + `color_range`(limited/full) + `color_trc`/`color_primaries`まで構造体で明示。
- **OpenColorIO (Nuke/Houdini/Blender等VFX標準)**: バッファの色空間タグに加え、パイプライン全体の変換ルールを一元管理。

**採用方針**: プラグイン間で受け渡すフレーム記述子は最低限以下を持つ:
```
{ width, height, stride, pixel format (wgpu::TextureFormat準拠 + YUV系数種),
  color space (linear / sRGB / Rec.709 等), premultiplied alpha flag }
```
ネイティブプラグイン間では`wgpu::Texture`ハンドルをそのまま共有(CPUバウンスなし)。WASM側には基本バッファを見せない。

### 5-3. UI表層はWeb技術寄せでよい

- 最初の設計メモの構成(WebView=UIチェイン、ネイティブ=コア)に回帰する形で矛盾しない。
- OpenCut(MIT)のReactタイムラインをそのまま流用候補にできる。
- ネイティブコアとUIの間は、5-2で決めたフレーム記述子に沿ったテクスチャ/画像をIPC経由で渡すだけなので、UI側の実装技術に依存しない。

## 6. 擬似3D(2.5D)のスコープ

- AEの3D機能(複数カメラのクロスフェード、null経由のカメラリグ、被写界深度のボケ形状制御、ライトリグ)は「Blender並みにやりすぎ」という判断。
- **提案**: 2Dレイヤーと3Dメッシュを最初から同じシーングラフに統一する。2Dレイヤーを「Z座標を持つ、動画テクスチャを貼った平面」として扱えば、OBJ/glTFメッシュと同じシーンに共存でき、二重レンダリングパスを避けられる。Z軸視差は特別な機能ではなく、平面のZ座標とカメラ移動から自然に生じる結果として実現される。
- **カメラは最小限に**: 単一カメラ(位置・注視点/回転・FOV)のみ。複数カメラのクロスフェードやnullリグは作らない。被写界深度は光学ボケではなくZ距離に応じたポストプロセスブラーで代用。ライティングは初期は無し(unlit)か固定1灯とし、必要になった時だけPBR/シャドウを追加する。
