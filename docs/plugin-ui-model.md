# プラグインUIモデル: 宣言語彙 vs 自由描画

作成日: 2026-07-13
ステータス: **設計ノート(批判レビュー待ち)**。契約・凍結口への変更なし。先例表は未カウンターレビューの仮説([reviews/README.md](reviews/README.md)規律3)。採用判断の前にAM実機確認(§6)と反対側レビューを通す
関連: [plugin-authoring.md](plugin-authoring.md)§2(NodeDesc)/§8(凍結していない口)、[dev-experience.md](dev-experience.md)、[ae-pain-points.md](ae-pain-points.md)、[backlog.md](backlog.md) GAP-2/GAP-13、M3(UI統合)

## 1. 問題(なぜこの文書が要るか)

エフェクトパネルは今後、空間パラメータの表現(x/yを同時に動かすハンドル等)のためにUI拡張が必要になる。このとき「プラグインにUIをどこまで開けるか」は、後戻りできない公開契約の判断になる。

AE圏の観察(仮説): **独自UIは有料プラグインのメイン機能クラスの価値**だった — Optical Flares(フレアデザイナー)、Element 3D(シーンセットアップウィンドウ)、Flow(製品全体がベジェカーブエディタ)。ただしこれらを分解すると、独自UIの相当部分は**AEの宣言的パラメータ語彙が貧しすぎたことの穴埋め**(カーブエディタ・グラデーションエディタ・XYパッド・ヒストグラムをAEが標準で持たなかったため各社がバラバラの品質で再発明した)。「独自UI=差別化」に見えたものの中身は「ホスト語彙の欠落」だった、が本文書の作業仮説。

## 2. 原則: 開ける面をコードからデータに寄せる

**プラグインは「意図」をデータとして宣言し、解釈と実行はホストが独占する。** 開けるのはAPI(振る舞いの口)ではなく語彙(データの種類)。

これはMotoliiの既存契約が一貫して使っている手と同じもの:

| 領域 | プラグインが宣言するデータ | ホストが独占する解釈 |
|---|---|---|
| ピクセル | WGSL+descriptor | パイプライン生成・保持(F-10) |
| 状態 | `init/step/snapshot` | 保存・スクラブ・無効化(F-12) |
| 値 | DataTrack | 評価タイミング |
| 空間 | 正準座標 | px変換(F-1) |
| **UI(本文書)** | **パラメータ型+UIヒント+レイアウト+ギズモ宣言** | **描画・ヒットテスト・テーマ・入力方式** |

具体APIを個別に開けない理由:

1. **口は凍結資産になる** — 一度公開したAPIは引っ込められない。宣言語彙なら加法的に増やせて既存プラグインを壊さない
2. **解釈の自由がホストに残る** — 「vec2はハンドル操作」という宣言なら、タッチ対応・テーマ変更([ui-color-direction] Ableton風)・a11yをホスト側だけで進化させられる。プラグインが描画コードを所有するとAEと同じ「古いUIを捨てられないホスト」になる
3. **検証がデータ検証になる** — `validate_node_desc`の延長で機械チェック可能。INF-7「人間差し戻しをCIへ」がUIにも伸びる。LLM量産(concept)の前提とも整合
4. **ホットリロードが自明** — ウィジェット実装は全部ホストコードなので、[dev-experience.md](dev-experience.md)のはしごがUIにもそのまま効く

トレードオフ: 表現力に天井ができる。天井対策は§4のレベル4(逃げ道の予約)で扱い、v1では開けない。

## 3. 先人の解決策(仮説台帳。未カウンターレビュー)

| 先人 | 機構 | 判定の方向 |
|---|---|---|
| AE エフェクトコントロール | 宣言的パラメータ型(`PF_ADD_SLIDER`/`PF_ADD_POINT`等)、描画はホスト。Pointはパネルにx/y数値+**コンポビューアの十字ハンドルで同時操作** | 基本形として採る。「空間はビューポートで直接掴む」はAMとも一致 |
| AE `PF_CUSTOM_UI` | プラグインが自前でピクセル描画+イベント処理 | **採らない**。重い・不統一・ホストがUIを進化させられない温床 |
| AE ScriptUI → CEP → UXP | 宣言的リソース記述 → Chromium埋め込み(自由HTML) → 再び統制へ | 3世代の振り子。自由(CEP)は一貫性と資源で破綻し、Adobe自身がUXPで統制へ戻した(要一次資料確認) |
| **Blender アドオンUI** | 自由にパネルを組めるが、描けるのは**ホストのウィジェットだけ**(`row()`/`column()`/`prop()`の宣言的レイアウトAPI)。数千アドオンでも見た目が統一 | **中間解の本命**。レベル2の型紙 |
| OpenFX Interact | ビューポートオーバーレイの標準口。ただしプラグインがGL描画+イベント処理を所有 | 口の存在は採る、方式は採らない → 宣言化する(レベル3) |
| VST/AU(音声プラグイン) | 完全自由描画UI。UIそのものが製品(Serum等) | 市場実証はあるが、UI一貫性ゼロ・ホスト進化不能。Motoliiの位置取り(閉じたAMと開きすぎたAEの間)では採らない |

## 4. はしご(開ける順番自体が設計)

| レベル | 内容 | 時期 | 先例 |
|---|---|---|---|
| 1 | **ホスト標準ウィジェット語彙を厚くする**: XYパッド、カーブ、グラデーション、角度ダイヤル等。AEで独自UI化した定番を最初から`ParamDef`の型/UIヒントとして供給 | v1(M3) | AEの欠落の裏返し |
| 2 | **宣言的レイアウト**: 行/列/グループ/条件表示をプラグインがデータで記述、描画はホスト | v1.x(語彙が足りなくなってから) | Blender `layout.prop()` |
| 3 | **ビューポートギズモの宣言語彙**: vec2等に紐づくハンドル・線・角度弧を`NodeDesc`で宣言。描画・ヒットテスト・スナップはホスト | v1(M3。空間パラメータの本線) | OFX Interactの宣言化。正準座標(F-1)によりparam値→画面位置の写像はホストが一意に引ける |
| 4 | **自由描画UI**(Optical Flaresデザイナー級) | **v1では開けない**。必要が実証されたらv2でWASMサンドボックス+口の設計 | VST/AE圏。開けるなら最後 |

**順番の理由**: レベル4を先に開けるとレベル1〜3を誰も待たず、AE圏の再演(各社バラバラの自前UI、ホストはテーマ一つ変えられない)になる。逆順なら、レベル1〜3で Flow(カーブUI)・XYハンドル系・カラコレ系の主要ユースケースを**作者のUI実装コストゼロ+AM級の統一感**でカバーでき、レベル4に本当に残る需要だけを観測してから口を設計できる。

## 5. スキーマへの影響(v1でやること/やらないこと)

- **やる(M3設計時)**: `ParamDef`への**省略可能・加法的なUIヒント**(レンジ、対数スケール、ウィジェット種別、vec2ペアリング)の形を決める。加法的フィールドだが公開スキーマに触るため、正式追加は解凍手続きの要否を確認してから
- **やる(方針の明文化)**: 自由描画UI(プラグイン所有の描画・イベント処理)はv1で開けない旨を[plugin-authoring.md](plugin-authoring.md)§8に一行予約 → 済
- **やらない(予約のみ)**: レベル2レイアウト語彙・レベル3ギズモ語彙の具体設計。レベル4のWASM UI口
