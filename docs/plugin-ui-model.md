# プラグインUIモデル: 宣言語彙 vs 自由描画

作成日: 2026-07-13(同日、批判レビュー7点を反映して改訂)
ステータス: **非規範の設計仮説(批判レビュー対応済み・採否判断待ち)**。契約・凍結文書への変更なし。**v1の公開境界は[plugin-ui-v1-boundary](reviews/2026-07-12-plugin-ui-v1-boundary.md)が正** — `NodeDesc`自動生成パネルのみ。本仮説の「3段構え」記述は将来候補の棚であり、現行M3仕様の規範ではない(§7は歴史的対比)。
関連: [plugin-authoring.md](plugin-authoring.md)§2、[specs/M3-ui-integration.md](specs/M3-ui-integration.md)§拡張方式、[dev-experience.md](dev-experience.md)、[backlog.md](backlog.md) GAP-2/GAP-13

## 1. 問題(なぜこの文書が要るか)

エフェクトパネルは今後、空間パラメータの表現(x/yを同時に動かすハンドル等)のためにUI拡張が必要になる。「プラグインにUIをどこまで開けるか」は後戻りできない公開契約の判断になる。

AE圏の観察(仮説): **独自UIは有料プラグインのメイン機能クラスの価値**だった。ただし性質は一様ではない:

- **語彙の穴埋め型**: カーブエディタ・グラデーションエディタ・XYパッド・ヒストグラム。AEが標準パラメータ型として持たなかったため各社が再発明したもの
- **専用データモデル編集環境型**: Optical Flares(フレア構成の編集)、Element 3D(3Dシーンのセットアップ)。単なるウィジェット不足ではなく、**プラグイン固有のデータモデルを編集するミニアプリ**

初版は前者だけを見て「独自UIの相当部分は語彙欠落の穴埋め」と結論していたが、後者が宣言語彙で再現できるかは未判定(レビュー指摘7)。判定には§6-1の能力コーパスが要る。

## 2. 原則(仮説): 開ける面をコードからデータに寄せる

**プラグインは「意図」をデータとして宣言し、解釈と実行はホストが独占する。** 開けるのはAPI(振る舞いの口)ではなく語彙(データの種類)。

これはMotoliiの既存契約が一貫して使っている手と同じもの:

| 領域 | プラグインが宣言するデータ | ホストが独占する解釈 |
|---|---|---|
| ピクセル | WGSL+descriptor | パイプライン生成・保持(F-10) |
| 状態 | `init/step/snapshot` | 保存・スクラブ・無効化(F-12) |
| 値 | DataTrack | 評価タイミング |
| 空間 | 正準座標 | px変換(F-1) |
| **UI(本仮説)** | **パラメータ型+表示ヒント+レイアウト+ギズモ宣言** | **描画・ヒットテスト・テーマ・入力方式** |

具体APIを個別に開けない理由:

1. **口は凍結資産になる** — 一度公開したAPIは引っ込められない。宣言語彙なら既存プラグインを壊さず増やせる(ただし増やし方自体に互換計画が要る。§5)
2. **解釈の自由がホストに残る** — タッチ対応・テーマ変更(Ableton風)・a11yをホスト側だけで進化させられる。プラグインが描画コードを所有するとAEと同じ「古いUIを捨てられないホスト」になる
3. **検証がデータ検証になる** — `validate_node_desc`の延長で機械チェック可能。INF-7「人間差し戻しをCIへ」がUIにも伸びる
4. **ホットリロードが自明** — ウィジェット実装は全部ホストコードなので[dev-experience.md](dev-experience.md)のはしごがUIにもそのまま効く

トレードオフ: 表現力に天井ができる。天井の実在量は§6-1のコーパスで測ってから、レベル4(§4)の扱いを決める。

## 3. 先人の解決策(仮説台帳。未カウンターレビュー)

| 先人 | 機構 | 判定の方向 |
|---|---|---|
| AE エフェクトコントロール | 宣言的パラメータ型(`PF_ADD_SLIDER`/`PF_ADD_POINT`等)、描画はホスト。Pointはパネルにx/y数値+**コンポビューアの十字ハンドルで同時操作**。上流エフェクトの原点オフセットもホストが吸収([SDK公式](https://ae-plugins.docsforadobe.dev/effect-basics/parameters/)、2026-07-13確認) | 基本形として採る |
| AE `PF_CUSTOM_UI` | 自前描画+イベント処理の口が**2面**ある: ECW(パネル内)とComp/Layerウィンドウ(ビューポート内)。`PF_PUI_CONTROL`フラグで宣言([SDK公式](https://ae-plugins.docsforadobe.dev/effect-ui-events/effect-ui-events/)) | 採らない方向(仮説)。重い・不統一・ホストがUIを進化させられない温床 |
| **Cavalry Layer Tools** | 一部レイヤー種別が**ビューポートに紫色の専用操作UI**を出し、設定を直接調整できる。表示はViewport Settingsの**グローバルトグル**で一括制御、配色も全種別で統一([公式docs](https://cavalry.studio/docs/user-interface/menus/window-menu/viewport/)、2026-07-13確認) | **レベル3の生きた実例**。「ノード種別ごとの専用ギズモ+描画/配色/表示制御はホスト」が現行製品で成立している。ただしCavalryの該当ノードはファーストパーティのみ(サードパーティ native pluginは無い)なので、**第三者契約として開けた実例ではない**点は割り引く |
| **Cavalry Graph Attribute** | Attribute Editorの**標準ウィジェット型としてのカーブエディタ**(Falloffの強度カーブ等)。点追加・ベジェ切替・プリセット・反転をホストが供給([公式docs](https://cavalry.studio/docs/user-interface/menus/window-menu/attribute-editor/widgets/graph-attribute/)) | **レベル1bの生きた実例**。「カーブ」がパラメータ型+ホストウィジェットとして製品成立している(AEで各社が独自UIで再発明したものの標準装備化) |
| AE ScriptUI → CEP → UXP | 宣言的リソース記述 → Chromium埋め込み(自由HTML) → 再び統制へ | 3世代の振り子。自由(CEP)は一貫性と資源で破綻し、Adobe自身がUXPで統制へ戻した(要一次資料確認) |
| **Blender アドオンUI** | 自由にパネルを組めるが、描けるのは**ホストのウィジェットだけ**(`row()`/`column()`/`prop()`の宣言的レイアウトAPI)。数千アドオンでも見た目が統一 | **中間解の本命**。レベル2の型紙 |
| OpenFX Interact | ビューポートオーバーレイの標準口。ただしプラグインがGL描画+イベント処理を所有 | 口の存在は採る、方式は採らない → 宣言化する(レベル3) |
| VST/AU(音声プラグイン) | 完全自由描画UI。UIそのものが製品(Serum等) | 市場実証はあるが、UI一貫性ゼロ・ホスト進化不能。Motoliiの位置取りでは採らない方向(仮説) |

## 4. はしご(開ける順番自体が設計)

レビュー指摘4を受け、旧レベル1を**表示ヒント(1a)と新しい値型(1b)に分離**した。両者は実装レーンも依存も別物。

| レベル | 内容 | 影響範囲 | 時期 | 先例 |
|---|---|---|---|---|
| 1a | **WidgetHint(既存値型の表示ヒント)**: `Vec2`→XYパッド/ハンドル、`F64`→角度ダイヤル・対数スライダー等。**値の意味論・保存形式は不変**、変わるのは表示だけ | UIのみ(M3) | v1(M3) | AE `PF_ADD_POINT`等の表示指定 |
| 1b | **新しいsemantic ValueType**: グラデーション(複数stop)、伝達カーブ(control point列)等。**現行`ValueType`に無い複合データ**であり、Document永続化・補間・検証・キャッシュキーへの寄与まで新規定義が要る | スキーマ+eval+M4(**M3だけの仕事にできない**) | v1.x(必要が立ってから型ごとに解凍手続き) | AEに無くて各社が再発明した型そのもの |
| 2 | **宣言的レイアウト**: 行/列/グループ/条件表示をプラグインがデータで記述、描画はホスト | UIのみ | v1.x(1aの語彙が足りなくなってから) | Blender `layout.prop()` |
| 3 | **ビューポートギズモの宣言語彙**: vec2等に紐づくハンドル・線・角度弧を宣言。描画・ヒットテスト・スナップはホスト。**宣言には最低限3点が必須**(レビュー指摘6): (a) **座標空間の指定**(ソース/レイヤーローカル/コンポ/将来カメラ) (b) 画面への**順変換**(エフェクト→変形→親グループの評価鎖のどの時点の値か) (c) ドラッグ→パラメータ値への**逆写像**。正準座標(F-1)は単位を固定するだけで空間の帰属までは決めない | UI+eval(変形鎖の照会口) | v1(M3。空間パラメータの本線) | OFX Interactの宣言化 |
| 4 | **自由描画UI**(専用データモデル編集環境級) | 全域 | **本仮説ではv1で開けない**。ただし§7のM3方式と競合中であり、採否は§6-1コーパス判定後 | VST/AE圏 |

**順番の理由(仮説)**: レベル4を先に開けるとレベル1〜3を誰も待たず、AE圏の再演(各社バラバラの自前UI、ホストはテーマ一つ変えられない)になる。逆順なら、レベル4に本当に残る需要だけを観測してから口を設計できる。**「レベル1〜3で主要ユースケースをカバーできる」は現時点で未検証の仮説**であり、§6-1のコーパス判定を経ずにv1禁止を確定しない。

## 5. スキーマへの影響(レビュー指摘3の反映)

**`ParamDef`へのフィールド追加は加法的変更ではない。** 現行`ParamDef`(crates/motolii-plugin/src/lib.rs)は公開structで、プラグインはstruct literalで構築する。`Option<WidgetHint>`の追加でも既存の全literalがコンパイルエラーになる。したがって:

- WidgetHint導入は**解凍手続き必須**(「要否確認」ではない)
- 解凍時の完了条件に**次回以降の追加を非破壊にする互換計画**を含める: constructor/builder化(`ParamDef::new(id, ty, default)` + `.with_hint(…)`)、または将来拡張用ネスト(`ui: UiHints`をDefault付きで最初から置く)等の選択と、既存参照プラグイン・`new-plugin.sh`型紙・plugin-authoring §5スケルトンの一括移行
- レベル1b(新ValueType)は毎回それ自体が解凍案件(`ValueType`/`Value` enumへのvariant追加+serde+補間+キャッシュキー)

初版でplugin-authoring§8に入れた予約行は撤回済み(凍結契約への未レビュー仮説の持ち込みのため。レビュー指摘2)。規約への反映は採否判断+解凍/契約変更手続きを通してから行う。

## 6. 未解決(採否判断前の確認事項)

2. ~~AM実機確認~~ → **確認済み(2026-07-13、ユーザー実機)**: **AMにはエフェクトの位置/x-y同時操作パラメータの比較対象が存在しない**。AMをレベル1a/3の根拠にはしない。代替観察はCavalry(§3のLayer Tools/Graph Attribute。docs確認済み、実機での出現条件・複数パラメータ時の切替は未確認)とAE標準Point挙動

## 7. 現行M3仕様との関係(歴史的対比 — v1境界は確定済み)

> **2026-07-12以降**: v1安定境界は`NodeDesc`自動生成のみ([plugin-ui-v1-boundary](reviews/2026-07-12-plugin-ui-v1-boundary.md))。以下の「3段構え」表は、本仮説起草時点のM3草案との対比であり、**現行の規範ではない**。未決はGAP-13のWidgetHint/宣言語彙採否のみ — U4はブロックしない。

[specs/M3-ui-integration.md](specs/M3-ui-integration.md)旧案§「プラグインパネルの拡張方式(3段構え)」は、**本仮説が閉じようとしていた経路をv1経路として想定していた**:

| M3の段 | 内容 | 本仮説との関係 |
|---|---|---|
| M3-1 自動生成パネル | NodeDesc宣言→ホストが汎用パネル生成 | **一致**(本仮説のレベル1a既定形) |
| M3-2 カスタムパネル(.slint実行時ロード) | プラグインが`.slint`を同梱、slint-interpreterでロード | **競合**。本仮説のレベル2(ホスト解釈の宣言的レイアウト)とレベル4の中間に位置する。`.slint`は宣言的だが、レイアウト語彙の統制・テーマ追従・検証可能性の扱いが未整理 |
| M3-3 フルカスタム描画(wgpuテクスチャ) | プラグインが描き、ホストが埋め込む | **正面競合**。本仮説のレベル4=v1で開けない、と矛盾 |

**解消手順**: v1境界は確定済み(上記)。残る未決はGAP-13のWidgetHint/宣言語彙採否のみ — U4はブロックしない。採用するなら`ParamDef`拡張の解凍手続きが先。
