# M5: 3D合成とポストプロセス

ステータス: **ドラフト**(凍結ゲートで確定)

## 目的(退治する落とし穴)

C-4(2.5Dとブレンドモードの衝突)、F-1(glTFとの軸整合)、F-6(テキスト基盤の分界)。

## 方針([concept.md](../concept.md)の2.5D/3D合成スコープをC-4の割り切りで具体化)

- v1の3Dは「3Dシーンエディタ」ではなく、**カメラ文脈を参照できるレイヤーソース**として扱う。点群/メッシュ等の3D系LayerSourcePluginは、共有`CompCamera`とレイヤー配置パラメータを入力に、内部で投影・ラスタライズ・必要なZバッファ解決を行い、最終的にpremultiplied RGBAの2Dテクスチャを返す
- **合成順序はレイヤー(タイムライン)順が正、Zはカメラ投影(変形)にのみ使う**。レイヤー間の深度合成はしない。メッシュ/点群内部の深度のみ3D系レイヤーソース内で解決する
- **軸整合(F-1)**: 世界座標は正準座標系と同じ**Y-up・原点中央**(glTFのY-upと一致し、変換なしで同居)。2Dレイヤーは「Z=0のXY平面に置かれた高さ1.0のクワッド」として世界に置く。正準空間がY-upなので2D↔3Dで軸反転が発生しない(テクスチャのV原点等、ラスター側の上下はmotolii-gpu内部で吸収し、パラメータ空間には漏らさない)
- カメラ文脈はコンポ全体で共有する。v1の`CompCamera`は単一(位置・注視点・FOV・roll程度)で、レイヤーごとのカメラ、カメラ切替、グループ内カメラ、複数ビューは作らない
- ユーザーが直接触る3Dレイヤー側の値は「素材の配置」に限定する。位置/スケール/回転/奥行き/点サイズ等のモーショングラフィック的なパラメータに留め、オブジェクト階層・ライト・コレクション・制約・レンダーレイヤー等のBlender/Nuke的な概念はv1に入れない
- ライティングはunlit(必要になったら固定1灯)
- OBJはglTF変換パスで受ける(内部はglTFのみ)
- 被写界深度はZ距離ベースのポストブラーで代用
- ポストプロセスは通常のRenderNodeとして実装(専用機構を作らない)

## カメラ文脈とLayerSourcePluginの分界

3Dをプラグインへ寄せる目的は、コアを3Dアプリ化させず、2Dモーショングラフィックの操作感を保つこと。したがって、v1では次の分界を守る。

- コア(Document/Render): `CompCamera`、正準座標系、レイヤー順、グループ/マスク/エフェクトの評価順、Quality、出力`FrameDesc`を所有する
- 3D系LayerSourcePlugin: `t`、`Quality`、`CompCamera`、レイヤー配置パラメータ、出力`FrameDesc`を受け取り、GPU上でRGBAテクスチャへ描く
- プラグインはレイヤー内部のZバッファや点群ソートを使ってよいが、他レイヤーの深度情報を要求しない。他レイヤーとの関係は、出力後の通常Compositeだけで決まる
- プラグインは決定論的であること。`render_frame(t, Quality)`の入力から同じ出力を返し、前フレーム状態に依存するシミュレーションを持たない
- 将来、厳密なレイヤー間深度遮蔽や複数カメラが必要になった場合は、v1のLayerSourcePlugin境界とは別の拡張として扱う

## 動的シミュレーション(物理演算)について — M5固有の補足

**正本は[concept.md](../concept.md)の根本コンセプト「馬鹿正直にシミュレートしない」と、その横断決定「逐次依存シミュレーションは持たない」**(いずれも2026-07-10)。[B-5](../pitfalls-and-roadmap.md)・[M4](M4-cache-and-analysis.md)と同じ2D合成パイプライン上の制約である。ここでは3Dレイヤー文脈だけ補足する。

- **3Dは「別エンジン」ではない**: v1の3Dは上記のとおり**レイヤーソース→2Dテクスチャ**の経路。Blenderの布/液体/パーティクル/ソフトボディの例は「外部ツールが重い理由」の**たとえ**であり、「3D物理だけ特別に禁じる」境界線ではない。2Dパーティクル・2D剛体・オプティカルフローも同じく逐次依存として扱う
- **M5で許す3D**: 「時刻t → 頂点/ピクセル」が純関数で決まるもの(静的メッシュ、キーフレーム変形、ベイク済み頂点アニメの再生、剛体トランスフォーム)。ラスタライズ自体はGPUの得意分野
- **ベイク持ち込み**: 他ツールでシミュレーションし、頂点アニメ付きglTF等の**焼き込み済みシーケンス**として読み込む。うちは再生のみ(ステッパなし)
- **キャッシュ**: 3Dレイヤーも他レイヤーと同じ[M4](M4-cache-and-analysis.md)機構(ノードID×時間区間×パラメータハッシュ)に乗る。3D専用キャッシュは作らない。Draft品質で回し、確定後に区間ベイク(M4 K7)

## タスク分割(粗案)

| ID | 内容 | 依存 | 完了条件(概要) |
|---|---|---|---|
| P1 | glTF読み込み(メッシュ・マテリアル最小限)+ OBJ→glTF変換パス | 凍結ゲート | サンプルアセットの読み込みゴールデンテスト |
| P2 | 3D系LayerSourcePlugin境界: `CompCamera`を参照し、メッシュ/点群/動画テクスチャ平面をpremultiplied RGBAへレンダ(レイヤー単位に1パス) | P1 | Zを持つ平面レイヤーがカメラ移動で視差し、出力後は通常Compositeで2Dレイヤーと合成されるゴールデンテスト |
| P3 | `CompCamera`のドキュメントモデル統合(キーフレーム可能な共有カメラパラメータ) | P2, M2-D1 | カメラアニメーションのE2Eテスト。レイヤーごとのカメラを持てないこともスキーマ上確認 |
| P4 | ポストプロセスNode群: ブラー(+Z距離マスク)、色調整(リフト/ガンマ/ゲイン)、グレイン | 凍結ゲート | 各ノードのゴールデンテスト |

| P6 | motolii-text: 最小テキスト基盤(F-6)— フォント列挙/読み込み(fontdb系)・フォールバック解決・シェーピング(rustybuzz)・グリフ→ベクターパス出力(Vello接続)。組版(縦書き・ルビ・行組・歌詞タイミング)はプラグイン側 | 凍結ゲート | 日本語文字列(かな漢字混在・約物)のシェーピング→パス描画のゴールデンテスト。フォールバック(欠字時の代替フォント解決)テスト |
