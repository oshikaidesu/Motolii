# M3前UIギャップ調査: U1〜U8に席が無いUI要素とコア側前提の欠落(2026-07-16)

ステータス: **調査メモ**([reviews/README.md](README.md)規律を適用。本書の結論をそのまま設計根拠・着手許可にしない。採否は[M2基盤再締結ゲート](2026-07-15-m2-foundation-reclosure-gate.md)解除後のM3入場PR起草時に、最新mainへ再翻訳した上で判定語付きで行う)

基準: main `17211d1`。UIに単一の正解は無いため、本書は「良し悪し」ではなく**割当の欠落**(どのタスク・台帳にも席が無い、または文書間で矛盾している)だけを列挙する。

## これは何か

M3入場PRの起草前に、次の3系統を突合した:

- (a) [M3仕様](../specs/M3-ui-integration.md)のU1〜U8タスク表と実装ガード11項目
- (b) 既存台帳: [backlog GAP群](../backlog.md)・[M3/M4ゲート台帳](2026-07-12-M3-M4-gate-ledger.md)・[実装準備台帳](2026-07-15-implementation-readiness-ledger.md)
- (c) コアの現物コード(motolii-doc / gpu / plugin / eval / audio / media / ui、spikes/s1-slint、CI)

分類は3種: **A=文書間の矛盾**(README「矛盾はバグ」対象)、**B=U1〜U8に席が無いUI領域**、**C=U発注前に必要なコア側前提の欠落**。既に台帳へ登録済みのものは新規発見と区別して「席」欄に明記する。

## C. U発注前に必要なコア側前提の欠落

| ID | 内容 | 現物 | 影響 |

| C-1 | **コア→UIの状態購読機構が無い**。差分通知・observer・イベントは存在せず、あるのは編集世代`revision: u64`と、`SetBpm`のみのプレースホルダ`WriterMessage`。「snapshot全体差し替え+revision比較」か「差分イベント」かの設計決定が無いままU2を発注できない | crates/motolii-doc/src/lib.rs:157,169-175 | U2完了条件「状態購読(コア→UI)」/ U3の大規模タイムライン再描画 |

## D. 既知の未決事項(文書に明示済み — 再掲のみ、本書で新判断しない)

- 深いIME実機チェックリスト(M3ガード1)/ 起動時間・アイドルメモリの数値(ガード10、U1実測待ち)/ a11y範囲の明示決定(ガード11)/ マルチモニタスパイク(ガード9)/ OpenCut棚卸し(U3着手時)
- GAP-13採否手順(能力コーパス→AM実機→判定語→採用時はM3§拡張方式同時改訂+解凍手続き)
- CompCamera採否(再締結ゲートA.2)/ GAP-6/GAP-3/GAP-7 / Slint live-preview評価(INF-8非ブロッキング)
- U1f/U2f/U2gの可変UI要素(scrim濃度・modifier物理key・socket形状等)

## E. 意図的スコープ外(本書で再提起しない)

- DPI `scale_factor`契約・`.slint`バージョン交渉(ゲート台帳が「M3へ追加しない」と明示)
- 空間モーションパス(concept: v1コア外)/ ~~可変ドッキング(M3方針: 固定分割)~~ → 2026-07-18 egui採用で組み込みpreset+利用者可変panelへ変更 / ビート自動検出(concept: やらない)/ 多トラック音声(concept: 楽曲1本のみ)

## F. 提案(採否はM3入場PR/仕様改訂PRで判定語付きで行う)

1. **B群の各項目に「席を作る」か「v1非目標と明記する」かをM3入場PRで割り当てる**。ガード11の「やらない範囲は『やらない』と書く」をa11y以外にも一般化する(特にB-1/B-2/B-3のアプリ枠UIは、席が無いまま実装が始まると各Uタスクへ散在する)
2. **GAP-13採否と`ParamDef`拡張(C-2)をU4発注の直列前提として扱う**(AGENTS.md「未決事項に依存するタスクに着手しない」の適用)
3. **C-1の購読形式(snapshot+revision比較 or 差分イベント)をU2発注書に設計決定として明記する**
4. **B-11の二重台帳はM3入場PRの再翻訳で一本化し、M3仕様のU表から実装準備台帳へ相互参照を張る**

本書は調査第一陣であり、反例探索(「席が無い」ことの見落とし)は網羅していない。ゲート・仕様へ採用する際は反対側レビューを併読すること(reviews/README.md 規律2/6)。
