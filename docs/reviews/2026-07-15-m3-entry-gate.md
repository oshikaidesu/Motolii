# M3入場条件(2026-07-15)

## これは何か

[M2入場条件](2026-07-11-M2-entry-gate.md)が「Documentスキーマ並列の前に塞ぐもの」だったのに対し、これは「**M3 UIタスク(U0V〜U8)を段階発注する前に塞ぐもの**」。M3が怖い理由は3つの重なり — ①**審判の弱さ**(UIの正しさはピクセルテストだけでは足りない) ②**乗算穴**(Slint API染み×移行費用、カスタムUI契約×プラグイン数) ③**準恒久UX**(ショートカット/キーマップはユーザーの筋肉記憶)。

**性格**: 本ゲートは[候補台帳](2026-07-12-M3-M4-gate-ledger.md)のM3行を**最新mainで再点検**して翻訳したもの。**新規リスクの追加はない**。解消済み項目は理由付きで落とす。満たせない事情が出たら黙って緩めず本ファイルを改訂する(H-3)。

**段階発注可の意味**: 仕様ステータスを「段階発注可」にし、**タスク表の依存と本ゲートの区分を満たした行だけ**着手する。ゲート全文達成をU0Vの前提にはしない。

### タスク別の着手条件(要約)

| タスク | 着手に要るもの | 備考 |
|---|---|---|
| **U0V** | S1(INF-1合格)、[ui-visual-language](../ui-visual-language.md) | ゲート必須の完了を待たない |
| **U2** | M2-D2完了 + **本ゲート「必須」全項目** | IPCは依存方向違反が早期に積もる |

**後続UI-PROTOTYPE**(製品タスクIDではない): [実装準備台帳](2026-07-15-implementation-readiness-ledger.md)の **U1f / U2f / U2g** は、U1/U2/U3の製品実装が成立した後の交換可能プロトタイプ。**U0c/U0d等の未定義IDを仕様に発明しない** — 依存が必要なら「別PRでタスク定義」と明記([M3仕様](../specs/M3-ui-integration.md)タスク表脚注)。

## ゲートが止める対象

- **必須**: U2およびU2後に並列するU3/U4/U6の着手前。**U0Vは対象外**
- **対象外(前倒しスパイク)**: M3実装ガード1(IME、[#56](https://github.com/oshikaidesu/Motolii/issues/56))・ガード2(タイムライン描画ベンチ、[#57](https://github.com/oshikaidesu/Motolii/issues/57)) — 依存ゼロで`spikes/`に完結、M2と並走可
- **対象外(期間中並走)**: 台帳「M2期間中に消化する」のTM/CQ/LG行 — ゲートで発注停止しないが、放置するとゲートが重くなる

## 記入規則(M2ゲートから継承)

- 完了したらチェックを入れ、**コミットSHA(またはPR番号)とテスト名を併記**

- 判定方式: **[自動]**=CI/テスト、**[レビュー]**=文書マージ・存在確認

## 宣言方針

1. **U0Vはゲート待ちしない** — 視覚言語とモックが正本になった時点で発注可
2. **U2/U3/U4/U6はM3E-1〜3 + M3E-9(文書)を必須** — 特にM3E-1(依存方向)は並列前に必須
3. **U1/U5は上記 + M3E-4〜8 + M2-D5** — Transport未実装の間はU1のTransport非接続部分のみ可
4. 全必須項目に証跡が揃ったら冒頭ステータスを「**達成**」にし、残りUIタスクの並列を明示解禁する
