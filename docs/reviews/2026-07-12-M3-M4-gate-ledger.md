## これは何か

[M2入場条件](2026-07-11-M2-entry-gate.md)の「入場条件に含めない」表は、GR-2〜GR-7 / CQ-2/CQ-4〜CQ-8 / LG群の行き先を「各フェーズの入場条件または期間中チケットへ」と先送りしていた。本台帳はその**割当を確定**し、M2消化中に既知所見が散逸しないようにする。M2ゲートと同じ性格を保つ — **新規リスクの追加はなく**、全項目が[実コード監査](2026-07-11-code-audit-pre-m2.md)・各仕様書・[backlog](../backlog.md)へ逆引きできる。本台帳で新たに決めたのは**割当**のみ(判断を要した箇所は文中に理由を書く)。

- 監査所見はf020ec8基準。M2期間中にコードは大きく動くため、**ゲート本文の起草時に全項目のfile:lineと存否を最新mainで再点検する**(M2ゲート記入規則と同じ)
- 台帳の項目は候補。起草時に「解消済み」「M2で形が変わった」で落ちるのは正常(落とす時は理由を残す)
- なおGR-6(ffmpeg stderrドレイン)は**PR #30で解消済み**(監査冒頭の追記)。GR-1の出力エイリアスもIssue #16で部分対処済み(プール化は未了)

## 次の「M」は一直線ではない(ゲートが止める対象の先取り)

M2完了後、M3とM4はカレンダー上のフェーズではなくタスク集合として交錯する:

- M3-U1はM2-D5(Transport)依存、U2はM2-D2(コマンド)依存
- M4-K1はM2-D8(所有権)依存、**K4はM2-D1のみ**、K6(SVG/Vello)はM1-T7のみ
- したがってM4-K4/K6はM3より先に発注可能なケースがある。ゲートは「フェーズの門」ではなく「タスク集合の門」として設計する(M2ゲートでD4/D6を対象外にしたのと同型)

想定するゲートの対象:

- **M3ゲート候補**: U1〜U8(全UIタスク)。旧案ではM3実装ガード1/2の2スパイクを対象外としたが、M2基盤再締結ゲート発効中はこの例外を無効とする
- **M4ゲート**: K1/K2/K7(キー・並行契約系)+**K4**(source_id内容指紋という恒久カタログを持つため対象に含める)。**対象外=K6** — キャッシュ非依存。ただしCQ-7(straight→premulアダプタ)だけはK6の先行条件

## D発注書に足す1文(M2ゲートの積み残し疑い2点)

ゲート改訂ではなく、該当タスクの発注書(M2仕様)への追記で足りる:

1. **D1: CompCameraをDocumentスキーマに入れない**(入れる判断をするなら単位宣言=CQ-5が先)。現D1タスク表にカメラは無いが、「コンポ設定」を書くエージェントが自然に足し得る(出典: 監査C-7「M2が確定前にスキーマへ焼く」)
2. **D3: 評価時刻の意味論を宣言** — 「v1=評価時刻はソースPTS(タイムライン=ソースの縮退)。M2でtimeline_timeに再定義」の1文(出典: 監査T-9/LG-3)。M2E-11の宣言5点に**含まれていない**。D3が暗黙に再定義すると、同じに見えて評価時刻が違うゴールデンが焼かれる

## M3入場条件の候補

M2ゲートの3軸(審判/乗算/恒久)をM3に写像する:

- ①**審判**: UIの正しさはピクセルテストで判定できない。M3の審判=依存方向CI+ベンチ数値+操作列プロパティテスト(M3ガード7)+観測HUD(ガード8)
- ②**乗算穴**: Slint APIが染みた行数×移行費用(Ardour/YTKの教訓=ガード4)。加えて`.slint`カスタムパネル契約は**プラグイン境界そのもの**で、プラグイン数で乗算する
- ③**恒久物**: ショートカット/既定UXはユーザーの筋肉記憶として、キーマップファイルはユーザーデータとして恒久化する

| 出典 | 候補 | ゲートに置く理由 |
|---|---|---|
| M3ガード4 | 依存方向CI: slint依存をuiクレートに限定(負例込み) | 審判の要。U並列発注の前に無いと、違反が積もってから気づく(M2E-1と同型) |
| GR-2 | GpuCtx起源タグ(Headless/UiShared)+`poll(Wait)`型ガード | U1でUI共有デバイス初登場。誤用症状=UI数百msフリーズは原因特定が最悪 |

| G-1契約 | `RenderedFrame`寿命契約の明文化(「次の`render_graph_cached`で無効」+U1は表示前コピー)**または**GR-1プールの前倒し | U1のtearingは検証エラーが出ない意味的破壊。GR-1(M4割当)を待つなら、せめて契約+コピー設計をU1発注書に |

| INF-2 | 性能ハーネス(ベンチ受け皿+起動/アイドルメモリの計測枠) | M3ガード2/10の完了条件が数値を要求する。審判系はU発注前(M2E-1と同型) |

## M4入場条件の候補

3軸の写像:

- ①**審判**: キャッシュのバグはM2ジャーナルと同じ「後から出る不可視破損」。審判=キー網羅の変異テスト(M4ガード1)+破損透明性テスト(ガード5)+平衡/リークテスト(ガード9)+loom
- ②**乗算穴**: 完全性原則「API口1つ=キー寄与1つ」(仕様方針・凍結API審査項目)。テクスチャ生成箇所の分散
- ③**恒久物**: ディスクキャッシュ形式・永続キー(再オープン再利用=ガード12)・source_id内容指紋(ガード2)は**ユーザーのディスクに残る**。Documentと同格の恒久性で、逃げ道はグローバルsalt(ガード3)のみ

| 出典 | 候補 | ゲートに置く理由 |
|---|---|---|
| GR-1 | RenderSession中間テクスチャのrefcountプール化(K1ハンドル形状の前倒し) | M2ゲートが名指しでM4入場候補と予約済み。M2-D3着手時の再評価結果をここに合流 |
| GR-3 | `GpuCtx::create_texture`一本化(VRAM計上カウンタの席) | K1予算判定の前提。生成箇所が5→Nに散る前が最安(M2期間中の前倒しも可) |
| TM-7 | Value正準ハッシュ契約+`keys_in(range)`/`next_key_after`シグネチャ | K1キー材料。未定義だとM4エージェントがf64ビットハッシュを再発明(-0.0/NaN散在) |
| TM-5 | TimeMap Eq/Hash正準化 | M2E-16(f)が確定したのは**受理範囲**のみ。正準性(2/1≡4/2)はキャッシュキーの偽ミス防止に残っている |
| CQ-6 | FrameDescにPAR追加(または構築関数経由の徹底)**【解凍手続き対象】** | キー完全性。キー実装後のフィールド追加=全キー無効化+serde移行 |
| CQ-7 | straight→premul単一アダプタ(Vello受け皿) | K6(SVG)とM5テキストの2チームが別々に境界変換を書く前。**K6の先行条件** |
| LG-2 | Encoder Drop警告+YuvToRgba保持契約のM4仕様(K4/K1)転記 | 呼び出し元が増えるM4ジョブ化の前。`render_frame`入口注記はM3仕様側へ |
| INF-5 | loom(または等価)をworkspace依存に追加+並行契約の模範例1本 | M2E-6(proptest)と同じ理屈: ツールの独断選定はH-3の分裂を生む。K1完了条件のストレステストの土台 |
| M4ガード1 | キー網羅変異テストのハーネス形の確認 | 仕様に記載済み。ゲートでは「K1発注書に完了条件として残っているか」の存在確認のみ |

**注**: メディア再リンク/可搬性(GAP-3/7)とGPUベンダ差運用(INF-3)の先例調査は[2026-07-12調査](2026-07-12-prior-art-gap-survey.md)で検証生存ゼロのまま未カバー — **M4ゲート招集前に専用の再調査ラウンド**を行う(同調査「未カバー領域」参照)。
