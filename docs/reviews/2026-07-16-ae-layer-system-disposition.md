# AEレイヤー方式への処置台帳と出戻り一次声調査

作成日: 2026-07-16(ユーザー議論に基づく)

## 目的

「AEのレイヤー方式にMotoliiは何と答えるか」を1枚に台帳化する。対象は (1) カット時のレイヤー複製 (2) グループ化 (3) タイムライン可読性 (4) AEユーザー互換の要否。既決事項の再宣言・唯一の未決点・M3採択候補・出戻り一次声調査に仕分けし、**新規の発明を採らない**(発注前の判定はレビュー規律6点に従う)。

## 0. 問題の再定義(2026-07-16 ユーザー)

- AEの「分割で上にレイヤーが複製される」挙動そのものは本質的な問題ではない(慣れで吸収されている。§4bの声とも整合)
- 本当の問題は次の2つ: **(a) AEがグループ化に適切な答えを出せなかったこと**、**(b) タイムラインの可読性**(全レイヤーバーが均一で、構造が視覚に現れない)
- 恐れるべきは「AE方式 vs トラック方式」の二者択一に見えること。AEユーザーの歴史の厚みは無視できない

## 1. 既決事項の再宣言(新規決定なし。正本リンクのみ)

| 問い | 答え | 正本 |
|---|---|---|
| カットで行(レイヤー)が増えるか | **増えない**。Track=順次クリップ列であり、分割は同一トラック内のその場分割(1クリップ→隣接2クリップ)。AEの複製挙動は「1レイヤー=1ソース」というデータモデルの副作用であり、思想ではない | A4決定(OTIO採択)= [decision-pack-adoption](2026-07-13-decision-pack-adoption.md)、D1aスキーマ(`Track { items: Vec<TrackItem> }`) |
| グループ化 | **その場で畳める1枚のレイヤー**。GroupはClipと同格の`ItemEnvelope`を持ち、エフェクトは子を合成した1枚に適用(AM式)。プリコンポ(別タイムライン分離)は作らない | [concept.md](../concept.md)「プリコンポは作らない」(2026-07-10) |
| 由来が同じ映像+音声 | **最初から1本**。`ClipSource::Asset`が video/audio component を1 Clip内に持つ(AG-1)。分離は明示コマンド(`plan_detach_audio`)で、分離後は完全独立(同期の隠れリンクを作らない)。AviUtlの自動グループが「別オブジェクト+リンク」で建てた能力を、構造で再建した形 | [audio-generalization-design](2026-07-14-audio-generalization-design.md) §4.3–4.4 |
| AEとの互換性の階層 | **データモデルの層では上位互換が文字通り成立**: 「1レイヤー=1クリップ」は「クリップを1個しか置かないトラック」という退化ケースであり、AEで組める配置は全表現可(逆は不可)。二者択一はスキーマには存在せず、選択が実在するのは**ジェスチャの既定値**(分割の配置先等)の層のみ | 本文書§3、[M3実装ガード5](../specs/M3-ui-integration.md)(既定=業界標準・革新はオプトイン・ショートカット初日からカスタマイズ可) |

> **A4(同一レーン内の時間重なり禁止)はTrack限定。Groupの子の時間重なりは合法であり、リスト順=重ね順(F-3ステップ5の通り)。**

- スキーマ変更なし・serde影響なし。反映は3点のみ: (1) [M2仕様](../specs/M2-document-model.md)「方針」への1文 (2) `audio_edit.rs:37`コメントの「(または同一Group)」削除 (3) 将来のA4 validate実装の対象宣言

## 4. 出戻り・残留の一次声調査(2026-07-16、3系統並列)

### 4.0 調査の限界(必読)

本環境ではReddit・HN・各ページの直接取得がプロキシで遮断され、**根拠はWebSearchスニペット経由**(引用は検索結果内に原文表示されたもののみ、それ以外は要旨と明記して収集)。reviews/README.md運用注の恒久出典基準(公式文書・バグトラッカ)を満たさない出典を多く含むため、規律1・3に従い**本調査の結論を単独で設計根拠にしない**。用途は既存決定の追認と優先度判断の材料に限定し、全結論は「**仮説と整合する事例**」に留める。

### 4a. 英語圏: 代替ツールを試して→AEへ戻った/離れられなかった声(18件)

理由分類の頻度: **能力の穴 7**(Resolve/Fusion=レイヤー式コンポジット不在、Cavalry=プラグイン生態系・リギング、Rive=動画書き出し)、**筋肉記憶/学習コスト 6**、**クライアント/チーム互換 5**(.aep納品要求)、**生態系/資産 4**、**代替側のUI/UX 3**(Fusionのノード忌避、Riveの未成熟UI)、**式(エクスプレッション) 0**。

代表声: 「100% Resolveに移行したいが、レイヤー式コンポジットが無いためAdobeに留まらざるを得ない」(Blackmagic公式フォーラム)/「AEキラーは全部潰れるか、機能が1/4しかない」(X, 2024)/「締切が迫るとき、筋肉記憶は強力だ」(Envato)。唯一の明確な脱出成功例(Intercom→Rive, 2024)は成果物がAEの納品領域(動画ファイル)の外だったことが成立条件。

### 4b. 英語圏: 「嫌いだが離れられない」+UI不満の解像度(20件)

- **グループ化**: 「プリコンポなしのレイヤーグループ」はAdobe公式リクエストで**2009年から続き300票超**。スレッド内の伝聞では、Adobe側が試みた上で「レイヤー/コンポ挙動がコードに深く焼き付いており不可能」と結論した経緯が語られる。「ネストコンポを開くと現在のコンポの文脈を失う」「タイムラインがレイヤーの壁(wall of layers)に変異する」(Mt. Mograph)が定番の言語化
- **分割挙動**: 「カミソリツールが無い」「同じ行に2区間置けない」「Premiereの方が格段に速い」— 摩擦としては実在するが、怒りの筆頭ではない(§0の再定義と整合)
- **行の表示**: 「行の高さが固定で変更できない」不満が2025年まで継続。ただし「均一さが階層を壊す」という枠組みの声は未発見 — この枠組みは本プロジェクトの先鋭化であり、コミュニティ需要としては「壁・飽和・畳めない」の形で実在する
- **最も生々しい怒り**: グラフエディタ(「rubbish」「stuck in the late 90s」)— 区間イージング方式の既決([concept.md](../concept.md))と整合する事例

### 4c. 日本語圏: AviUtl⇔AE等の移行の声(21件)

理由分類の頻度: **操作の慣れ 10**(最頻。「AviUtlに慣れきると他ソフトになじめず戻ってくる」が定型)、**機能の穴 9**(音合わせ/一音単位トリミング、BPMグリッド、グリッド左右反転、PSDToolKit立ち絵、エクスプレッションのプリセット保存)、**無料プラグイン/スクリプト資産 6**、**重さ・価格 6**(AE本体+有料プラグイン数万〜十数万円)、**UI/見やすさ 4**。

支配的帰結: **全面乗り換えの失敗→用途別併用への着地**。AEへ移行完遂した音MAD/YTPMV作者ですら「音合わせ工程だけAviUtlに戻る」部分回帰が半ば標準化(「一音一音に合わせる作業はAEでは面倒、AviUtlの方が格段に楽」「AEにはBPMグリッドがない」)。AviUtl→AviUtl2ですら旧32bitプラグイン資産の非互換が移行を年単位で遅らせている。

### 4d. 読み(いずれも「整合する事例」。単独で設計根拠にしない)

1. **出戻り要因にUIの流儀は実質不在**: 「代替のタイムラインがAEと違ったから戻った」声は両言語圏で0件。AE式ジェスチャ互換は獲得・残留の戦略軸ではない(§3の【縮小】判定の根拠)
2. **グループ化と壁の解消は、AE側が構造的に直せないと自認した領域**: 既決のその場グループ方式は、15年・300票超の文書化された需要に正面から当たり、かつAEには模倣困難 — 差別化点として希少
3. **音合わせ・BPMグリッドは日本語圏で実証済みの侵入口**: AE移行者すら当該工程だけ出戻る = 既決の楽曲波形+BPMグリッド基準線([concept.md](../concept.md)/[M2 A6](../specs/M2-document-model.md))が狙う工程そのもの
4. **最強のロックイン(.aep納品・チーム互換)はMV特化が構造的に回避**: Motoliiの成果物=最終mp4であり、Intercom→Rive成功例と同型の脱出経路にいる(対象選定の追認)
5. **プラグイン/スクリプト資産が両言語圏共通の最強残留要因**: 低障壁プラグイン境界+LLM authoring路線([ae-pain-points](../ae-pain-points.md)二枚看板)と整合
6. **注記**: [ae-pain-points C-補遺](../ae-pain-points.md)の一次聞き取り(「式があるからAEに戻る」)に対し、本調査の英語圏スニペット検索では式を出戻り理由とする声が0件。直接聞き取りとスニペット検索の感度差であり既存決定(式の能力カバレッジ)を覆さないが、乖離として記録する

## 6. 先例追補(2026-07-16 同日後続議論)

### 6.1 仮出力(ベイク)= DAWフリーズの語彙対応

concept.md既決「グループ化+仮出力」の性能面は、DAWフリーズと意味論が点対点で一致する(類推ではなく対応):

| DAWフリーズ | Motolii仮出力(既決) |
|---|---|
| トラックを音声にレンダしプラグインを休止 | グループ出力を時間範囲でキャッシュ(M4-K7) |
| 非破壊 — アンフリーズでいつでも編集へ | 編集で自動無効化(=自動アンフリーズ) |
| プロジェクトファイルの一部ではない | Documentに入らない(文書/キャッシュ分離) |
| フリーズ中もフェーダー/センドは生きる(安いつまみは触れる) | 既定ベイク点=**子合成直後・グループスタック適用前** — グループ側のつまみは高価な子ベイクを無効化しない |

反映候補: concept.mdの「映像制作におけるVST」対応表へ **Freeze = 仮出力(ベイク)** の行を追加(文言のみ・追加的)。プリコンポの役割分解では、仮出力がカバーするのは**性能**の役割であり、**再利用**は別(§5需要札)。

### 6.2 グループの先例系譜: 「整理の束ね」と「処理の束ね」を1物にする

| 先例 | 形 | 判定 |
|---|---|---|
| AviUtl | 4機構に分解(グループ化=移動バインド / グループ制御=位置カウントスコープの変形 / フレームバッファ=合成結果への迂回 / シーン=別タイムライン) | 反面: 位置カウントは行挿入で効き先が黙って変わる。フレームバッファの存在自体が「合成結果1枚に掛けたい」需要の証拠(per-child不採用の補強 — concept.md記録済み) |
| Cubase | フォルダトラック(整理・音は通らない)とGroup Channel(バス)が**別オブジェクト**、手動紐付け | 2つの平行構造をユーザーが同期し続ける形 — AviUtl分解と同族 |
| Logic | Track Stacks(2013)が folder/summing の**2モード** — UIは1つだが作成時に意味を選ばされる | 中間形 |
| **Ableton** | Group Track(Live 8) — **畳む=バス=エフェクト置き場が1オブジェクト・1ジェスチャ**。ただしネスト許可はLive 11まで約12年を要した(fold=バス束ねの代償) | **採択済みの形と同型**: MotoliiのGroup(fold+ItemEnvelope+常に分離合成)はAbleton型。ネストの代償は「中間テクスチャはエフェクト/マスク保持時のみ生成(スキーマ外の実装詳細)」の既決が構造的に回避 |

エフェクト付与点の帰結(既決の系): 付与は`ItemEnvelope`のみ(Trackは意図的に裸)。「レーンにエフェクト」はレーングループ化のUI糖衣(concept.md調整レイヤー決定の一例)で受け、レーン内無重なり(A4)によりグループのエフェクトは各クリップが自分の時間帯にチェーンを通過=DAWチャンネルストリップと同挙動。散在クリップへの同一エフェクトはShared Effect(D1l)。粒度階層: シェイプ(PathOpのみ) < クリップ < グループ(再帰) < レーングループ(糖衣) < ルートグループ。ピクセルエフェクトのシェイプ単位スコープはv2(D1i-1)。
