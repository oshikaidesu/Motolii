# 先例調査: 「2Dレイヤー順合成 × 3D深度合成」の境界を各ツールがどう切っているか(2026-07-14)

ステータス: **調査メモ。反対側レビュー前。設計根拠にしないこと**（[規律6点](README.md)の1・2・3・6項）。所見は「一次資料で確認できた範囲」と「仮説と整合する事例」を区別して書く。「裏付けられた」「証明された」は書かない。判定語（採用/縮小/延期/棄却）は反対側レビュー後に付す。既存決定 [C-4](../pitfalls-and-roadmap.md)（合成順序はレイヤー順を正、Zは投影のみ）・[M5仕様](../specs/M5-3d-and-post.md)（レイヤー間深度遮蔽は別拡張）は本メモで変更しない。

## これは何か

M5の割り切り（C-4: レイヤー間の深度合成をしない）と、その延長にある「明示3Dグループで深度合成をスコープする」という検討中の設計案について、**先行ツールがこの境界を実際にどう切っているか**を列挙した調査メモ。目的は「われわれの割り切り／グループ案に、先行例と食い違う穴がないか」の候補列挙であり、グループ深度合成をv1に入れる/入れないの判定はしない（[M5未決事項](../specs/M5-3d-and-post.md)「厳密なレイヤー間深度遮蔽は別拡張」は据え置き）。

調査方法: 多系統Webサーチ + ソース取得 + 主張ごとの敵対的検証ワークフローで素材を集め、そのうえで**一次資料URLに当たれたものだけを本メモの所見に採用**した（[運用注](README.md): 調査ワークフローの「検証済み」申告自体は根拠にしない）。二次資料（個人ブログ）は、一次資料で挙動が確認できる主張の補足としてのみ引用し、その旨を明記する。

## 問い1: 各ツールは 2D順 と 3D深度 の境界をどう定義しているか

### D-1. Apple Motion — 「2Dグループ／3Dグループ」の明示型トグルで境界を切る（一次資料で確認）

- Motionはグループ単位で2D/3Dを切り替える。**2Dグループの子はレイヤーリスト順で合成し、Z位置によらず深度ソートも交差もしない**。3Dグループの子はZ深度順で合成する。交差に参加させるにはグループを明示的に2D→3Dへ変換する必要がある。
- **3D同士が交差できるのはレイヤーリスト上で連続している場合だけ**。「3D objects in different groups to intersect, the layers ... must be contiguous in the Layers list」——間に別レイヤーが挟まると交差ランが分断される。
- 3Dグループが2Dグループの内側にネストされると自動で平坦化され、カメラを無視する1枚のカードとして振る舞い、他グループのレイヤーと交差しない。3Dオブジェクトが非3Dレイヤーと接するときは**リスト順が勝ち、Z位置は描画順に影響しない**。
- 3Dグループには **`Layer Order` チェックボックス**という脱出ハッチがあり、これを入れるとグループ内をZ順でなくリスト順で合成する。
- 出典（一次資料で確認）: [2D and 3D group properties](https://support.apple.com/guide/motion/2d-and-3d-group-properties-motn2fb59f53/mac) / [3D object intersection and layer order](https://support.apple.com/guide/motion/3d-object-intersection-and-layer-order-motn5e2c432e/mac)

### D-2. After Effects (Classic 3D) — レイヤースタイル／挟まった2Dレイヤーが3Dランを分断する（挙動はAdobe一次資料で確認、「理由」の逐語はブログ）

- 3Dレイヤーに**レイヤースタイルを付けると、そのレイヤーは他の3Dレイヤーと交差できず、影の授受もできなくなる**。さらにそのレイヤー自体が3Dランのブレーカーになり、上下の3Dレイヤーが相互に交差・影を落とせなくなる（挟まった2Dレイヤー・調整レイヤーと同じ効果）。
- Adobeの説明として引用される「レイヤースタイルの複雑なブレンド演算は3D交差と両立しない」という**因果の逐語**は二次資料（ProVideo Coalition, Rich Young）由来。ページは直接fetchで403のため、逐語の帰属は**確度中**とする。ただし挙動自体（レイヤースタイル付き3Dレイヤーが交差・影を失い、ランを分断する）はAdobe公式ドキュメントで確認できる。
- これは**Classic 3Dレンダラの挙動**。2023+のAdvanced 3Dレンダラで同じ制約が残るかは**今回未確認**。
- 出典: [ProVideo Coalition（二次・確度中）](https://www.provideocoalition.com/the_limitation_of_layer_styles_for_3d/)、および挙動確認としてAdobe helpx（3Dレイヤー／ブレンドモード関連ページ）。

### D-3. ディープコンポジット (OpenEXR deep / Nuke) — 深度は厳密に解くが、演算子を `over` に限定する（一次資料で確認）

- ディープ画像はピクセルごとに深度タグ付きサンプルを任意個保持する（各サンプルは前後Z=`Z`/`ZBack`を持ち、体積的な奥行きも表現できる）。マージは順序非依存で前景/背景の区別すら不要——「for each output pixel, concatenate the sample lists」。合成順序をレイヤー順でなくサンプル深度に委ねられる。
- 平坦画像を得るにはピクセルを"tidy"にして（Zソート・非重複化）front-to-backで合成する。この**合成演算子は `over` のみ**。Nukeドキュメントは「Deep compositing only supports the over blend mode」と明言し、乗算/スクリーン等の任意ブレンドモードはディープモデルに存在しない。
- 出典（一次資料で確認）: [OpenEXR: Interpreting Deep Pixels](https://openexr.com/en/latest/InterpretingDeepPixels.html) / [OpenEXR wiki: Theory of Deep Samples](https://github.com/AcademySoftwareFoundation/openexr/wiki/The-Theory-of-OpenEXR-Deep-Samples)
- 補足（検証で反証された主張）: 「ディープサンプルは位置・順序に完全非依存で、深度方向に分割・移動しても結果RGBAが変わらない」は敵対的検証で否決（0-3）。体積サンプルの分割は無条件に自由ではないため、Stage 2でサンプル分割を実装する場合の注意点として記録する。

## 問い2: ブレンドモード × 深度ソートの衝突をどう扱うか

一次資料で確認できた範囲では、**この衝突を両立させたツールは今回の証拠に無い**。2つの妥協系が観測された:

- **A系（Motion / AE Classic 3D）= 平坦化逃避**: ブレンドモード・不透明度・大半のフィルタ・マスクを掛けると対象がラスタライズ（bitmap平坦化）され、3Dグループなら深度交差を失ってレイヤー順の1枚に降格する。Motionはこれを**UIに明示**する——ラスタライズを誘発したパラメータの横に**箱囲みの "R" インジケータ**を出す。「任意ブレンドは持てるが平坦なレイヤー順空間でだけ、深度交差は諦める」。出典（一次資料で確認）: [Motion: Intro to rasterization](https://support.apple.com/guide/motion/intro-to-rasterization-motne6b2ddf0/mac)
- **B系（ディープ / Nuke）= 演算子制限**: 深度順は per-sample で厳密に解くが、合成演算子を `over` に限定する。「深度は完璧だが乗算/スクリーンは持てない」。

つまり半透明×任意ブレンド×深度ソートは順序非可換で、**「深度的に正しい合成」と「任意ブレンドモード」を同時に満たした先行例は、今回の一次資料の範囲では観測されなかった**。各ツールはどちらの半分を捨てるかを選んでいる、という読み方ができる（反例未探索のため「整合する観測」に留める）。

## 問い3・4（gotcha / パーティクル回り込み）: 今回の証拠でカバーできていない

「2Dキャラの周りをパーティクルが前後に回り込む」ショットの実制作手順（前後マット分割 / Zソートパーティクルレイヤー / パーティクル単位深度のいずれか）を、一次資料で確認できた主張として**今回は取得できなかった**。実務上は前後マット分割が定番とされるが、本メモでは未確認として扱う。

## われわれの検討中の設計案との関係（仮説と整合する観測。判定はしない）

以下は「先行例と整合する観測」であり、設計の正当化ではない（規律6点の1・3）。判定は反対側レビュー後。

- **明示3Dグループでスコープする案は、Motionの2D/3Dグループトグルと同型の観測**がある。Motionはトップレベルの2D順を保ちつつ、深度合成をグループ境界の内側に閉じ込めている。これはC-4（トップレベルはレイヤー順が正）と整合する構造。
- **「レイヤーリストは並べ替えず、描画順はZから派生させる」という案は、Motionがリストを書き換えずZ順を描画時に導出している観測と整合**する。今回の証拠の範囲では、ドキュメントのレイヤーリスト自体を毎フレーム並べ替えるツールは観測されなかった。
- **境界を隠さずUIに出す（バッジ表示）案は、Motionの "R" ラスタライズインジケータと整合**する観測。AEのランブレーカーが暗黙であることとの対比は、反例未探索のため「観測された差」に留める。
- **Stage 2（ピクセル単位深度マージ）を `over` 限定にするという検討は、ディープコンポジットが `over` に限定している一次資料の観測と整合**する。これは好みでなく、深度合成側の先行実装が採った制限と同じ方向。

これらは既存のM5割り切り（C-4・レイヤー間深度遮蔽は別拡張）を**変更する材料ではなく、別拡張を検討する際の設計候補の裏付け候補**として記録する。採用可否は、下記の穴を埋めたうえで反対側レビューで判定する。

## 反対側レビューで問うべき点（[規律6点](README.md)の2項）

- **転移条件**: Motion/AEの「平坦化逃避」はブレンドと深度の両立を諦めた結果。われわれの2.5D割り切り（Zは投影のみ）はそもそも深度交差を持たないので、この失敗条件が転移するのは「明示3Dグループを足した後」だけ。足す前は無関係、という切り分けが正しいか。
- **より小さい対策**: グループ深度合成を入れずに、分割Z平面プラグイン（[前提: `render_frame(t)`決定論、B-4]）で回り込みの実需をどこまで満たせるか。境界をコアに焼かずプラグイン側に閉じる選択の評価。
- **因果**: 「Motionが許容されAEが嫌われるのは境界の明示/暗黙の差」という帰属は、反例（明示でも嫌われる/暗黙でも許容される事例）を探すまで「観測された差」に留める。
