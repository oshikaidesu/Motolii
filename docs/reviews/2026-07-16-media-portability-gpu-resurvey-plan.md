> 2026-07-23歴史監査: cutoff全1版を[Unit 5F回収](2026-07-23-historical-media-portability-gpu-resurvey-plan-recovery.md)で処分した。調査は現在も未実施で、質問や落とし穴仮説を採択済み所見として扱わない。ラウンド1はGAP-3／7とM4-K4の恒久な指紋・再リンク・package意味、ラウンド2はINF-3の実機Final再現性方針の前だけを止める。段階入場済みのK0や独立K1契約、既存GPU health検出を一括で巻き戻さない。

## これは何か

[2026-07-12先例調査](2026-07-12-prior-art-gap-survey.md)は、メディア再リンク/可搬性とGPUベンダ差運用の2領域で**検証を生き残ったクレームがゼロ**のまま終了し、[M3/M4ゲート台帳](2026-07-12-M3-M4-gate-ledger.md)は「M4ゲート招集前に専用の再調査ラウンド」を予約した。本書はその実施計画を固定する。

**期日が到来している**: M4ゲートの招集トリガー「M2-D8マージ時」は**PR #88(2026-07-13)で成立済み**。ラウンド1(GAP-3/7)はM4ゲート本文の起草前に完了させる。ラウンド2(INF-3)はINF-3着手前で足りるが、同じプロトコルで併走できる。

## 規律

[reviews/README.md](README.md)の規律6点と運用注(再確認可能な公開恒久文書のみを出典とする)を全面適用する。加えて本ラウンド固有の規約:

1. クレームは**3票検証**(独立した一次資料での裏付け。前回調査と同じ基準)を通過したものだけを「検証済み」と表記し、通過しなかった取得物は棄却理由つきで残す
2. 反対側レビューは調査本体と**別セッション**で行う
3. 「実例が存在する」ことを採用理由にしない。Motoliiの既決力学(source_id内容指紋、キャッシュ完全性原則、hard budget)を完成させる所見だけを採る — [ui-interaction-language.md](../ui-interaction-language.md)§3.1と同型のゲート

## ラウンド1: メディア再リンク/可搬性(GAP-3/7 → M4ゲート起草前)

設計材料の受け手: GAP-3(パス解決規約+欠落検出+再リンクモデル)、GAP-7(パッケージ化のスキーマ予約)、M4-K4(source_id内容指紋 — [M4実装ガード2](../specs/M4-cache-and-analysis.md)は既にmtime/パス排除を決定済み。本ラウンドはその外側=**Document側のパス解決と可搬性**を扱う)。

### 調査質問

| # | 質問 | 落とし穴仮説(検証対象であり結論ではない) |
|---|---|---|
| R1-1 | AE「ファイルを収集」の既知の欠落パターンは何か(フォント/プラグイン/プロキシ/キャッシュのどれを集め損ねるか) | 収集対象の列挙が機能追加へ追随せず、恒久的な「収集したのに開けない」苦情源になる |
| R1-2 | Premiere Project Managerの取り違え・除外の実例(consolidate/transcodeでの参照切れ、同名ファイル混同) | パスベース同一性が可搬化処理自体を壊す — K4のsource_id指紋が正しい理由の裏書きまたは反証 |
| R1-3 | FCPXライブラリ(素材内包モデル)の肥大・共有・部分納品の苦情構造 | 「全部内包」は再リンク問題を消す代わりにサイズ・共同作業・クラウド同期の問題へ転嫁する |
| R1-4 | Resolveアーカイブ/PostgreSQL DBモデルの移行失敗例 | プロジェクト正本がファイルでないと、バックアップ・diff・バージョン管理の全導線が専用UI頼みになる |
| R1-5 | 相対パス基準点の各社規約(プロジェクト基準/素材ルート基準/ドライブ文字・UNC・クラウド同期フォルダの扱い) | 基準点の意味論が未定義だと、同じDocumentがマシンごとに別の素材を指す |
| R1-6 | 再リンクUXの成立条件(一括再リンク、フォルダ推測、指紋照合の有無と精度) | 欠落時UIは「1件ずつダイアログ」だと実用不可 — 指紋があれば自動照合できるはず |

### 一次資料候補

community.adobe.com / helpx.adobe.com(AE収集・Premiere Project Manager KB)、Apple公式FCPXライブラリ管理白書+discussions.apple.com、Blackmagic公式マニュアル+forum.blackmagicdesign.com、OTIO/ALE等の交換形式におけるメディア参照仕様(F-5「OTIO-shaped」との整合確認)。

### 出口

所見→対処候補→割当先(GAP-3/7スキーマ予約、K4完了条件への追記、M4ゲート項目)の台帳形式。**スキーマへ席を予約するだけの判断**と**v1実装を要求する判断**を区別して書く。

## ラウンド2: GPUベンダ差・ドライバ運用(INF-3 → INF-3着手前)

設計材料の受け手: INF-3(golden=lavapipe固定、実機Final出力の許容/非再現の明文化)。前回調査で**有望な一次資料は特定済み**: wgpu公式のKnown Driver Issues台帳、Blender GPUモジュールのブラックリスト運用文書。

### 調査質問

| # | 質問 | 落とし穴仮説 |
|---|---|---|
| R2-1 | wgpu Known Driver Issuesの分類(ベンダ×バックエンド×症状)と、wgpu自体のworkaround方針 | 「wgpuが吸収してくれる」前提の穴を実台帳で確定する |
| R2-2 | Blenderのdriver blacklist/GPU要件運用(判定基準、ユーザー通知、fallback経路) | ブラックリストは保守コストが線形増 — 採るなら判定と通知の最小形を先例から |
| R2-3 | lavapipe(CI)と実機の既知の出力差の種類(丸め・補間・NaN処理・精度) | golden一致=正しさの審判が、実機では成り立たない範囲の明文化が必要 |
| R2-4 | 他ソフトの「Final出力のGPU非依存性」の公約実例(Resolve/AE/Blender Cyclesの決定論宣言の有無と範囲) | 「bit一致」をどこまで公約できるかの相場観。M4のFinal bit一致完了条件との整合 |
| R2-5 | ドライバ起因クラッシュ/device lostの運用(検出・診断メッセージ・ユーザー導線) | GR-5(check_health)の先の「ユーザーへ何を言うか」が未設計 |

### 出口

INF-3のdocs化(許容誤差の根拠つき)に必要な材料の台帳。**M4のFinal bit一致系完了条件(K1b/K7a/K8b)が「同一マシン・同一ドライバ内」の公約であることの明文化**を含むか判定する。
