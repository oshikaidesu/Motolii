# 現設計の落とし穴分析と実装ロードマップ

作成日: 2026-07-07
前提資料: `design-memo.md` (2026-07-05), `discussion-log-2026-07-06.md`

このドキュメントの目的: **現在の設計(Tauri + Rust + wgpu コア / WebView UI / 最小コア+全機能プラグイン / ノード依存キャッシュ / 2.5D統一シーングラフ / 並列エージェント開発)をこのまま実装した場合に踏む可能性が高い落とし穴**を洗い出し、それらを潰す順序としてロードマップを定義する。

---

## Part 1: 落とし穴カタログ

リスク度は「踏んだ場合の手戻りの大きさ × 踏む確率」の主観評価。

### A. アーキテクチャ境界の落とし穴

#### A-1. WebView UI ↔ Rust ネイティブ描画の橋渡し【リスク: 最高】

design-memoで「未解決」とされたまま残っている項目だが、**これは未解決事項の一つではなく、プロジェクト全体の成立可否を握る単一最大リスク**。Tauriには「Rust側でレンダリングしたテクスチャをWebViewにゼロコピーで見せる」標準機構が存在しない。

選択肢と各々の罠:

| 方式 | 内容 | 罠 |
|---|---|---|
| (a) ネイティブ子ウィンドウ重ね合わせ | WebViewに「穴」を開け、その下/上にwgpuのswapchainを持つ子ウィンドウを重ねる | いわゆるairspace問題。WebViewのDOM要素(モーダル、ドロップダウン、パネルリサイズ)がプレビューと重なれない。z順・DPIスケーリング・ウィンドウ移動追従がOSごとに別実装。macOS/Windows/Linuxで挙動が全部違う |
| (b) CPU読み戻し→IPCでWebViewへ転送 | wgpuでレンダ→`copy_texture_to_buffer`→map→UI側の`<canvas>`に描画 | 1080p RGBA = 約8MB/フレーム、60fpsで約500MB/s。TauriのデフォルトIPC(JSON文字列化)では論外。カスタムURIスキームハンドラ/共有メモリ/ローカルWebSocketなら1080p30は現実的だが、4Kは厳しい。GPU→CPU読み戻し自体もパイプラインストールを起こす |
| (c) プロセス間GPUテクスチャ共有 | DXGI shared handle (Win) / IOSurface (macOS) / dmabuf (Linux) でWebView側プロセスとテクスチャ共有 | 完全ゼロコピーだが、3プラットフォーム×WebView実装ごとの深い専門知識が必要。wgpuの外部メモリ対応はhalのescape hatch経由で不安定。実装コストが本体機能を圧迫する |
| (d) GPUI等でUIごとフルネイティブ化 | WebViewを捨てる(OpenCutデスクトップ版と同じ判断) | OpenCutのReactタイムライン流用が不可能になる。GPUIはZed以外での採用実績が少なくドキュメント薄。UI開発速度が大きく落ちる |

**対策**: 最初のコードを書く前に、(a)(b)を各2〜3日でスパイク実装して実測する。判断基準を「プレビュー1080p30がUI操作(パネル開閉・スクラブ)と共存できるか」に固定する。**MVPは(b)の割り切り(プレビューは1080p30・プロキシ解像度)で始め、(c)は必要になってから**が現実的。ここを決めずにコア実装を進めると、後からUI統合で全面手戻りになる。

#### A-2. 「最小コア+全機能プラグイン」を初日からやる罠【リスク: 高】

プラグインアーキテクチャの古典的な失敗パターン: **プラグインが何を必要とするか分かる前にプラグインABIを設計することになる**。1個目のプラグインを書くたびにABIに穴が見つかり、インターフェース変更→全プラグイン修正のループに入る。並列エージェント開発と組み合わさると、凍結したはずのtraitが凍結できず、全エージェントの作業が相互に壊れ続ける最悪の形で発現する。

さらにRust固有の罠:

- **RustにはABI安定性がない**。dylibプラグインは「ホストと完全に同一のコンパイラバージョン・同一のクレートバージョン」でビルドしない限りUB。`wgpu::Texture`のようなRust型をdylib境界を越えて渡すのは、同一ビルド環境を強制できる場合以外は成立しない。C ABI化(`#[repr(C)]` + ハンドルID方式)か`abi_stable`クレートが必要になり、設計コストが跳ね上がる。
- wgpuのリソースは生成元の`Device`に紐づく。プラグインが別のwgpuバージョンをリンクしていたら共有不能。

**対策**: プラグイン境界は**まず「同一バイナリ内のRust trait」として設計する**(静的リンク、Cargo featureで有効/無効)。動的ロード(dylib/C ABI)は垂直スライスが動いてtraitが3〜4個の実プラグインで検証された後に導入する。「プラグインベースの設計」と「動的ロード機構」は別物であり、前者は初日から、後者は後回しでよい。WASMサンドボックス(パラメータ系)も同様に、まずはネイティブ関数として書き、境界だけWASM化可能な形(構造化データのみ授受)を守る。

#### A-3. wgpu自体のAPI変動【リスク: 中】

discussion-logで認識済みだが具体策がない。wgpuは数ヶ月ごとに破壊的変更が入る。

**対策**: バージョンをworkspaceで一元ピン留めし、アップグレードは専用PRでまとめて行う。wgpuを抽象化する自前レイヤーを作るのは過剰設計なのでやらない(プラグインに見せる型だけフレーム記述子+ハンドルIDに限定しておけば、内部のwgpu更新は閉じ込められる)。

**改訂(2026-07-08、レビュー指摘#1)**: UI基盤にSlintを採用したため、**wgpuの更新ペースは自分では決められない**。Slintのwgpu統合(`unstable-wgpu-XX`)が対応するメジャーに全クレートを固定する(現在29)。wgpu型はクレートバージョンが違うと別型になり、コアとUIのバージョンが割れると「ゼロコピー埋め込み」(A-1の解決策)そのものが型エラーで成立しなくなるため、**バージョン分裂は禁止**(workspace依存で一元管理し、スパイク等の別workspaceも同メジャーを使う)。

### B. 映像処理そのものの泥沼

#### B-1. 音声が設計に一切存在しない【リスク: 高・設計漏れ】

**MV制作ツールなのに、design-memo/discussion-logのどこにも音声の記述がない。** 音楽に合わせて編集する用途では、音声波形表示・再生同期・(できればビート検出)は「あると嬉しい」ではなくコア要件のはず。音声を後付けすると:

- タイムラインの時間表現がフレーム番号ベースで固まった後に、サンプルレートベースの音声クロックと整合しなくなる(A/V同期のドリフト)
- プレビュー再生ループの主クロックを「映像フレーム」で作ってしまうと、音声途切れ・同期ズレの温床になる。**再生クロックの主は音声**(オーディオコールバック駆動)が定石

**対策**: 実装は後回しでよいが、**タイムラインの時間表現と再生アーキテクチャの設計には初日から音声を含める**。時間は浮動小数の秒ではなく有理数タイムスタンプ(FFmpegの`time_base`方式: 分子/分母のi64)で持つ。

#### B-2. 動画デコードの想定甘さ【リスク: 高】

「ネイティブffmpegを直接叩ける」の一行で済まされているが、ここには複数の罠がある:

- **Rustバインディングの保守状態**: `ffmpeg-next`等のバインディングはビルドが重く、FFmpegバージョン追従も不安定。加えてFFmpegを静的リンクするとLGPL/GPL・特許(H.264/HEVC)の配布問題を背負う。
- **シークの罠**: コーデックのシークはキーフレーム単位。フレーム正確なシークには「直前キーフレームへシーク→目的フレームまでデコード読み捨て」が必要で、スクラブ性能に直結する。
- **VFR(可変フレームレート)素材**: スマホ撮影動画はほぼVFR。フレーム番号ベースのタイムラインはVFR素材で音ズレ・フレームズレを起こす。
- **ハードウェアデコード**: NVDEC/VideoToolbox/D3D11VAとwgpuのゼロコピー連携は各OS固有の深い沼。最初から狙うと数ヶ月溶ける。

**対策**: MVPは**ffmpegをサイドカープロセスとして起動し、rawvideoをパイプで受け取る**方式にする(リンク不要、ライセンス問題回避、クラッシュ隔離、決定的)。スクラブ用にはプロキシ(低解像度中間ファイル)生成 or 全Iフレーム中間コーデックへの変換をインポート時に行う設計を最初から入れる。VFRはインポート時にCFR正規化。ハードウェアデコードはv2以降の最適化と明記してスコープから外す。

#### B-3. 色空間・YUVの事故【リスク: 中〜高】

フレーム記述子に色空間フィールドを持たせる方針(5-2)は正しいが、フィールドを「持っている」ことと「正しく変換している」ことは別:

- デコード出力はYUV(通常limited range / BT.709)。wgpuにはネイティブYUVサンプラがないため、**YUV→RGB変換シェーダを自前で書く**ことになる。limited/full rangeの取り違え・601/709係数の取り違えは「なんとなく白っぽい/色が薄い」という気づきにくいバグとして出る。
- sRGBテクスチャフォーマット(`Rgba8UnormSrgb`)とリニア合成の混在ミス。ブレンド・リサイズをsRGB空間でやると暗部が濁る。
- premultiplied alphaの取り違えは半透明の縁に黒フリンジとして出る。

**対策**: 既知のカラーパターン動画(カラーバー等)を使った**ゴールデンイメージテストをM1の時点で導入**する。「見た目でなんとなく合ってる」を禁止し、ピクセル値の数値比較で守る。10bit/HDRはv1スコープ外と明記。

#### B-4. プレビューと書き出しの分離が「分岐」になる罠【リスク: 中】

Remotion方式の分離は正しいが、素朴に実装すると「プレビュー用レンダパス」と「書き出し用レンダパス」が別コードになり、**書き出したら見た目が違う**という動画編集ツール最悪のバグを生む。

**対策**: レンダラは「時刻tと品質パラメータを受け取り決定的にフレームを返す」単一の純関数として実装し、プレビューと書き出しは**同じ関数の呼び出し方(解像度・スキップ・キャッシュ方針)だけが違う**構造にする。書き出しはフレーム列をffmpegサイドカーにパイプしてエンコード。

#### B-5. ノード単位キャッシュと時系列依存の衝突【リスク: 中】

「変更部分だけ再計算」のダーティフラグ設計は、**フレーム内で閉じる処理**には有効だが、本題のトラッキング解析は時系列依存(オプティカルフローはフレームN-1に依存、色トラッキングは全区間の解析)を持つ。パラメータを1つ変えた時に「どこからどこまで再解析が必要か」はフレーム単位のダーティフラグでは表現できない。

さらにキャッシュのメモリ予算管理が未設計。4K RGBAフレームは1枚約33MB。デコードキャッシュを無邪気に持つとVRAM/RAMは数秒分で枯渇する。

**対策**: キャッシュキーを「ノードID × 時間区間 × 入力パラメータのハッシュ」で設計し、時系列解析ノードは**区間単位のキャッシュエントリ**として扱う。メモリ予算(RAM/VRAM別の上限とLRU追い出し)をキャッシュ層の初期設計に含める。ただしこれらはM2以降の話で、MVPは「キャッシュなし・毎回全計算」で正しさだけを検証する。

#### B-6. OpenCV依存の重さ【リスク: 中】

Rustの`opencv`クレートはビルドが重く(システム依存・クロスプラットフォーム配布が面倒)、MVPで必要な「色トラッキング」(支配色抽出・色距離マスク・重心追跡程度)にはオーバーキル。

**対策**: MVPの色解析は**wgpuコンピュートシェーダ(ヒストグラム・リダクション)で自前実装**する。どうせピクセルはGPU上にあるので、CPUに下ろしてOpenCVに渡すよりデータ移動も少ない。オプティカルフロー等の高度な解析が必要になった段階で初めてOpenCV(またはONNX Runtime等)を「解析プラグイン」として隔離導入する。

### C. データモデルの落とし穴

#### C-1. Undo/Redoの後付け【リスク: 高】

設計資料に登場しないが、編集ツールでUndoを後付けするのは地獄の定番。エンジン内部状態とUI状態が絡み合った後からでは、まともに実装できない。

**対策**: **プロジェクト状態を単一の純データ構造(serde可能なドキュメントモデル)とし、全編集操作をコマンド(差分)として適用する**構造を最初のタイムライン実装時から強制する。エンジンはドキュメントモデルを読むだけの立場に置く。これはプロジェクトファイル保存(C-2)と同じ構造で解決できる。

#### C-2. プロジェクトファイル形式とスキーマ進化【リスク: 中】

未決定事項に挙がっているが、罠は形式選びではなくスキーマ進化。開発中はスキーマが毎週変わるので、初期からバージョンフィールド+マイグレーション関数の枠組みだけ用意しておく。凝ったバイナリ形式は不要、バージョン付きJSON(またはRON)で開始し、性能問題が出たときだけ再考する。

#### C-3. キーフレーム/評価エンジンの過小評価【リスク: 中〜高】

設計資料はレンダリングとトラッキングに集中しているが、モーションツールの操作感の核心は**パラメータ評価エンジン**(キーフレーム補間、イージング、パラメータ間リンク=Tracery的生成ルールの土台)にある。ここが「あとで作る雑務」扱いになっているのが危険。トラッキング→生成のパイプラインとは「トラッキング結果という時系列データがパラメータを駆動する」ことに他ならず、**評価エンジンこそがTracery的機能の本体**。

**対策**: 「時刻t → 全パラメータ値」の評価器を独立クレートとして早期に作る。キーフレーム補間と「他のデータ列(トラッキング結果)を参照するパラメータ」を同一の仕組みで表現できるかをM1で検証する。

#### C-4. 2.5D統一シーングラフとブレンドモードの衝突【リスク: 中】

「2Dレイヤーも3Dメッシュも同じシーングラフ」は美しいが、**2Dコンポジットのブレンドモード(乗算・スクリーン等)は描画順依存**であり、3Dの深度ソートと根本的に相性が悪い。半透明メッシュ同士の交差、Zを持つレイヤーの乗算合成の「正しい」順序は一般には定義できない。AEが3Dレイヤーでも結局レイヤー順を基準に合成するのはこのため。

**対策**: 「シーングラフは統一するが、**合成順序はレイヤー(タイムライン)順を正とし、Zは変形(カメラ投影)にのみ使う**」というAE互換の割り切りをv1仕様として明文化する。メッシュ内部の深度は3Dパス内で解決し、レイヤー間はあくまで順序合成。真の深度合成(depth-aware compositing)はやらない。

### D. 開発プロセスの落とし穴

#### D-1. 「trait早期凍結→並列エージェント開発」の順序が逆【リスク: 高】

未決定事項に「traitインターフェースの早期凍結」とあるが、**検証されていないインターフェースを凍結すると、間違ったインターフェースが凍結される**。A-2と同根。並列化の効果は、正しい境界が引けて初めて出る。

**対策**: ロードマップ上に明示的な**凍結ゲート**を置く: 垂直スライス(M1)を1エージェント(or 少数)で完成させ、実際に動いたインターフェースだけを凍結し、そこから並列化を解禁する。凍結前の並列化は「別クレートで独立検証できるもの」(評価エンジン、ffmpegサイドカーラッパー、UIプロトタイプ等)に限定する。

#### D-2. エージェントが自己検証できないコードベース【リスク: 中】

並列エージェント開発の前提は「エージェントが自分の変更の正誤を自動で確認できる」こと。GPUレンダリング結果はユニットテストしにくく、放置するとエージェントは「コンパイルが通った=完成」で進み、統合時に視覚的バグが山積する。

**対策**: M1完了時点でゴールデンイメージテスト基盤(既知入力→レンダ→参照画像と許容誤差つき比較)をCIに入れる。CIはGPUベンダー間の浮動小数差で不安定になるので、**ソフトウェアラスタライザ(lavapipe / WARP)を参照環境として固定**する。

#### D-3. OpenCut UIの流用が「見た目より高くつく」【リスク: 中】

OpenCutのReactタイムラインはMITで流用可能だが: (1) OpenCut自体がフルリライト中で、今フォークするコードは上流で破棄されていく、(2) タイムラインUIはOpenCutのデータモデル/状態管理と密結合しており、こちらのドキュメントモデルとの間に変換層が必要、(3) upstream追従は事実上不可能(フォークして取り込みきり)。

**対策**: 「コード流用」ではなく**「デザイン・操作仕様の流用+コンポーネント単位のつまみ食い」**と期待値を下げる。タイムラインUIの状態管理はこちらのドキュメントモデル(C-1)に直結させて書く。

#### D-4. スコープと士気の管理【リスク: 中】

動画編集ツールのフル自作は個人+エージェント体制でも数ヶ月〜年単位。「使える瞬間」が遠いと確実に失速する。

**対策**: 各マイルストーンの完了条件を「技術デモ」ではなく**「自分のMV制作の実工程で1回使える」**に寄せる。特にM1の書き出し(連番→ffmpegエンコード)を最優先し、「他ツールに素材として持ち込める1ショットが作れる」状態を最速で達成する。

### E. その他の明記すべき割り切り

- **ターゲットOS**: 未決定のままだと3プラットフォーム分のWebView/ウィンドウ差異を常時払い続ける。**v1は開発主機のOS 1つに固定**し、他OSはCIビルドのみ(動作保証しない)と決める。
- **HDR/10bit、ハードウェアデコード、真の深度合成、複数カメラ、ライティング/PBR、プラグインの動的ロード配布**: すべてv1スコープ外と明文化する(やらないことリスト)。

## Part 2: ロードマップ

方針: **最大リスク(A-1ブリッジ、B-2デコード)を先頭のスパイクで潰し、単一の垂直スライスでインターフェースを検証してから凍結・並列化する。** 各マイルストーンに「退治する落とし穴」を対応付ける。

```
M0 スパイク ──▶ M1 垂直スライス ──▶ [凍結ゲート] ──▶ M2〜M5(並列化解禁)
```

### M1: 垂直スライス「1ショット作れる」(単一開発ライン、凍結前)

内容: 1本の動画を読み込み → GPUコンピュートで色解析(支配色/色マスク重心) → 解析データがパラメータを駆動するプロシージャルオーバーレイ(2D図形)を生成 → 合成 → **連番画像+ffmpegエンコードでmp4書き出し**。UIなし(CLI+設定ファイルでよい)。キャッシュなし。

完了条件:
- 自分の実素材で1ショット(数秒)を書き出し、MV制作の素材として実際に使えること(D-4)
- プレビュー(S1の方式)と書き出しが同一レンダ関数を通ること(B-4)
- カラーバー素材のゴールデンイメージテストがCI(lavapipe)で通ること(B-3, D-2)
- パラメータ評価器(キーフレーム補間+トラッキング列参照)が独立クレートとして動くこと(C-3)

退治する落とし穴: B-3, B-4, C-3, D-2

### M2: ドキュメントモデルとタイムライン(並列レーン1)

内容: ドキュメントモデル(クリップ・トラック・キーフレーム・プラグインインスタンス)、コマンドベース編集+Undo/Redo、プロジェクト保存/読み込み、音声インポートと音声主クロック再生(波形表示はUI側のM3)。

完了条件: 保存→再起動→読み込み→同一レンダ結果。全編集操作がUndo可能。音声同期プレビューでドリフトなし(10分素材で実測)。

### M3: UI統合(並列レーン2)

### M4: キャッシュ層と解析の拡充(並列レーン3)

内容: ノードID×時間区間×パラメータハッシュのキャッシュ、RAM/VRAM予算とLRU、インポート時プロキシ/CFR正規化。必要ならこの段階で初めてOpenCV等を解析プラグインとして導入(オプティカルフロー)。

退治する落とし穴: B-5, B-6

完了条件: 動画テクスチャ平面+glTFメッシュ+トラッキング駆動オーバーレイが1シーンで合成され、書き出せる。

退治する落とし穴: C-4

## Part 3: 現設計からの変更提案まとめ

Part 1の対策のうち、design-memo/discussion-logの記述を明示的に修正・追加するもの:

1. **音声をコア要件に追加**(B-1): 時間表現は有理数タイムスタンプ、再生主クロックは音声。
2. **プラグインは「静的trait先行、動的ロードはv2」**(A-2): discussion-log 5-1の隔離方式の結論は維持しつつ、動的ロード機構の導入時期を凍結ゲート後どころかv2まで遅延。
3. **MVPのデコードはffmpegサイドカー方式**(B-2): ネイティブバインディング/ハードウェアデコードはスコープ外。
4. **MVPの色解析はwgpuコンピュート自前実装、OpenCVはM4まで導入しない**(B-6)。
5. **2.5Dの合成順序はレイヤー順を正とする**(C-4): discussion-log 6の統一シーングラフ方針に合成順序の割り切りを追記。
6. **並列エージェント開発はM1完了後の凍結ゲートから解禁**(D-1): 「traitの早期凍結」を「検証済みtraitの凍結」に読み替える。
7. **v1はターゲットOS 1つに固定**(E)。
