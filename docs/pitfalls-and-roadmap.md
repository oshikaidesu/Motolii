# 現設計の落とし穴分析と実装ロードマップ

作成日: 2026-07-07
前提資料: `design-memo.md` (2026-07-05), `discussion-log-2026-07-06.md`

このドキュメントの目的: **現在の設計(Rust + wgpu コア / Slint UI / 最小コア+全機能プラグイン / ノード依存キャッシュ / 2.5D統一シーングラフ / 並列エージェント開発)をこのまま実装した場合に踏む可能性が高い落とし穴**を洗い出し、それらを潰す順序としてロードマップを定義する。

2026-07-08 改訂: UI基盤を **WebView/Tauri案からSlintへ転換** 済み。旧A-1(WebViewブリッジ)は構造的に解消し、UI統合の主戦場は M3 + `spikes/s1-slint` に移った。

---

## Part 1: 落とし穴カタログ

リスク度は「踏んだ場合の手戻りの大きさ × 踏む確率」の主観評価。

### A. アーキテクチャ境界の落とし穴

#### A-1. WebView UI ↔ Rust ネイティブ描画の橋渡し【リスク: 解消(2026-07-08)】

design-memo段階ではプロジェクト全体の成立可否を握る単一最大リスクだった。**2026-07-08にUI基盤をSlintへ転換し、本項目は構造的に解消した**。Slintの公式wgpu統合(`unstable-wgpu-29`、`Image::try_from(wgpu::Texture)`)により、コアとUIが同一`wgpu::Device`でテクスチャをゼロコピー共有できる。WebView/Tauri経路は採用しない。

**現行の対策(S1/M3)**: `GpuCtx::new_for_ui()`でコンポジタ要件のデバイスを生成し、`WGPUConfiguration::Manual`でSlintに渡す。レンダは専用スレッド、UIは受信テクスチャの表示のみ。詳細は [`specs/M3-ui-integration.md`](specs/M3-ui-integration.md) と [`spikes/s1-slint.md`](spikes/s1-slint.md)。

以下は**廃止したWebView/Tauri検討経路の記録**(後から同判断を繰り返さないため):

| 方式 | 内容 | 罠 |
|---|---|---|
| (a) ネイティブ子ウィンドウ重ね合わせ | WebViewに「穴」を開け、その下/上にwgpuのswapchainを持つ子ウィンドウを重ねる | airspace問題。DOMとプレビューのz順・DPI・追従がOSごとに別実装 |
| (b) CPU読み戻し→IPCでWebViewへ転送 | wgpuでレンダ→CPU map→UIの`<canvas>`に描画 | 1080p60で約500MB/s。GPU→CPUストール。Tauri既定IPCでは論外 |
| (c) プロセス間GPUテクスチャ共有 | DXGI/IOSurface/dmabufでWebViewプロセスと共有 | 3OS×WebView実装の深い専門知識。wgpu外部メモリは不安定 |
| (d) GPUI等でUIごとフルネイティブ化 | WebViewを捨てる | OpenCut React流用不可。GPUIは採用実績・ドキュメントが薄い |

旧S1a/S1b(WebViewブリッジ2方式)は廃止。現S1はSlintスパイク(`spikes/s1-slint/`)に置き換え済み。

#### A-2. 「最小コア+全機能プラグイン」を初日からやる罠【リスク: 高】

プラグインアーキテクチャの古典的な失敗パターン: **プラグインが何を必要とするか分かる前にプラグインABIを設計することになる**。1個目のプラグインを書くたびにABIに穴が見つかり、インターフェース変更→全プラグイン修正のループに入る。並列エージェント開発と組み合わさると、凍結したはずのtraitが凍結できず、全エージェントの作業が相互に壊れ続ける最悪の形で発現する。

さらにRust固有の罠:

- **RustにはABI安定性がない**。dylibプラグインは「ホストと完全に同一のコンパイラバージョン・同一のクレートバージョン」でビルドしない限りUB。`wgpu::Texture`のようなRust型をdylib境界を越えて渡すのは、同一ビルド環境を強制できる場合以外は成立しない。C ABI化(`#[repr(C)]` + ハンドルID方式)か`abi_stable`クレートが必要になり、設計コストが跳ね上がる。
- wgpuのリソースは生成元の`Device`に紐づく。プラグインが別のwgpuバージョンをリンクしていたら共有不能。

**対策**: プラグイン境界は**まず「同一バイナリ内のRust trait」として設計する**(静的リンク、Cargo featureで有効/無効)。動的ロード(dylib/C ABI)は垂直スライスが動いてtraitが3〜4個の実プラグインで検証された後に導入する。「プラグインベースの設計」と「動的ロード機構」は別物であり、前者は初日から、後者は後回しでよい。WASMサンドボックス(パラメータ系)も同様に、まずはネイティブ関数として書き、境界だけWASM化可能な形(構造化データのみ授受)を守る。

#### A-3. wgpu自体のAPI変動【リスク: 中】

discussion-logで認識済みだが具体策がない。wgpuは数ヶ月ごとに破壊的変更が入る。

**対策**: バージョンをworkspaceで一元ピン留めし、アップグレードは専用PRでまとめて行う。wgpuを抽象化する自前レイヤーを作るのは過剰設計なのでやらない(プラグインに見せる型だけフレーム記述子+ハンドルIDに限定しておけば、内部のwgpu更新は閉じ込められる)。

**改訂(2026-07-08、レビュー指摘#1)**: UI基盤にSlintを採用したため、**wgpuの更新ペースは自分では決められない**。Slintのwgpu統合(`unstable-wgpu-XX`)が対応するメジャーに全クレートを固定する(現在29)。wgpu型はクレートバージョンが違うと別型になり、コアとUIのバージョンが割れるとゼロコピー埋め込みそのものが型エラーで成立しなくなるため、**バージョン分裂は禁止**(workspace依存で一元管理し、スパイク等の別workspaceも同メジャーを使う)。

### B. 映像処理そのものの泥沼

#### B-1. 音声が設計に一切存在しない【リスク: 高・設計漏れ】

**MV制作ツールなのに、design-memo/discussion-logのどこにも音声の記述がない。** 音楽に合わせて編集する用途では、音声波形表示・再生同期・(できればビート検出)は「あると嬉しい」ではなくコア要件のはず。音声を後付けすると:

- タイムラインの時間表現がフレーム番号ベースで固まった後に、サンプルレートベースの音声クロックと整合しなくなる(A/V同期のドリフト)
- プレビュー再生ループの主クロックを「映像フレーム」で作ってしまうと、音声途切れ・同期ズレの温床になる。**再生クロックの主は音声**(オーディオコールバック駆動)が定石

**対策**: 実装は後回しでよいが、**タイムラインの時間表現と再生アーキテクチャの設計には初日から音声を含める**。時間は浮動小数の秒ではなく有理数タイムスタンプ(FFmpegの`time_base`方式: 分子/分母のi64)で持つ。

#### B-2. 動画デコードの想定甘さ【リスク: 高】

「ネイティブffmpegを直接叩ける」の一行で済まされているが、ここには複数の罠がある:

- **Rustバインディングの保守状態**: `ffmpeg-next`等のバインディングはビルドが重く、FFmpegバージョン追従も不安定。加えてFFmpegを静的リンクするとLGPL/GPL・特許(H.264/HEVC)の配布問題を背負う。
- **シークの罠**: コーデックのシークはキーフレーム単位。フレーム正確なシークには「直前キーフレームへシーク→目的フレームまでデコード読み捨て」が必要で、スクラブ性能に直結する。
- **VFR(可変フレームレート)素材**: スマホ撮影動画はほぼVFR。フレーム番号ベースのタイムラインはVFR素材で音ズレ・フレームズレを起こす。
- **ハードウェアデコード**: NVDEC/VideoToolbox/D3D11VAとwgpuのゼロコピー連携は各OS固有の深い沼。最初から狙うと数ヶ月溶ける。

**対策**: MVPは**ffmpegをサイドカープロセスとして起動し、rawvideoをパイプで受け取る**方式にする(リンク不要、ライセンス問題回避、クラッシュ隔離、決定的)。スクラブ用にはプロキシ(低解像度中間ファイル)生成 or 全Iフレーム中間コーデックへの変換をインポート時に行う設計を最初から入れる。VFRはインポート時にCFR正規化。ハードウェアデコードはv2以降の最適化と明記してスコープから外す。

#### B-3. 色空間・YUVの事故【リスク: 中〜高】

フレーム記述子に色空間フィールドを持たせる方針(5-2)は正しいが、フィールドを「持っている」ことと「正しく変換している」ことは別:

- デコード出力はYUV(通常limited range / BT.709)。wgpuにはネイティブYUVサンプラがないため、**YUV→RGB変換シェーダを自前で書く**ことになる。limited/full rangeの取り違え・601/709係数の取り違えは「なんとなく白っぽい/色が薄い」という気づきにくいバグとして出る。
- sRGBテクスチャフォーマット(`Rgba8UnormSrgb`)とリニア合成の混在ミス。ブレンド・リサイズをsRGB空間でやると暗部が濁る。
- premultiplied alphaの取り違えは半透明の縁に黒フリンジとして出る。

**対策**: 既知のカラーパターン動画(カラーバー等)を使った**ゴールデンイメージテストをM1の時点で導入**する。「見た目でなんとなく合ってる」を禁止し、ピクセル値の数値比較で守る。10bit/HDRはv1スコープ外と明記。

#### B-4. プレビューと書き出しの分離が「分岐」になる罠【リスク: 中】

Remotion方式の分離は正しいが、素朴に実装すると「プレビュー用レンダパス」と「書き出し用レンダパス」が別コードになり、**書き出したら見た目が違う**という動画編集ツール最悪のバグを生む。

**対策**: レンダラは「時刻tと品質パラメータを受け取り決定的にフレームを返す」単一の純関数として実装し、プレビューと書き出しは**同じ関数の呼び出し方(解像度・スキップ・キャッシュ方針)だけが違う**構造にする。書き出しはフレーム列をffmpegサイドカーにパイプしてエンコード。

#### B-5. ノード単位キャッシュと時系列依存の衝突【リスク: 中】

「変更部分だけ再計算」のダーティフラグ設計は、**フレーム内で閉じる処理**には有効だが、本題のトラッキング解析は時系列依存(オプティカルフローはフレームN-1に依存、色トラッキングは全区間の解析)を持つ。パラメータを1つ変えた時に「どこからどこまで再解析が必要か」はフレーム単位のダーティフラグでは表現できない。

さらにキャッシュのメモリ予算管理が未設計。4K RGBAフレームは1枚約33MB。デコードキャッシュを無邪気に持つとVRAM/RAMは数秒分で枯渇する。

**対策**: キャッシュキーを「ノードID × 時間区間 × 入力パラメータのハッシュ」で設計し、時系列解析ノードは**区間単位のキャッシュエントリ**として扱う。メモリ予算(RAM/VRAM別の上限とLRU追い出し)をキャッシュ層の初期設計に含める。ただしこれらはM2以降の話で、MVPは「キャッシュなし・毎回全計算」で正しさだけを検証する。

#### B-6. OpenCV依存の重さ【リスク: 中】

Rustの`opencv`クレートはビルドが重く(システム依存・クロスプラットフォーム配布が面倒)、MVPで必要な「色トラッキング」(支配色抽出・色距離マスク・重心追跡程度)にはオーバーキル。

**対策**: MVPの色解析は**wgpuコンピュートシェーダ(ヒストグラム・リダクション)で自前実装**する。どうせピクセルはGPU上にあるので、CPUに下ろしてOpenCVに渡すよりデータ移動も少ない。オプティカルフロー等の高度な解析が必要になった段階で初めてOpenCV(またはONNX Runtime等)を「解析プラグイン」として隔離導入する。

### C. データモデルの落とし穴

#### C-1. Undo/Redoの後付け【リスク: 高】

設計資料に登場しないが、編集ツールでUndoを後付けするのは地獄の定番。エンジン内部状態とUI状態が絡み合った後からでは、まともに実装できない。

**対策**: **プロジェクト状態を単一の純データ構造(serde可能なドキュメントモデル)とし、全編集操作をコマンド(差分)として適用する**構造を最初のタイムライン実装時から強制する。エンジンはドキュメントモデルを読むだけの立場に置く。これはプロジェクトファイル保存(C-2)と同じ構造で解決できる。

#### C-2. プロジェクトファイル形式とスキーマ進化【リスク: 中】

未決定事項に挙がっているが、罠は形式選びではなくスキーマ進化。開発中はスキーマが毎週変わるので、初期からバージョンフィールド+マイグレーション関数の枠組みだけ用意しておく。凝ったバイナリ形式は不要、バージョン付きJSON(またはRON)で開始し、性能問題が出たときだけ再考する。

#### C-3. キーフレーム/評価エンジンの過小評価【リスク: 中〜高】

設計資料はレンダリングとトラッキングに集中しているが、モーションツールの操作感の核心は**パラメータ評価エンジン**(キーフレーム補間、イージング、パラメータ間リンク=Tracery的生成ルールの土台)にある。ここが「あとで作る雑務」扱いになっているのが危険。トラッキング→生成のパイプラインとは「トラッキング結果という時系列データがパラメータを駆動する」ことに他ならず、**評価エンジンこそがTracery的機能の本体**。

**対策**: 「時刻t → 全パラメータ値」の評価器を独立クレートとして早期に作る。キーフレーム補間と「他のデータ列(トラッキング結果)を参照するパラメータ」を同一の仕組みで表現できるかをM1で検証する。

#### C-4. 2.5D統一シーングラフとブレンドモードの衝突【リスク: 中】

「2Dレイヤーも3Dメッシュも同じシーングラフ」は美しいが、**2Dコンポジットのブレンドモード(乗算・スクリーン等)は描画順依存**であり、3Dの深度ソートと根本的に相性が悪い。半透明メッシュ同士の交差、Zを持つレイヤーの乗算合成の「正しい」順序は一般には定義できない。AEが3Dレイヤーでも結局レイヤー順を基準に合成するのはこのため。

**対策**: 「シーングラフは統一するが、**合成順序はレイヤー(タイムライン)順を正とし、Zは変形(カメラ投影)にのみ使う**」というAE互換の割り切りをv1仕様として明文化する。メッシュ内部の深度は3Dパス内で解決し、レイヤー間はあくまで順序合成。真の深度合成(depth-aware compositing)はやらない。

### D. 開発プロセスの落とし穴

#### D-1. 「trait早期凍結→並列エージェント開発」の順序が逆【リスク: 高】

未決定事項に「traitインターフェースの早期凍結」とあるが、**検証されていないインターフェースを凍結すると、間違ったインターフェースが凍結される**。A-2と同根。並列化の効果は、正しい境界が引けて初めて出る。

**対策**: ロードマップ上に明示的な**凍結ゲート**を置く: 垂直スライス(M1)を1エージェント(or 少数)で完成させ、実際に動いたインターフェースだけを凍結し、そこから並列化を解禁する。凍結前の並列化は「別クレートで独立検証できるもの」(評価エンジン、ffmpegサイドカーラッパー、UIプロトタイプ等)に限定する。

#### D-2. エージェントが自己検証できないコードベース【リスク: 中】

並列エージェント開発の前提は「エージェントが自分の変更の正誤を自動で確認できる」こと。GPUレンダリング結果はユニットテストしにくく、放置するとエージェントは「コンパイルが通った=完成」で進み、統合時に視覚的バグが山積する。

**対策**: M1完了時点でゴールデンイメージテスト基盤(既知入力→レンダ→参照画像と許容誤差つき比較)をCIに入れる。CIはGPUベンダー間の浮動小数差で不安定になるので、**ソフトウェアラスタライザ(lavapipe / WARP)を参照環境として固定**する。

#### D-3. OpenCut UIの流用が「見た目より高くつく」【リスク: 中】(2026-07-08 改訂: コード流用は不可)

Slint採用によりOpenCutのReactタイムラインを**コード流用できない**(UI基盤が異なる)。参考にできるのは操作仕様・レイアウトのみ。加えて: (1) OpenCut自体がフルリライト中で上流コードは破棄されていく、(2) タイムラインUIはOpenCutのデータモデル/状態管理と密結合、(3) upstream追従は事実上不可能。

**対策**: 「コード流用」ではなく**「デザイン・操作仕様の参考」**に格下げする。タイムラインUIはSlintで自前実装し、状態管理はこちらのドキュメントモデル(C-1)に直結させる。

#### D-4. スコープと士気の管理【リスク: 中】

動画編集ツールのフル自作は個人+エージェント体制でも数ヶ月〜年単位。「使える瞬間」が遠いと確実に失速する。

**対策**: 各マイルストーンの完了条件を「技術デモ」ではなく**「自分のMV制作の実工程で1回使える」**に寄せる。特にM1の書き出し(連番→ffmpegエンコード)を最優先し、「他ツールに素材として持ち込める1ショットが作れる」状態を最速で達成する。

### E. その他の明記すべき割り切り

- **ターゲットOS**: 未決定のままだと3プラットフォーム分のウィンドウ/レンダラ差異を常時払い続ける。**v1は開発主機のOS 1つに固定**し、他OSはCIビルドのみ(動作保証しない)と決める。
- **HDR/10bit、ハードウェアデコード、真の深度合成、複数カメラ、ライティング/PBR、プラグインの動的ロード配布**: すべてv1スコープ外と明文化する(やらないことリスト)。

### F. 基礎アーキ層の負債(2026-07-08 追加: 先行プロジェクト死因分析より)

コードレベルの指摘が収束した後の第4ラウンドとして、同思想の先行プロジェクトの「死因」を調べ、それに照らして**まだこのドキュメント群に無かった基礎的負債**を挙げる。

先行プロジェクトの死因(二の舞リスト):

| プロジェクト | 死因/大手戻りの原因 |
|---|---|
| Natron(開発停止) | ①2人プロジェクトの人的バス係数(作者談「2人は10人にならず大抵死ぬ」)②長尺動画のメモリ問題 ③マルチスレッドのrace/deadlock ④キャッシュ(trimap)のデッドロック→作り直し未完。**並行性とキャッシュが殺した** |
| Olive(0.1→0.2フルリライト) | 技術的負債の蓄積 + OCIO(色管理)・OTIO(タイムライン交換)を最初から入れ直すため。作者談「アーキが固まるまで性能はずっと最悪だった」 |
| OpenCut(スクラッチ再構築) | Web専用エンジンを「エンジン/UI分離 + 不変ステートツリー + plugin-first」へ。**境界を後から入れられず全書き直し** |
| DaVinci Resolve / Blender compositor | 位置が絶対ピクセル依存で解像度非依存が崩壊(Resolveは位置座標が±canvas依存で頭打ち、Blenderは解像度%変更でpixel単位ノードが破綻し「relative space」改修に難航中) |

#### F-1. 解像度非依存の座標系・アンカー・変形階層が未定義【リスク: 最高】

モーショングラフィックツールなのに、**「合成の正準座標空間」が定義されていない**。位置/サイズの単位(px? 正規化?)、原点(左上? 中央?)、Y軸の向き(2D慣習は下向き、glTFはY-up — 2.5D統一シーングラフで両者が同居するのに未調整)、アンカーポイント、親子変形(ペアレンティング/ヌル)、グループが変形コンテナか単なる合成コンテナか — すべて未決。モーショングラフィックにペアレンティングは必須で、後付けは全クリップのスキーマ変更になる。

**致命的な自己矛盾**: 「Draft=半解像度でも `render_frame(t, Quality)` は同一だから見た目分岐なし」(B-4/performance-model)という約束は、**pxベースの空間パラメータ(ブラー"N px"等)が解像度相対でないと崩れる**。ブラー半径を絶対pxで持つとDraft(1/2解像度)とFinalで効き方が2倍変わる=約束破綻。BlenderとResolveがまさにこれで苦しんでいる。

**対策(凍結ゲート必須項目に昇格)**:
- **正準合成空間**を定義する: 単位なし・原点=コンポ中央・**Y-up**(glTF/3Dと軸一致)・高さ基準正規化(コンポの高さ=1.0、幅=アスペクト比)。OTIOの座標系提案(単位なし・単一原点・Y-up)と同方向
- **全空間パラメータ(位置・アンカー・ブラー半径・マスク位置・図形サイズ)は正準空間の値で持つ**。ピクセルへの変換はレンダ直前(解像度確定後)の1箇所のみ。「pxでパラメータを持つノード」をレビューで禁止する
- **グループは変形コンテナとする**(合成コンテナ兼変形階層。ヌル的な使い方をカバー)。レイヤー単位のペアレンティング(親参照)はドキュメントモデルのスキーマにフィールドを予約する
- glTFとの軸整合(2D平面をY-up世界にどう置くか)をM5仕様に明文化する

#### F-2. アプリ全体の並行性・所有権モデルが未定義【リスク: 最高】(Natronの直接の死因)

プレビューのレンダスレッド規約(M3)はあるが、**アプリ全体で「誰がドキュメントを所有し、誰がいつ書き換えるか」の契約が無い**。UI編集・再生・バックグラウンドのプロキシ生成/解析/インポート(M4)・書き出しが同時にドキュメント/キャッシュへ触る。Natronはまさにここ(race/deadlock、キャッシュのデッドロック)で死んだ。M2/M3/M4を並列レーンで走らせる前にこれを凍結しないと、「別々に作った後、統合でデータ競合」=Natronの再来になる。

**対策(凍結ゲート必須項目に昇格)**: **単一writer + 不変スナップショット**モデルを採用する(OpenCutが再構築で選んだ"immutable state tree"と同構造):
- ドキュメントの書き換えは**編集スレッド(コマンド適用)ただ1箇所**。他は全員読み手
- レンダ・書き出し・解析・プロキシ生成は**不変スナップショット(`Arc<Document>`等)**を受け取って動く。実行中にドキュメントが変わっても自分のスナップショットは不変(ロック不要、race定義上なし)
- バックグラウンドジョブの成果(解析結果・プロキシ)は共有状態への直接書き込みではなく、**メッセージとしてwriterに返し、writerが適用**する
- キャッシュの並行契約(読み手が読んでいる間に誰が追い出すか — Natronがデッドロックした正確なポイント)はM4仕様に明文化する: 読み手はハンドル(参照カウント)を握り、追い出しは「参照ゼロになってから解放」の遅延方式。ロック保持中に他のロックを取らない(ロック階層を1段に保つ)

#### F-3. 効果スタック+クリップマスク+グループ+2.5D+合成を束ねる「単一評価モデル」が未統合【リスク: 高】(Oliveの「アーキが固まらない」)

エフェクトスタック(concept)、クリッピングマスク(M2-D7)、グループ再帰(M2-D1)、2.5Dカメラ投影(M5/C-4)、レイヤー順合成(C-4)は個別に決めたが、**「3Dレイヤーが、マスク付きグループの中で、エフェクトスタックを持ち、2Dレイヤーと合成される」時の単一の決定的評価順序**が仕様化されていない。概念が5つ並存したまま並列実装に入ると、M2-D3(doc→グラフ変換)が矛盾の噴出点になり、Oliveの「アーキが固まるまで性能最悪」の期間を再演する。

**対策(凍結ゲート必須項目に昇格)**: 評価順序を1本に固定して仕様書に明文化する。v1の評価モデル(案。凍結ゲートでM1実証を経て確定):

```
フレーム評価(スナップショット, t, Quality):
  トラック項目を再帰的に評価(レイヤー順=タイムライン順):
    1. ソース取得: クリップの時間写像でローカル時刻を解決(F-4)→ フレーム/図形/メッシュ
    2. エフェクトスタックを線形チェーンで適用(正準空間パラメータ、F-1)
    3. 変形: アンカー→スケール/回転→位置、親(グループ/親参照)の変形を合成、
       3Dレイヤーはここでカメラ投影(Zは投影のみ、C-4)
    4. クリッピングマスク: 直下レイヤーの結果でクリップ(モード適用)
    5. グループ: 内部をレイヤー順で合成した1枚を「1つのレイヤー」として上位へ
  最上位をレイヤー順+ブレンドモードで合成 → ポストプロセスノード → 出力
```

#### F-4. タイムリマップ/速度変更が時間モデルに無い【リスク: 中〜高】(MV必須・後付け激痛)

RationalTime(B-1)は綺麗だが、**クリップのローカル時刻↔タイムライン時刻の写像**(リタイミング・速度ランプ・逆再生)がモデル化されていない。MVは速度ランプを多用する。後から入れると全クリップのソース参照方法が変わる(AEはCollapse Transformationsとtime-remapが壊れ合う前例)。

**対策(凍結ゲート必須項目に昇格)**: **「クリップは時間写像(TimeMap)を持つ」をスキーマに初日から予約**する。v1実装は恒等写像+定数速度(オフセット+speed)のみでよいが、**ソースへの時刻アクセスは必ずTimeMapを通す**構造にする(単調なキーフレーム写像=速度ランプ・逆再生への拡張はスキーマ互換のまま可能)。時間写像は解析(DataTrack)・キャッシュキー(時間区間)にも波及するため、予約だけは今やる。

#### F-5. 交換標準(OTIO/OCIO)への姿勢【リスク: 中】(Oliveのリライト理由そのもの)

- **色(OCIO)**: v1はsRGB近似・OCIOはv2、は単独MVツールとして妥当。ただしOliveは「OCIOを入れ直すため」に全書き直しした。「リニア化・変換をoc-gpuの共通関数1箇所に閉じ、各エフェクトで勝手に実装しない」ガード(performance-model)が既にあり、**これがOCIO-shapedを保つ生命線**。この規律を絶対に崩さない(レビュー最重視項目に追加)
- **タイムライン(OTIO)**: 独自JSONは維持しつつ、**スキーマの素性をOTIO互換寄り**にする(トラック/クリップ/ソース区間を有理数時刻で持つ、階層は名前付き構造 — 現行方針とほぼ一致しており追加コストは小さい)。OTIO書き出しはv2候補。今すぐ実装は不要、「OTIOに写像できない構造を発明しない」ことだけ守る

#### F-6. フォント/テキスト基盤が丸ごとプラグイン任せ【リスク: 中】(歌詞=主用途)

歌詞組版をファーストパーティプラグイン第1号にする方針(concept)は維持するが、**フォント読み込み・フォールバック・日本語シェーピング・グリフ→パス変換は共有基盤**であり、コアに無いと第1号プラグインがテキストスタックまるごと自作になる(Velloのテキストは日本語シェーピングが非自明)。

**対策**: **「シェーピングまではコア、組版はプラグイン」**の分界を決定する。コアに最小のテキスト基盤クレート(フォント列挙/読み込み=fontdb系、フォールバック解決、シェーピング=rustybuzz、グリフ→ベクターパス出力=Velloへ接続)を置く。縦書き・ルビ・行組・歌詞タイミングはプラグイン側。導入時期はM5前後でよいが、**分界の決定自体は今**(プラグインAPIにテキストの口を後から開けると境界変更になるため)。

#### F-メタ. Natronの最大の教訓と凍結ゲートの関係

「2人プロジェクトは10人にならず大抵死ぬ。最初に人を集めろ」(Natron作者)。このプロジェクトの「並列AIエージェント開発」はこの人的バス係数への回答になり得るが、エージェントが暴れないための"チーム"の実体は**仕様書+凍結ゲート+ゴールデンテスト**である。F-1〜F-4を凍結ゲートに足して先に固めることが、Olive/Natronが溶かした「アーキが固まらない期間」を最短化する唯一の道。
