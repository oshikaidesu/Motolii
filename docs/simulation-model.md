# 時間軸の自由度モデル — 物理シミュレーションと前後フレーム参照の設計

作成日: 2026-07-10
ステータス: **設計方針確定・口の予約段階**(凍結ゲートでスキーマ/traitの口を予約、実装はv1.x以降)

この文書は「液体・布・パーティクル等の**本物の物理シミュレーション**を、キャッシュ・フレーム並列・決定論というプロジェクトの土台を壊さずに一級市民として持てるか」への回答。結論: **持てる**。ただし自由の置き場所を設計する必要があり、その設計がこの文書。

関連: [concept.md](concept.md)(決定台帳)、[performance-model.md](performance-model.md)§6(フレーム並列)、[specs/M4-cache-and-analysis.md](specs/M4-cache-and-analysis.md)(区間キャッシュ)、[specs/M5-3d-and-post.md](specs/M5-3d-and-post.md)(2026-07-07決定の改訂)、落とし穴F-10。

## 0. なぜこの文書があるか(動機と危機感)

- **物理シミュレーションはMV表現の本命の一つ**。布・液体・パーティクルを「Blenderでベイクして持ち込め」だけで済ませるのは、コミュニティ先導開発というコンセプトと衝突する — 書きたいものが書けない境界は、コミュニティを他ツールへ流出させる
- 一方で、先行プロジェクトの死因分析(pitfalls F群/G群)が示す通り、**無契約の自由も同じくらい確実にプロジェクトを殺す**。AviUtl/AE圏でプラグインがグローバル変数に前フレーム状態を隠し持つ「シミュレーションハック」は実在する定番だが、あれは**ホストにシミュレーションの契約が無いから作者がハックせざるを得ない**のであって、自由の証明ではない。隠れ状態プラグインはシーケンシャル再生ではたまたま動き、キャッシュON・スクラブ・並列書き出しで**サイレントに壊れる**(ユーザーから見ると「ランダムに壊れるエフェクト」)
- したがってこの文書の設計目標は二重: **(a) 表現の自由を最大化する。(b) その自由を、壊れる形(隠れ状態ハック)ではなく壊れない形(ホスト管理のベイク境界)で提供する**

## 1. 核心の分解: 「決定論」と「フレーム独立性」は別物

これまでB-4/純関数契約は2つの性質を一括りにしていた。分解する:

| 性質 | 内容 | 依存するもの | 扱い |
|---|---|---|---|
| **決定論** | 同じDocument(+同じ素材)→ 同じ出力 | プレビュー=書き出し一致(B-4)、ゴールデンテスト、ジャーナル再生、並列エージェント開発の自動検証 | **絶対に守る。不変** |
| **フレーム独立性** | 任意の時刻tを単独で計算できる | フレーム並列、点キャッシュ、自由なスクラブ | **性能特性。区間キャッシュで代替可能** |

**固定シード+固定タイムステップの逐次シミュレーションは完全に決定論的**である(同じ入力から必ず同じ状態列が得られる)。失われるのはフレーム独立性=ランダムアクセス性だけで、それは「一度走らせた結果を区間キャッシュに焼く(ベイク)」ことで取り戻せる。このプロジェクトには既にその枠がある:

- 解析DataTrackの**区間キャッシュ**(B-5対策、M4-K1: ノードID×時間区間×パラメータハッシュ)
- **グループ仮出力(ベイク)**(K7: プリコンポ代替)
- `ParamDriver::build_track` — **すでに逐次計算が合法な場所**(区間を一括生成する契約なので、内部でフレーム順に積分してよい。外から見れば決定論的な純関数)

つまり「シミュレーションを持てない」のではなく、「シミュレーションの結果を**レンダ経路の外で**確定させる場所が未定義」だっただけ。この文書はその場所を定義する。

## 2. 時間軸自由度の5段階(プラグイン作者に提供するはしご)

作者(人間/LLM)には、欲しい表現に対して**最も安いレベルを選ぶはしご**を提供する。上のレベルほど自由で、ホストのコスト(メモリ・無効化の粒度)が上がる。

| レベル | 名前 | できること | API | 状態 |
|---|---|---|---|---|
| **0** | tの閉形式純関数 | バネ/バウンスイージング(減衰振動の閉形式)、シード+tで決まる手続きパーティクル、ウィグル | 現行の全render系trait | **今すぐ可能** |
| **1** | 区間一括生成(内部逐次) | 数値状態のシミュレーション: バネ質点の軌道、群れ(boids)の重心、簡易物理の位置列 | `ParamDriver::build_track`(内部でフレーム順積分OK) | **今すぐ可能**(本文書で明文化) |
| **2** | 宣言的時間窓 | エコー/残像、フレームブレンド、タイムディスプレイス、スーパーサンプリング型モーションブラー | `NodeDesc`の時間フットプリント宣言(§6) | **口を予約**(凍結ゲート) |
| **3** | 逐次状態シミュレーション | **布・液体・本物のパーティクル**(前フレーム状態に依存する積分) | `SimulationPlugin` + StateTrack(§3〜§5) | **口を予約**(凍結ゲート)、実装v1.x |
| — | 禁止のまま | 未宣言の隠れ状態(`&self`に状態を溜める)、レンダ経路での自己出力フィードバック、壁時計・スレッドID等の非決定入力 | — | **恒久禁止**(§7) |

レベル0/1で表現できるものをレベル3で書かないこと(レビュー観点)。逆に、レベル3が必要な表現をレベル0/1に無理に押し込んでハックしないこと。

## 3. レベル3の設計: SimulationPlugin + StateTrack

### 3.1 アーキテクチャ: 2相評価

```
[シミュレーション相]  逐次・ホスト所有・バックグラウンド
  SimulationPlugin::init(seed, params@t0) → state₀
  step(state₀, dt) → state₁ → step → state₂ → …   (固定dt、sim_rate)
  N フレームごとに snapshot → StateTrack(チェックポイント列)としてキャッシュ層へ

[レンダ相]  純関数のまま・フレーム並列のまま
  render_frame(t, Quality) は StateTrack(t) を「読むだけ」
```

**`render_frame(t, Quality)`は純関数のまま変わらない。** 入力に「ベイク済みStateTrack」が加わるだけで、StateTrack自体がDocumentの決定論的関数なので、合成関数全体は依然としてDocumentの純関数である。絶対規律3(純関数)・6(プレビュー/書き出し同一関数)は形を変えずに守られる。フレーム並列も守られる: 逐次なのはシミュレーション相だけで、レンダ相は任意フレームを並列に計算できる。

### 3.2 trait契約スケッチ(凍結ゲートで確定させる叩き台)

```rust
/// 逐次状態シミュレーション。状態はホストが所有・保存・無効化する。
/// プラグインは「1ステップ進める方法」と「状態の凍結/解凍」だけを提供する。
pub trait SimulationPlugin: Send + Sync {
    fn desc(&self) -> &NodeDesc; // PluginKind::Simulation

    /// シミュレーション時間解像度(固定dt)。プレビューfpsから独立。決定論の要
    fn sim_rate(&self) -> Fps;

    /// 状態スナップショット1個の概算サイズ(バイト)。ホストの予算管理・
    /// チェックポイント間隔の決定に使う
    fn state_budget_bytes(&self) -> u64;

    /// t=0の状態を作る。乱数はseed(Document由来)のみから導出する
    fn init(
        &self, gpu: &GpuCtx, params: &ResolvedParams, seed: u64,
    ) -> Result<SimState, PluginError>;

    /// 状態を1ステップ進める。dtは常に 1/sim_rate(可変dt禁止)。
    /// paramsはそのsim時刻で評価済みの値(キーフレーム/DataTrack駆動可能)
    fn step(
        &self, gpu: &GpuCtx, encoder: &mut wgpu::CommandEncoder,
        state: &mut SimState, sim_t: RationalTime, params: &ResolvedParams,
    ) -> Result<(), PluginError>;

    /// チェックポイント用に状態を凍結/解凍する(GPUバッファ→バイト列)。
    /// フォーマットはプラグイン私有(opaque)+version番号
    fn snapshot(&self, gpu: &GpuCtx, state: &SimState) -> Result<StateSnapshot, PluginError>;
    fn restore(&self, gpu: &GpuCtx, snap: &StateSnapshot) -> Result<SimState, PluginError>;
}
```

- **状態の実体はGPUバッファでよい**(`SimState`はwgpuバッファ束のハンドル)。布のバネ質点系・パーティクル・安定化流体ソルバはwgpu computeで実装できる — **VRAM常駐原則(絶対規律1)ともWGSLのみ規約(F-9)とも矛盾しない**。シミュレーションはGPUが苦手なのではなく、「前フレーム依存だからフレーム並列できない」だけ。1ステップ内部の粒子並列・グリッド並列はむしろGPU向き
- **描画への出口は新設しない**: シミュレーション状態を絵にする段は、既存の`LayerSourcePlugin`境界を再利用する(StateTrackの該当サンプルを入力に受けてラスタライズ)。「状態を作る(Simulation)」と「状態を描く(LayerSource)」を分離することで、同じパーティクル状態を複数の描き方(点・軌跡・メッシュ)で使い回せる
- **ホストが実行を所有する**: プラグインは`step`を提供するだけで、いつ・どこからどこまで走らせるか・どこにキャッシュするか・いつ無効化するかは全部ホスト(スケジューラ)の責務。これが**AviUtl式ハックとの決定的な差**であり、作者体験でも上位互換(作者はキャッシュ・スクラブ・並列の面倒を一切見なくてよい。LLM作者にとっても`init`/`step`だけ書けばよい方が易しい)

### 3.3 StateTrack: DataTrackの大容量版=ベイクの一般化

```
StateTrack = {
  node_id, params_hash, version,
  sim_rate: Fps,
  checkpoints: [(sim_t, StateSnapshot)],   // Nフレームごと
}
```

- **キャッシュキーはM4-K1と同一の枠**: ノードID×時間区間×パラメータハッシュ。区間キャッシュ機構(解析DataTrack用に予約済み)と**グループ仮出力(K7)の間に自然に収まる第3のクライアント**であり、キャッシュ層に新しい概念を要求しない
- **Documentには入れない**: StateTrackはピクセルキャッシュ・解析DataTrackと同じく**再生成可能なキャッシュ/サイドカー**(concept 2026-07-09「文書とキャッシュの分離」に従う)。Documentに入るのはシミュレーションノードのレシピ(プラグインID・パラメータ・seed)だけ
- **メモリ階層に乗る**: スナップショットはVRAM→RAM→ディスク(memory-model P1〜P4)へ退避可能。ディスクキャッシュはK7で既にv1採用済み

### 3.4 無効化とスクラブ(チェックポイントの意味)

- **無効化**: 時刻Tに影響するパラメータ変更 → **Tより前の最後の有効チェックポイントから先だけ**を破棄・再シミュレーション。曲の後半をいじっても前半のシムは生き残る。「1パラメータ変更→影響範囲のみ再計算」(K2)の時間軸版
- **スクラブ**: 任意tへのアクセス = 直近チェックポイントから最大「チェックポイント間隔」分だけリプレイ。間隔がリプレイコストの上限を与える(間隔1秒なら最悪30step)。UXはグループ仮出力と同一の言葉で説明できる: 「シミュレーションはベイクすると滑らか、編集すると該当範囲から自動再ベイク」
- **逆再生・タイムリマップ(F-4)との相互作用**: シミュレーションは**クリップローカル時間**で走り、TimeMapは「ベイク済みトラックの読み出し写像」として掛かる。ライブシミュレーションでは逆再生が原理的に定義不能だが、**ベイク方式なら逆再生・速度ランプが自明に正しく動く** — ベイク境界の追加の利点

### 3.5 実行モデルと所有権(F-2整合)

シミュレーションジョブは**プロキシ生成・解析ジョブと同じバックグラウンドジョブ**として走る(M2-D8/M4の所有権モデルそのまま):

- スナップショット(`Arc<Document>`)読み+結果メッセージ返し。ドキュメント/キャッシュへ直接書かない
- **書き出し時**: シミュレーション相はレンダ相より**先行してパイプライン化**する。シム1ステップのコストは通常レンダ1フレームより桁で小さい(§5の試算)ので、フレーム並列レンダの供給を止めない。すでにベイク済みなら書き出しはStateTrackを読むだけ
- プレビュー時: 未ベイク区間に再生ヘッドが入ったら、Draft品質同様「追いつくまで粗く/止まる」でよい(3D区間ベイクと同じ体験設計。M5参照)

### 3.6 決定論の但し書き(正直に書く)

- **同一マシン・同一ドライバでは完全再現**する(固定seed・固定dt・壁時計禁止なので)
- **GPUベンダ/ドライバ間ではfloat演算がbit一致しない**。通常のエフェクトなら許容誤差内に収まるが、**カオス系(流体・多体衝突)は誤差を指数的に増幅する**ため、クロスGPUでの見た目一致は「保証しない」と明文化する(AE/Houdini/Blenderも同様の立場)。ゴールデンテストはlavapipe固定(INF-3と同一方針)。「同じプロジェクトを別マシンで開いたら流体の飛沫が違う」は仕様であり、確定した絵が必要ならベイク結果(StateTrack/仮出力)を持ち運ぶ
- `step`内の禁止事項はrender系と同じ: 壁時計・スレッドID・未seed乱数・順序非決定なatomic reduction(浮動小数の加算順が変わる並列reductionは決定論を壊す。順序固定のreductionを使う)

## 4. なぜ「Filterに状態を隠すハック」を禁止し続けるか

ユーザー(コミュニティ)に最大の自由を渡したいからこそ、**壊れる自由ではなく壊れない自由**を渡す:

| | 隠れ状態ハック(AviUtl/AE圏の定番) | SimulationPlugin + StateTrack |
|---|---|---|
| シーケンシャル再生 | 動く(ように見える) | 動く |
| スクラブ/シーク | 壊れる(状態が飛ぶ) | 動く(チェックポイントからリプレイ) |
| キャッシュON | 壊れる(キャッシュは状態を知らない) | 動く(StateTrack自体がキャッシュ) |
| 並列書き出し | 壊れる(評価順が非決定) | 動く(レンダ相は純関数のまま) |
| 逆再生/タイムリマップ | 壊れる | 動く(ベイク済みトラックの読み出し) |
| Undo/ジャーナル再生 | 再現しない | 再現する(seed+レシピから再生成) |
| 作者の負担 | キャッシュ・シークの面倒を作者が(見えないまま)背負う | `init`/`step`だけ書く |

前例: Houdiniのsim cache、Blenderのpoint cache/Alembicベイク、ゲームエンジンの決定論リプレイ — 「逐次シミュレーションはホストがベイク管理する」のが業界の収束点であり、本設計はそれをプラグイン境界として最初から定義するだけ。**Natron式に「なんでもノードで自由に」を無契約で許した結果のキャッシュ/並行地獄**(F-2)の対極。

## 5. メモリ・コスト試算(成立性の確認)

スナップショットサイズの代表値(5分@30fps=9,000フレーム、チェックポイント間隔30フレーム=300個と仮定):

| シミュレーション | 状態サイズ/スナップショット | 300個合計 | 判定 |
|---|---|---|---|
| パーティクル10万個(pos+vel+age+seed=32B/個) | 3.2MB | ~1GB | ディスク階層必須(P4)。RAM常駐は直近数個 |
| 布 100×100質点(pos+vel+prev=48B/頂点) | 0.5MB | 144MB | RAMで余裕 |
| 2D流体 512²グリッド(vel+density=16B/セル) | 4.2MB | 1.3GB | ディスク階層必須 |
| 3D流体 128³グリッド(16B/セル) | 33MB | 10GB | 間隔を伸ばす(例: 2秒)+ディスク、または解像度で予算内に収める |

- 4K RGBAフレーム1枚=33MB(memory-model)と比べると、**シミュレーション状態はピクセルよりだいたい安い**。ベイク済みピクセル(グループ仮出力)を持てる設計なら状態列も持てる
- `state_budget_bytes`宣言により、ホストはチェックポイント間隔・退避階層を予算から自動決定できる
- **step計算コスト**: 10万パーティクルのオイラー積分はGPUで<0.1ms、1080pレンダ1フレーム(数ms〜)より桁で小さい。布・2D流体も同様。3D流体の高解像度だけが例外で、それは間隔とDraft品質(粗グリッド)で吸収する

## 6. レベル2の設計: 宣言的時間窓(前後フレーム参照)

シミュレーションとは別に、**状態を持たないが前後フレームを読む**エフェクト(エコー/残像・フレームブレンド・タイムディスプレイス・スーパーサンプリング型モーションブラー)がある。これらは`出力(t) = f(入力(t−k..t+k), params)`という**tの純関数のまま**であり、決定論もフレーム並列も壊さない。壊れるのはキャッシュの依存範囲が「点」から「窓」になることだけ。

- `NodeDesc`に**静的な時間フットプリント宣言**を予約する(凍結ゲート):

```rust
pub struct TemporalFootprint {
    /// 上流入力を追加で要求する範囲(フレーム数)。デフォルト(0,0)
    pub frames_before: u32,
    pub frames_after: u32,
    /// サブフレームサンプル要求(モーションブラー用)。Quality::effect_samplesが上限
    pub subframe_samples: u32,
}
```

- **AE式の「任意時刻をコールバックで引く」APIは採らない**。静的宣言だからこそスケジューラが依存を事前解析でき(prefetch・並列維持)、キャッシュキーが「ノード×時間**窓**×パラメータ」へ自然拡張されるだけで済む。任意アクセスを許すと依存が実行時までわからず、Natron型のキャッシュ地獄に直行する
- 窓は出力時刻系で宣言し、ソース参照は**必ずTimeMapを通して**写像する(F-4の構造が受け皿として既にある)
- **モーションブラー決定(concept)との整合**: 「プラグイン領域+Quality型にサンプル数の口」という既決定は、この時間窓宣言で初めて実装手段を得る(現行traitではプラグインがサブフレームサンプルを入手できず、決定とAPIが噛み合っていなかった)。なおベクター型モーションブラー(パラメータの時間微分から速度を得る)ならテクスチャ窓ゼロで実装できる — キーフレームトラックはtの純関数なので、ホストがt±εの評価値を渡すのはほぼ無料
- VRAMコスト: 窓×上流テクスチャ1枚(1080p fp16≈8MB)。窓3で+24MB/ノード — memory-modelの予算管理対象として明記

## 7. 恒久的に禁止のままにするもの(自由にしない部分とその理由)

1. **render系trait(`Filter`/`LayerSource`/`Composite`)の`&self`に状態を溜める** — §4の表の通り、キャッシュ・並列・シークで必ずサイレントに壊れる。状態が要るならレベル1/3へ
2. **レンダ経路での自己出力フィードバック**(自分の前フレーム*出力*を入力に取るFilter) — レンダを直列化しキャッシュを殺す。フィードバック的な絵(残像蓄積等)はレベル2(上流の窓)かレベル3(Simulationとして状態を積む)で表現する
3. **非決定入力**(壁時計・スレッドID・未seed乱数・環境依存値) — 決定論の破壊。乱数は必ずDocument由来のseedから
4. **可変タイムステップ**(プレビューfpsにdtを追従させる等) — プレビューと書き出しで結果が変わる=B-4違反の時間軸版

## 8. 段階導入(何をいつやるか)

| 段階 | 内容 | 根拠/接続 |
|---|---|---|
| **今(凍結ゲート)** | `PluginKind::Simulation`をenum予約(**実装済み 2026-07-10**)。`TemporalFootprint`のフィールドセットとSimulationPlugin traitの口を凍結項目に追加。ドキュメントスキーマにシミュレーションノードの席(プラグインID+seed+パラメータ)を予約。M4-K1のキー設計がStateTrackを第3のクライアントとして受けられることを確認 | 後付けの境界変更(F-7と同型の痛み)を回避 |

## 9. 未決事項(凍結ゲート/実装時に判断)

- `StateSnapshot`のフォーマット(opaque bytes+version で足りるか、圧縮を挟むか)
- チェックポイント間隔の決め方(固定1秒か、`state_budget_bytes`と予算からホストが自動調整か)
- シミュレーションノードの入力に他レイヤーのDataTrack(例: 音楽解析)を許すか(方向: 許す。DataTrackはtの純関数なので決定論を壊さない)
- **レイヤー間相互作用(衝突相手に他レイヤーの形状を使う等)はv1.xではやらない**。シミュレーションの入力は自パラメータ+DataTrackのみ。相互作用する複数シム(シムグラフ)はv2以降の拡張として席だけ意識しておく
- サブフレームシムレート(音楽同期でsim_rateを上げる)の要否
