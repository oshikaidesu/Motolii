# 自作動画編集ツール 設計メモ

作成日: 2026-07-05

## 背景・動機

- After Effectsの重さ、Cavalryのピクセル操作の遅さに限界を感じている
- Alight Motionの軽快な動作の仕組みに興味
- MV(ミュージックビデオ)制作用途で、実用に耐える軽さのツールを自作したい
- Traceryプラグイン(aescripts.com)のような「映像の色トラッキング解析→ジェネレーティブなオーバーレイ生成」の仕組みが好み
- 3D合成、OBJ読み込み、動画合成も行いたい

---

## 1. 各ツールが重い/軽い理由(推測)

| ツール | 特徴 | 理由(推測) |
|---|---|---|
| After Effects | 全体的に重い | 30年近い歴史のレイヤーコンポジット型。CPUベースの旧プラグインAPIを引きずり、GPUアクセラレーションは後付け。RAMプレビュー(全フレームキャッシュ)でしのいでいる |
| Cavalry | ベクターは速いがピクセル操作は遅い | ベクター/パラメトリック図形生成に最適化されたエンジン。ラスター処理は本来の得意分野ではない |
| Alight Motion | 軽い | モバイルネイティブ(C++系)、ベクターファースト、GPU合成、差分更新的設計と推測。Skiaのような軽量GPU 2Dベクターレンダリングライブラリ、または同種の自社エンジンを使っている可能性が高い(**注: 内部実装は非公開のため、これは確証のない推測**) |

**結論**: 軽さの正体は「ベクター中心の表現 + GPU合成 + 差分更新」の組み合わせと考えられる。

---

## 2. 要件から導かれる技術的な必然

- 3Dメッシュ(OBJ)は最終的にラスタライズ(ピクセル変換)が必要
- 動画はそもそもフレームごとのピクセル配列
- この2つを合成する以上、**最終的にピクセルベースの合成にならざるを得ない**
- ベクター表現(paper.js的なもの)は「素材の一部」を担う層として使えるが、コンポジット全体のコアエンジンにはならない

### パイプライン全体像

```
Tracery的な生成ルール(何を生成するか)
    ↓
① 解析層(ピクセルベース): OpenCV.js等で色トラッキング・オプティカルフロー
    ↓
② 生成・合成層: トラッキングデータをパラメータ化し、3D/2Dオーバーレイ生成
    ↓ three.js / wgpu などでレンダリング、OBJ/glTF読み込み、動画テクスチャ合成
③ 最終処理(ポストプロセス): 色調整・ブラー・グレイン等
    ↓
書き出し(フレーム単位の確定的レンダリング)
```

- ③(ブラー等の最終処理)は**後回しでよい/MVPには無くてもよい**。①②の正しさの検証には影響しない独立ステージ
- OBJよりglTFの方が将来的な拡張性(PBR、アニメーション、ボーン)で有利。OBJは最低限の入口として残しつつ、内部的にはglTF変換パスを用意しておくと良い

## 3. レンダリング技術の候補と評価

| 技術 | 用途 | 評価 |
|---|---|---|
| paper.js | ベクターパス編集 | Canvas2Dベースで高レベルAPI。プロトタイプの初速には向くが、レイヤー数が増えると重くなる。WebGLレンダラーなし |
| p5.js | 生成的グラフィック | デフォルトCanvas2D。WebGLモードもあるが高レベルAPIとの相性が悪い |
| PixiJS | 2D WebGLレンダラー | スプライトバッチングが強く実用速度が出る |
| Konva.js | Canvasシーングラフ | レイヤーキャッシュ機構あり、GUI編集機能(選択・変形等)が最初から揃っている |
| three.js | 3D統合 | OBJLoader/GLTFLoader標準搭載、EffectComposerでポストプロセスをnode的に構築可能。r171以降WebGPURenderer搭載(自動WebGLフォールバックあり) |
| wgpu(Rust) | ネイティブGPU描画 | ブラウザを介さずVulkan/Metal/DirectX12に直接マッピング。Tauri構成の要 |
| OpenCV.js / opencv4nodejs | 色トラッキング・解析 | WASM版(ブラウザ内)かネイティブバインディング(Electron/Tauri内)かで速度が変わる |

### 動画フレームの取り扱い

- ブラウザ内実行なら **WebCodecs API** でフレーム単位の`VideoFrame`を取得
- デスクトップアプリ(Electron/Tauri)ならネイティブffmpegを直接叩ける
- プレビュー(リアルタイムループ)と書き出し(フレーム単位の決定的レンダリング)は設計として最初から分離する(Remotion方式)

## 4. アプリ基盤の検討: Electron vs Tauri(Rust)

| 項目 | Electron | Tauri |
|---|---|---|
| レンダリングエンジン | フルChromium同梱、OS間で一貫 | OS標準WebView(Windows=WebView2/Chromium系、macOS=WKWebView/Safari系、Linux=WebKitGTK) |
| バイナリサイズ・メモリ | 大きい | 軽量 |
| WebGPUの扱い | Chromium基準で安定 | OSごとに実装が変わる(macOSはSafariのWebGPU実装に依存) |
| ネイティブ処理との親和性 | Node.js経由 | Rustネイティブ、wgpu直結 |

### 重要な整理: WebViewを介さない設計にすれば懸念は解消する

Tauriのバックエンドは基本Rustプロセスであるため、**映像処理・3D合成・トラッキングの本体をwgpu(Rustのネイティブ GPUライブラリ)で直接実装すればWebViewを一切経由しない**。wgpuはブラウザのJS APIとしてのWebGPUとは別レイヤーのネイティブ実装であり、Vulkan/Metal/DirectX12に直接マッピングする。

```
WebView(HTML/CSS/JS) = タイムラインUI・パネル・ボタンなど操作系のみ
Rustプロセス(wgpuでネイティブ描画) = 映像デコード・トラッキング・3D合成・書き出し本体
    ↓ 描画結果を別サーフェスとして重ねる、またはIPC経由でプレビュー領域に渡す
```

これによって「OSごとにWebViewのレンダリングエンジンが違う」という問題が実質的に無関係になる。**Figma(UIはWeb技術、レンダリングコアはC++/WASM)やVS Codeに近い設計思想**であり、Alight Motionの軽さの源泉(ネイティブエンジン)にも近づける方向性。

**結論**: MV制作のような重い処理・安定性重視の用途では、Tauri + Rust + wgpuネイティブ構成が有力な選択肢。

## 5. キャッシュ設計の方針: AE方式を踏襲しない

AEのRAMプレビュー(フレームを丸ごとキャッシュする方式)は、**レンダラー自体が遅いことの代償行為**であり、模範にすべき設計ではない。

### 代替案: ノード単位の依存関係キャッシュ(Nuke/Houdini/Blender方式)

| 処理 | キャッシュすべきか |
|---|---|
| 動画デコード結果 | ○ キャッシュ推奨(圧縮解凍を毎回やり直したくない) |
| トラッキング解析結果(色トラッキング、オプティカルフロー) | ○ キャッシュ必須(時系列全体の計算のため) |
| 3D合成・procedural生成 | △ GPUでリアルタイム再計算できるなら基本不要 |
| 最終ポスト処理(ブラー等) | △ 軽ければ都度計算、重ければパラメータ変更のないフレームのみキャッシュ |

「重い/毎回同じ結果になる処理だけを選択的にキャッシュし、変更があった部分だけ再計算する」**ダーティフラグ管理**が、AEの全キャッシュ依存方式よりも軽量かつスクラブ体験も良くなると考えられる。

## 6. WebGPUの現状(2026年半ば時点)

- 2025年11月25日以降、Chrome/Edge/Firefox/Safariが揃ってデフォルトでWebGPUを出荷
- Three.jsはr171以降`WebGPURenderer`を標準搭載、WebGL2への自動フォールバックあり
- Chrome/Edge: 2023年(v113)から安定、最も枯れている
- Safari: Safari 26.0からmacOS Tahoe 26/iOS 26等でデフォルト有効
- Firefox: 2026年半ば時点でも情報源によって状況の記述が分かれる(Windows/macOS ARM64では有効化済みとする情報源と、フィンガープリンティング懸念からデフォルト無効とする情報源が混在)

**Tauri+Rust+wgpuネイティブ構成であれば、上記のブラウザ間のばらつきは前提として無関係になる**(WebViewを描画に使わないため)。

---

## 7. 実装の進め方(提案)

1. Rust + wgpuの最小構成で「動画1本を読み込んでトラッキングなしで再生できる」プロトタイプを作成
2. 色トラッキング解析層(OpenCV.jsまたはネイティブOpenCVバインディング)を追加し、ボトルネックを実測
3. トラッキングデータを使った procedural なオーバーレイ生成・3D合成(OBJ/glTF)を追加
4. ノード単位キャッシュ + ダーティフラグ管理を実装
5. 最後にポストプロセス(ブラー・色調整等)を追加
6. プレビュー(リアルタイム)と書き出し(フレーム単位の確定的レンダリング)を分離して実装

## 技術スタックの決定

> **改訂 (2026-07-07)**: 当初案(WebViewがUI操作系を担い、描画本体だけをRust/wgpuでネイティブ化する Tauri 構成)を破棄し、**完全ネイティブアプリケーション**へ方針変更した。以下が確定方針。

- **Web版は作らない。** OSネイティブのデスクトップアプリ単一ターゲットとする。
- **UIも描画も同一ネイティブプロセス内**で完結させ、**wgpu(Rustクレート)を直接使用**する。UI層にWebView(Tauri/Electron)を使わない。
- **理由**: WebView UI ↔ ネイティブ描画層を分離すると、ブラウザWebGPU(JS API)やIPC/共有テクスチャを介したUI↔GPU間の入出力遅延・同期コストが生じる。**この経路自体を持たなければ遅延は発生しない。** レンダ結果を別プロセス/別コンテキストへ渡す必要がなくなる。
- **UIフレームワーク**: **GPUI(Zed製)を採用**(2026-07-07決定)。コンポーネントは `gpui-component`(shadcn/ui由来)を土台にする。Zed が本番運用し OpenCutデスクトップも採用する**確立された前例がある**ことを重視した(pre-alpha の Blitz/Dioxus Native はコアUIに使わない)。詳細と選定理由は `priority-tasks.md` P0-3 参照。
- 映像デコード・トラッキング解析・2D/3D合成・書き出しは、従来通りRustプロセス内でwgpuを直接使いネイティブ描画する。

この方針変更により、旧 §4(Electron vs Tauri)および discussion-log §5-3(UI表層はWeb技術寄せでよい)の「WebViewでUIを担う」前提は無効となる。wgpu の技術的位置づけ(Vulkan/Metal/DirectX12 直マッピング、ブラウザ非経由)自体は §4 の分析どおり維持する。

リポジトリ名 `p5.opencuts` は初期の仮称であり、p5.js/ブラウザ内レンダリングを採用する計画ではない(本方針変更でその可能性は完全に排除)。
