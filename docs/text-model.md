# テキストモデル: スタイルスパンとアニメーター(設計ドラフト)

ステータス: **ドラフト**(2026-07-12起草。凍結ゲート未通過 — 公約化しない)

対象: 「文字ごとのフォント/サイズ変更」「文字ごとのアニメーション」「縦書き」を、P6契約([M5-P6](specs/M5-3d-and-post.md)、2026-07-10決定のラン単位純関数3点)とF-6分界(組版はプラグイン)の上でどう成立させるかの設計。既存決定への追加はすべて「提案」と明記する。縦書きは調査メモ+反対側レビュー待ちのため**予約のみ**置く([レビュー規律6点](reviews/README.md))。

## 1. 背骨: スタイルとアニメーションの二層分離

**組版に影響する属性(スタイル)と、組版に影響しない毎フレーム変形(アニメーター)を、データモデルの段階から別の物として持つ。**

| 層 | 何が変わる | 再シェーピング/リフロー | 例 |
|---|---|---|---|
| スタイルスパン | ラン分割・グリフ選択・送り幅・行組 | する | 文字範囲のフォント/サイズ/字面色 |
| アニメーター | `glyph_transform`・不透明度のみ | **しない** | 文字が跳ねる・回る・湧き出す・ワイプ |

根拠:

- AE Text Animatorの先例と整合: アニメーターのScaleは組版(行送り・折返し)を変えない。「文字が踊っても行組は崩れない」はユーザーの期待とも一致する
- 性能: リフロー(itemize→shape→行組)は変形合成より桁で重い。アニメ経路をshape結果の下流に置けば、毎フレームの仕事はグリフ数×行列合成で済む
- 例外は決定済みのバリアブル軸アニメ(P6): 軸値はシェーピング入力なので毎フレーム再shape(µs級・キャッシュしない)が正規。二層分離の反例ではなく「軸だけはスタイル層がアニメ可能」という限定例外として扱う

## 2. データモデル(M2スキーマへの追加候補 — 提案)

テキストレイヤー(歌詞プラグイン第1号)のパラメータとしてDocumentに入る「レシピ」([concept](concept.md) 2026-07-09の文書/キャッシュ分離に従い、シェーピング結果やアトラスは入れない):

- **content**: 文字列。キーフレーム化はホールド補間のみ(AE Source Textの先例)
- **style_spans**: `[{range, font, size_ratio, fill, 軸値, …}]` — rangeの境界は**クラスタ境界に丸める**(§2.1)。itemize後のラン再分割の入力
- **animators**: `[{selector, properties}]`
  - **selector**: `{unit: クラスタ|単語|行, start, end, offset, shape: 矩形|ランプ|三角, seed}` — start/end/offsetは既存**ParamSource(f32)**。ここが本設計の急所: セレクタを普通のパラメータにすることで、キーフレーム・AM式イージング(Bounce/Elastic含む、2026-07-10決定)・DataTrack駆動(音量連動)・将来のParamDriverが**全部無償で乗る**。カラオケワイプは「offsetを時間駆動するだけ」に畳み込まれる(M5実装ガード10)
  - **properties**: `{position: Vec2, rotation, scale, opacity, 軸Δ}` — 各値もParamSource。セレクタ重みwで補間してグリフへ合成

### 2.2 複数アニメーターの合成規則(仮決定 2026-07-12)

**並び順に逐次適用+プロパティごとの加算合成(上書きしない)。合成モードのユーザー露出はしない(v2判断)。**

- 根拠: AM実機確認(2026-07-12、ユーザー本人 — トランスフォーム系エフェクトは純粋加算で上書き処理なし)。AEも並び順の相対変換スタック(≒加算的)、Riveは加算を含むモード露出 — 3先例が加算側で一致。口コミ調査(同日)でも合成順序自体への苦情は見つからず、AEで嫌われているのは**順序依存が暗黙**なこと(公式ヘルプが「打ち消し合ったらドラッグで並べ替えろ」と書く水準)とアンカー既定(§6)
- したがってUIは並び順=適用順を明示する(暗黙の打ち消しを作らない)
- グリフ変形はレイヤー変形の**内側**の座標階層 — 「文字別の細かい動き+テキストブロック全体の大きな動き」は構造的に共存する(AMで確認した気持ちよさの正体)
- scaleも含め**全プロパティ一律加算**(AM実機確認 2026-07-12、ユーザー本人)。プロパティ種別ごとの例外規則(スケールだけ乗算等)は作らない — ルールが1枚なら実装・説明・テストが全部単純になる

新しいアニメーション機構は**作らない**。アニメーターは「ParamSourceの束+グリフへの適用規則」であり、評価器(motolii-eval)の契約は不変。

### 2.1 単位はクラスタ

セレクタとスパン境界の単位は文字(char)でもバイトでもなく**クラスタ**(P6クラスタ対応表の単位)。結合文字・異体字セレクタ・絵文字ZWJ列で壊れない。P6が「クラスタ対応表は出力必須」と決めた理由の実地検証になる。

## 3. 毎フレーム評価パイプライン

1. **itemize + fallback**(コア)— style_spans境界でランをさらに分割
2. **shape**(コア)— ランごと。方向・言語・軸値(t)を入力、クラスタ対応表を得る
3. **行組**(プラグイン)— UAX#14改行・整列。**スタイル層だけから計算する(アニメーターの影響を受けない)**
4. **セレクタ評価**(プラグイン)— クラスタindex → 重みw∈[0,1]
5. **変形合成**(プラグイン)— properties(t)×w をグリフごとの`glyph_transform`へ
6. **draw_glyphs**(コア)— `normalized_coords` / `font_size` / `glyph_transform`貫通

キャッシュ: 1〜3は「テキスト+スタイル+軸値」の純関数で、軸アニメが無ければ区間で安定(M4)。4〜6は毎フレームだが軽い。P6の「軸アニメ経路はシェーピング結果をキャッシュしない」と整合。

## 4. ユーザー要求との対応

| 要求 | 実現 | 経路 |
|---|---|---|
| 文字ごとフォント/サイズ | style_spans | 再シェーピング(1–3) |
| 文字ごとアニメーション | animators | 変形のみ(4–6) |
| 「文字が大きくなる」演出 | animators.scale | 変形のみ — 行組不変(AE先例) |
| カラオケワイプ | selector.offsetを時間/DataTrack駆動 | 変形のみ |
| 縦書き | 行組段の方向差し替え(v1延期) | §5 |

「見かけのサイズ」(アニメ)と「組版のサイズ」(スタイル)は別物 — この区別はUI文言にも反映が要る(未決、M3)。

## 5. 縦書き(判定済み 2026-07-12: v1延期+契約文言のみ先行)

先例調査メモと反対側レビューの併読を経て判定(規律6点準拠、最終判定はユーザー 2026-07-12):

- **v1は横書きのみ(縦書き延期)**。ガード9(回転による簡易縦書き禁止)据え置き — 回転ベースの反面事例2件(resvg、Flutter公式回答)が調査で追加確認された
- **P6契約には費用≈0の文言3点のみ先行反映**: 出力の方向中立化(C-1)・OpenTypeフィーチャ透過口(C-2)・ラン再分割自由の明文化(C-3)。反映先: [M5-P6](specs/M5-3d-and-post.md)
- 縦書き着手時に回収: 縦行送りメトリクス口(C-4)・縦メトリクス欠落診断(C-5)。C-6(sideways描画)は棄却=現契約の`glyph_transform`貫通で既に可能
- 「方向は行組段(プラグイン)が所有」の分界は、調査で同型の先例と整合(Blink/Gecko、parley#634の再設計提案)。アニメーターがグリフ座標系で動く本設計は、縦書き導入後も段4–6が原理的に不変
- 併読: [調査メモ](reviews/2026-07-12-vertical-text-prior-art.md)・[反対側レビュー](reviews/2026-07-12-vertical-text-prior-art-counter-review.md)
