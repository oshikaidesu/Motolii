# After Effects ユーザー不満の体系化と、我々の立場

作成日: 2026-07-09(ユーザーコミュニティ調査に基づく)

## 目的

AEユーザーが実際に何に不満かをコミュニティ(Adobe公式フォーラム/Reddit/motion系ブログ/競合比較記事)から体系的に集め、**我々のツールがどう解決するか**をタグで分類する。「プラグインで解決できるものは後回し」というスコープ判断のための原本。

## 解決タグ

- `[コア解決]`: 我々のアーキテクチャが**すでに構造的に解決**している(設計決定済み)。
- `[プラグイン→後回し]`: 我々の**プラグイン模型で解決可能**。ユーザー方針により**後回し**(コアを軽く保つ)。
- `[予約済]`: 評価モデル/スキーマに**口だけ予約**済み、実装は後(F-7デュプリケータ等)。
- `[スコープ外]`: concept.mdの「ではないもの」に該当。意図的にやらない。
- `[コア要検討]`: コアのUX/機能として**別途決めるべき**(主にM3)。

## 痛点カタログ

### A. 性能・安定性(現時点で最も声が大きい)
出典: Adobe公式フォーラム(AE 2026)— 頻繁なクラッシュ、**メモリリーク(Undoがメモリ失敗→クラッシュ)**、UIマイクロフリーズ、2〜3時間使うと全体が重くなりTask Managerで強制終了、128GB RAMでも "out of memory"、リアルタイムプレビュー無し。

| 痛点 | 我々の立場 | 根拠/関連 |
|---|---|---|
| 全体的に重い・プレビューが遅い | `[コア解決]` | VRAM常駐+GPUネイティブ+メモリ帯域モデル(performance-model.md) |
| 長時間でメモリリーク/クラッシュ | `[コア解決]` | 単一writer+不変スナップショット、明示的メモリ予算/LRU(memory-model.md, F-2)。コマンドジャーナルで落ちても作業を失わない |
| Undoでメモリ失敗 | `[コア解決]` | Undo=コマンド差分(状態を複製しない)。M2-D2 |
| **根本原因**: 旧世代CPU/RAMキャッシュ架構。**プラグイン窓口とは無関係**。→ 我々の"軽さ"の中核的差別化 | | |

### B. モーショングラフィックのネイティブ機能欠如 → プラグイン/式に強制
出典: 「AE: A Legacy Tool in a Modern World」(Adobe Ideas)、Advanced Duplicator要望、UX in Motion feature requests。

| 痛点 | 我々の立場 | 根拠/関連 |
|---|---|---|
| モダンイージング(バウンス/イラスティック/オーバーシュート)が無く、Flow/式頼み | `[コア解決]` | イージングはネイティブ。Flow/AM式の区間イージングエディタを採用(concept決定、`Interp::Bezier`、y非クランプでオーバーシュート可)。AM式の高度イージング型(Bounce/Elastic/Steps)も`Interp` variantとして採用(2026-07-10、concept.md)— AEで式必須だった物理風モーションをGUI選択肢化 |
| グラフエディタが古く技術者向け | `[コア解決]` | AE式グラフビューを作らず区間イージングポップアップに置換(同上) |
| デュプリケータ/エフェクタ/インスタンシング(Cavalry MoGraph相当)が無い | `[予約済]`→将来 | 評価モデルにインスタンスインデックスを予約(F-7)。実装はコア確立後 |
| **実用パーティクルが標準に無く、Particular/Stardust等の高価な外部プラグイン任せ**(標準のCC Particle Worldは化石) | `[コア解決(標準搭載)]` | ファーストパーティパーティクルを同梱(2026-07-10決定、第2号=Simulation境界の実力テスト)。既定は閉形式L0でベイク不要、衝突有効でL3(StateTrack)へ自動昇格。音楽同期エミッション(BPM/DataTrack駆動)が一級要件。[simulation-model.md](simulation-model.md)§8 |
| ネイティブIK(Duik/Rubberhose相当)が無い | `[プラグイン→後回し]` | キャラリグはMV主用途外。必要ならプラグイン |
| イージングカーブのimport/exportにFlch必須 | `[コア解決]` | イージングは`cubic-bezier`4値=標準表現。入出力はその値の受け渡し |
| ノード/ビジュアルスクリプトの選択肢が無い | `[スコープ外]` | ノードグラフはユーザーに見せない(concept決定)。拡張はプラグインで得る |

### C. プラグイン/式の**authoring障壁が高い**(= ユーザー仮説の核心)
出典: 「Expression Editor Needs an Update」、theSauceCode 開発記(「AEのプラグインUIは限定的、リアルタイムプレビューや複雑なパラメータ制御に向かない」「SDKの理解に数週間」)、Autograph比較(「高レベル自動化がJS式にゲートされ、1つの構文エラーでプロジェクトが壊れる脆いパイプライン」)、AE式=JS。

| 痛点 | 我々の立場 | 根拠/関連 |
|---|---|---|
| 式=JavaScriptで難しく、1つのタイポで壊れる/エラーが不親切 | `[コア解決]` の設計思想 | 「境界のシンプルさ=書きやすさ最優先」。式に相当する汎用スクリプトはWASMパラメータプラグイン(v2)で隔離、コアは宣言的ParamSource/DataTrack |
| プラグインSDKが重い(C++、ライフサイクル理解に数週間) | `[コア解決]` の設計思想 | プラグイン境界=「テクスチャin+params→out」の単純trait。**LLMで書ける粒度**を最優先(concept決定) |
| **プラグインUIが限定的**(リアルタイム/複雑パラメータ制御に不向き) | `[コア要検討]`→設計済み | プラグインはパラメータ定義(NodeDesc)を宣言するだけで**プロパティパネルが自動生成**(M3、AEのエフェクトコントロール思想)。UIコードを書かせない |
| プラグインが高価($100-400/本)=コア欠落の穴埋め | `[コア解決]` の構造 | OSS+低障壁authoring。穴埋めを有償ベンダに依存させない |
| **根本原因**: 「プラグインの窓口が狭い」= 作る障壁が高く(C++ SDK/限定UI/式=コード)、結果として少数の高価な商用プラグインor脆い式に依存。→ **ユーザー仮説を裏付け**。我々の答え=低障壁・単純境界・自動UI・LLM前提 | | |

#### C-補遺: エクスプレッションは「不満」であると同時に**残留(リテンション)要因**(2026-07-10 追記)

一次証言(2026-07-10、AEユーザーからの直接聞き取り): 「結局AEにはエクスプレッションがあるので、(他ツールを試しても)AEに戻ってしまう」。同じ話者が同時に「一見さんお断り」「ダサい」「JSもカス」とも評している。つまり**本人も嫌っているのに、能力としては手放せない**。これは上の authoring 障壁の不満と矛盾しない: 式は書き味が最悪でも「**キーフレームを打たずにパラメータを動かす**」能力のほぼ唯一の窓口なので、この能力が無い代替ツールは(書き味がどれだけ良くても)乗り換え候補から脱落する。

含意: JS文字列式を非採用にする決定(concept 2026-07-10、GAP-8)は「書き味」への答えとしてはこの証言にも支持される(「一見さんお断り/ダサい/JSがカス」)。ただし**能力のカバレッジが揃うまでは、非採用そのものが移行障壁として働く**。幸い式の実利用は少数の定番パターンに集中しており、それぞれ宣言的な口に写像できる:

| 式の定番用途 | 体感頻度 | 我々の口 | 状態 |
|---|---|---|---|
| `wiggle()`(ランダム揺れ) | 最頻 | ParamDriverプラグイン(LFO/ノイズ)= CavalryのBehaviour相当 | 種別は分類済み(pitfallsのプラグイン種別)、実装未 |
| バウンス/オーバーシュート/慣性 | 高 | ネイティブイージング(y非クランプ`Interp::Bezier`+区間エディタ) | 決定済み |
| `loopOut()`/`loopIn()`(キーフレーム繰り返し) | 高 | **未定**。Keyframes系ParamSourceにループモード(cycle/pingpong/offset)の口が必要 | **小穴 — M2スキーマに1フィールド。要追記** |
| `time * n`(等速回転・スクロール) | 高 | `f(t)`のプロシージャルParamSource(ランプ/LFO) | ParamDriver領域 |
| 他レイヤー参照(ピックウィップ相当) | 中 | 型付きリンク LookAt/Follow/ParentRef(GAP-8) | 決定済み(2026-07-10)、実装未 |
| `index`連動(複製ごとのオフセット) | 中 | インスタンスインデックス(F-7) | 予約済み |
| 音声→パラメータ(Audio to Keyframes) | 中 | DataTrack/解析駆動(最終フェーズ) | 決定済み |
| 上記に収まらない任意の算術・条件分岐 | 少(ロングテール) | WASMパラメータプラグイン(v2)。UIは式入力欄にしない | v2に隔離 |

ロングテールの答え合わせ: 「式でしか書けない小物」をユーザー各自にJSで書かせる代わりに、**LLMにParamDriver/WASMプラグインとして書かせて配る**(本プロジェクトの中核戦略)。話者の「JSがカス/ダサい」評は、テキスト式エディタを主戦場にしない方針(型付きリンク+ターゲットピッカー、GAP-8)を支持する一次証言でもある。**採用判定: 式の「能力」はコアの宣言的な口で全数カバーし、式の「文法」だけを捨てる** — カバレッジ表の「未定」を潰し切ることが、AEへの出戻りを防ぐ条件。

## 「プラグインの窓口が狭い」仮説の検証

**結論: 仮説は"機能不満"についてはおおむね正しい。ただし現時点で最も声が大きいのは性能・安定性(コア架構)であり、別軸。**

- **プラグイン窓口起因(B/C/F)**: モーショングラフィック機能の欠如(B)+authoring障壁(C)+データ自動化(F)は、いずれも「ネイティブに無く、かつ自分で足す窓口(SDK/式/プラグインUI)が狭い・高価・脆い」ことに帰着する。**AEの機能不満の大きな塊(体感2/3)はここ**。Autograph(ビジュアルGenerators/Modifiers+OFX)・Cavalry(手続きBehaviors/Duplicators)という競合の解も、まさに「窓口を広げる/コードを不要にする」方向で、仮説を支持する。
- **窓口とは無関係な軸(A/D/E/G/H)**: 性能・安定性(A、旧世代エンジン)、価格(D、ビジネスモデル)、学習曲線(E)、Web/インタラクティブ(G)、重3D(H)は、プラグイン窓口では解決しない。**特にAは現在の最大の声**で、これは我々の"VRAM常駐GPUネイティブ"というコア架構が別途答える。

### 我々の二枚看板(この分析からの再確認)
1. **軽さ・安定性 = コア架構**(A を構造的に解決): VRAM常駐/GPUネイティブ/単一writer/コマンドジャーナル。
2. **広く低障壁なプラグイン窓口**(B/C/F を解決): 単純テクスチャ境界+自動パネル生成+LLM前提+ネイティブイージング。**AEが狭くした窓口を、あえて広く簡単にする**のが差別化の второй軸。

## アクション(スコープへの反映)

- `[コア解決]`/`[コア要検討]`は既存の決定・仕様に紐づく(perf-model / concept / M2 / M3)。新規タスクは不要か、既存M2/M3に含まれる。
- **C-補遺のカバレッジ表で唯一の「未定」= キーフレームのループモード(`loopOut`相当)**。M2のKeyframes系ParamSourceスキーマにループ指定(cycle/pingpong/offset)を1フィールド追加する形でM2仕様に反映する(式カバレッジの穴を残さない)。
- `[プラグイン→後回し]`/`[予約済]`(IK、デュプリケータ、データ自動化、解析駆動)は**コア完成後**。評価モデルの口(F-7)と解析(最終フェーズ)は予約済みなので後付けで詰まらない。
- `[スコープ外]`(価格n/a、ノードUI、Lottie/インタラクティブ、重3D)は「やらないことリスト」として維持し、スコープ膨張(D-4)を防ぐ。
