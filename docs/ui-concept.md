## 0. なぜこの文書か

現行のUI関連文書は、予防側が厚い。M3実装ガード(11項目)、先行ツールの苦情監査、境界汚染の予防、拒否条件 — いずれも「M3後半に発覚すると設計が覆る」失敗を先に塞ぐ文書であり、正しく機能している。

一方で、**「Motoliiの画面を初めて開いた人に何が起きてほしいか」を肯定形で語る1枚が無い**。禁止リストは画面を描く手を止められるが、描かせることはできない。散在する肯定的決定(楽曲基準線、区間イージング、意図単位のエフェクト、高密度一覧、軽さの数値目標)は、それぞれ別の文書の別の節にあり、束ねる体験像を持たない。

本書はその欠落を埋める。**新しい決定はほぼ無い** — 既にある決定を1つの物語に配置し、物語が実装可能であることを決定への参照で示し、体験として測れる審判の問いを付ける。

## 1. 一文

**Motoliiの画面は、1曲を演奏するための譜面台である。**

[VISION](../VISION.ja.md)は映像表現を「演奏・再利用・配布できる単位」にすると定めた。そのUIへの翻訳がこれだ。譜面台の上には楽曲(譜面=波形+BPMグリッド)が常に置かれ、制作者は素材と表現をその譜面に沿って配置し、演奏する。画面のすべての領域は「この1曲を完成させる」ことへ奉仕し、汎用編集ソフトの顔をした万能デスクにはしない。

5本の柱に展開する(§3):

| 柱 | 一言 | 主な既存決定 |
|---|---|---|
| 1 | 楽曲が背骨 | 音声=楽曲1本、BPM手動入力+グリッド、波形基準線 |
| 2 | 外殻は既知、因果は見える | 業界標準の操作を既定に(ガード5)、型付きリンク |
| 3 | 語彙は意図 | エフェクトは意図単位(F-8)、区間イージング=性格の名前 |
| 4 | 密度は資産、装飾は負債 | 高密度一覧、意味色+形、装飾gradient禁止 |
| 5 | 軽さは機能 | 起動・アイドルメモリの数値目標(ガード10) |

## 2. 体験の北極星: First Beat

初回起動から、**最初のビート同期モーションが音と一緒に動く瞬間**までを「First Beat」と呼ぶ。この60秒級のシーケンスを、教材なしで完走できることを体験の北極星とする。

1. **起動する** — 数秒で編集可能になる(ガード10。数値はU1実測で固定)
2. **楽曲をドロップする** — 波形がタイムライン最上段の基準線として固定表示される。BPMを打ち込むとビートグリッドが立ち、ルーラーが小節.拍を刻む(音声=1本の決定が、波形を「トラックの一つ」ではなく「定規」にすることを可能にする)
3. **ロゴ(SVG)をドロップする** — クリップがビートにスナップして置かれる(U6/U7)
4. **2箇所で位置のキーを打ち、区間を選ぶ** — ボタンが現れ、イージングポップアップで「バウンス」を選ぶ(区間イージング決定、`Interp`のパラメトリック補間型)
5. **Spaceを押す** — 音声デバイスクロックを主として、音と同期して再生される(D5トランスポート決定)

このシーケンスの価値は2つある。

- **実装可能性の証明**: 各ステップは既存決定だけで成立する。新しい契約を1つも要求しない北極星であること自体が、コンセプトが空想でない証拠になる
- **測れる体験**: 「初見でFirst Beatまで何分・何操作・何回の迷いか」は固定fixtureで人間審判できる。既存の5秒テスト(所在認知)が静的な審判であるのに対し、First Beatは動的な審判 — 両方をG0系の人間審判に置く候補とする(採否と数値はレビューで決める。本書では固定しない)

## 3. 五本柱

### 柱1: 楽曲が背骨(Song as Spine)

**波形+BPMグリッドは、トラックの一つではなく定規である。** タイムライン最上段に常設・ロックされ、クリップ・キーフレーム・(将来の)音楽同期エミッションのすべてがそこへスナップする。時間表示はタイムコードと小節.拍が並存する。

- **先例**: DAWのbeat-time ruler+グリッド。Abletonはルーラーが小節-拍-16分を刻み、グリッド吸着・密度変更が編集の土台になっている([Arrangement View — Ableton Reference Manual](https://www.ableton.com/en/manual/arrangement-view/))。映像側の需要の証拠はCapCutのBeat Sync — 音楽のビートへカット・トランジションを合わせる機能が主力AI機能として公式に押し出されている([CapCut Beat Sync](https://www.capcut.com/explore/beat-sync))
- **差別化**: 汎用NLE/AEは音声を「トラックの一つ」として扱うため、波形は編集対象の海に沈む。Motoliiは音声=楽曲1本の決定(concept.md)により、波形を定規に昇格できる。これは機能の追加ではなく**制約の配当**である
- **CapCutとの分岐**: 自動ビート検出はやらない(ビート検出をやらない決定済み)。MVの作り手はBPMを知っている。手動BPM+Tempo/Meter独立(2026-07-13採択)が正確で軽い
- **審判の問い**: 楽曲未設定の空プロジェクトでも、基準線領域が「ここに楽曲が来る」ことを示しているか。波形領域を非表示・並べ替え・削除できてしまわないか。ビートスナップの有効/無効が1キーで切り替わり、状態が見えるか

### 柱2: 外殻は既知、因果は見える

**見た瞬間は知っている制作ソフトに見え、触ると従来より因果関係が分かる。** 配置(左=素材、中央=ステージ、右=インスペクタ、下=タイムライン)、選択→詳細、drag&drop、Space再生、Undo — 学習済みの語彙は変えない。革新の予算は新しいパネル配置や専用ジェスチャではなく、従来隠れていたもの — 何に効くか(scope)、どの順で効くか(評価順)、値がどこから来たか(由来)、なぜできないか(失敗理由) — を見せることに使う。

- **先例**: Jakob's Law — ユーザーは時間の大半を他のプロダクトで過ごしており、既知の動作様式を期待する([NN/g](https://www.nngroup.com/videos/jakobs-law-internet-ux/))。FCPXのマグネティックタイムライン強制が招いた抗議(ガード5の出典)は、この法則を破った代償の実例
- **接続の可視化**: LookAt/Follow/親子/DataTrackは式ではなく型付きリンクであり(concept.md横断決定)、UIは「レイヤーをクリックしてターゲット指定」。接続中はカーソル近傍が「何を・何へ・どうなるか」を説明する
- **審判の問い**: 既存製品の経験者が、最初の10分でパネルの役割を推測できるか(5秒テストの動的版)。既存語彙から外れる新部品は「既存語彙ではこの意図を表せない」証拠を持っているか

### 柱3: 語彙は意図

**UIに出る名詞は、実装の名前ではなくユーザーの意図の名前にする。** エフェクトブラウザには「グロー」「シェイク」(意図単位、F-8)。イージングには「バウンス」「エラスティック」(動きの性格を補間型として選ぶ、AM式決定)。接続には「追従」「向ける」(型付きリンク)。原子ノードの組み立て・ベジェハンドルの数学・式のシンタックスを、標準の制作動線に出さない。

拡張機能全体の既知語彙は**プラグイン**とし、Effect stackでは**エフェクト**、生成入口では**ジェネレーター**、一時編集では**ツール**と文脈上の役割を出す。`Vism`／`.vism`は配布・開発者・ファイル詳細の語彙であり、通常の制作画面で覚えさせない。プラグインBrowserは名前だけの行一覧にせず、表現thumbnail、用途tag、導入・互換状態、同梱／local file等の由来を併用する。

- **先例(市場実証)**: FlowはAEのグラフエディタを「正規化された環境+25本のプリセットカーブ+ライブラリ」で置き換えて成立した有料プロダクト([aescripts Flow](https://aescripts.com/flow/))。「ハンドルの数学」ではなく「カーブの名前とプリセット」を求める支払い意思の直接証拠。Alight Motionのパラメトリック補間型(concept.md採択済み)も同じ方向の実証
- **裏面の規律**: 意図の名前は嘘をつかないこと。「バウンス」を選んだ結果はAdvancedで4値/パラメータとして検査できる(Simple/Advancedは同じ意味 — 単純化は隠蔽ではない)
- **審判の問い**: 新機能のUI名詞を、映像制作を知らない人に読ませて「何をしたい時に押すか」を答えられるか。逆に、名前から期待した結果とAdvancedの実値が乖離していないか

### 柱4: 密度は資産、装飾は負債

**制作ツールの画面は、情報を隠さないことが優しさである。** アセット・プレビュー・プロパティ・エフェクトスタック・ドライバ・タイムライン・トランスポートは常設領域として一覧でき、頻用値と状態を折り畳みの奥に隠さない。親しみやすさは彩度・丸み・celebration・マスコットで作らず、「対象・結果・戻し方が装飾なしで分かる」ことで作る。識別は文字を読む前に、位置・形・icon・意味色の組み合わせで成立させる。

- この柱の正本は視覚言語文書(§6)にある。本書での位置づけは「なぜ密度が優しさなのか」の一言に留める: 制作の中断は、機能が無いことよりも**探すこと**から生まれる。所在が常に同じで、状態が常に見えることは、初心者向け簡略化よりも深い意味でユーザーファーストである
- **審判の問い**: grayscale・CVDシミュレーション・motion 0でも主要状態を判別できるか。初見の人が所在を指せるか(5秒テスト)

### 柱5: 軽さは機能

**軽さは性能ページの数字ではなく、UIコンセプトの一部である。** First Beatの第1歩は起動であり、起動が遅い譜面台は開かれない。AviUtl本体が愛された第一の理由は軽さ(ガード10)であり、この層のユーザーはMotoliiの主要な迎え入れ先である。待ち時間は単に不快なのではなく、「今これを試す」を減らして創作の熱量を逃す。ゆえに軽さの最終的な価値は、同じ時間で何回見て、変えて、戻して、別案を試せるかにある。

- 起動時間・アイドルメモリ・操作応答(スクラブの体感即応、S1基準)は数値目標としてM3完了条件に入る(既定。数値はU1実測で固定)
- ドロップ→波形表示、BPM入力→グリッド描画のような「First Beat経路上の応答」は、ベンチの平均値ではなく体験の審判として個別に見る
- 平均fpsが同じでも、操作直後の停止、初回だけの長いcompile、同期readback、panelを開くたびの再生成は試行を分断する。計測は定常再生だけでなく、編集→反映→比較→Undoの往復を含める
- **審判の問い**: First Beat経路のどこかで「待たされた」と感じる箇所があるか。低スペック機(基準機はG0系で定義)でFirst Beatが完走できるか。同じ短い時間で複数案を試し、比較して戻せるか

## 4. 画面の物語

組み込み既定layout presetの各領域に、役割の一言を与える。利用者が再配置しても維持する機能の物語であり、固定座標や実装名ではない:

```text
┌─────────────────────────────────────────────┐
│ トランスポート — 演奏の手元                    │
├────────┬──────────────────────┬─────────────┤
│ 素材の棚 │ ステージ              │ 選択の意味    │
│ (左)    │ (中央: 直接操作の場)   │ (右: 由来と   │
│         │                      │  調整)       │
├────────┴──────────────────────┴─────────────┤
│ 譜面 — 楽曲(波形+拍)を背骨に、素材と表現を並べる │
├─────────────────────────────────────────────┤
│ 文脈ヘルプ — いま触っているものの説明と次の一手  │
└─────────────────────────────────────────────┘
```

- **Media Browser(左)**: Browserの`Media`面で、`All Media / Project / 複数の登録folder / Collections / Recent`を同じsource railから選ぶ。検索・preview・grid/listは共有し、同じfileはdedupeしてProject登録状態を表示する。外部候補はInboxへ受け取り、Project登録済み素材から確定配置する
- **ステージ(中央)**: 作品が見える場所であり、触る場所。選択ハンドル・ターゲット指定・接続の仮線はここに現れる。プレビュー品質の縮退(Draft)はバッジで正直に示す
- **選択の意味(右)**: いま選んでいるものが「何で・どこから値が来ていて・何が積んであるか」。変形→スタック→ドライバの縦の並びは評価の順序と一致させる。平面配置の`Position X/Y`と前後配置の`Depth Z`は別groupへ投影し、Z方向の平行移動を`Rotation Z`や暗黙の3D modeと混同させない
- **譜面(下)**: 最上段に楽曲基準線(柱1)。その下は1項目1横行も固定名列も持たない一枚の時間面で、Document項目のbarを重なり回避のために縦方向へpackingする。左端は選択一覧ではなくInboxとし、未配置素材、未解決review note、未確認background jobなど「まだ片付いていないもの」への参照だけを受け取る。Inbox右隣には従来の名前欄相当の細い無名帯アクションrailを置き、時間面のpacking帯と上下境界を一致させる。railのM/SはLane所有状態ではなく、押下時の帯上Object集合へ既存Object M/Sを一括適用する。帯高のresizeはrail各行下辺から触った帯だけに適用する。選択・hoverではInbox内容を入れ替えず、asset・note・jobの所有意味を一形式へ統合しない。処理済みは外し、空の時だけ既読管理可能なTipを一件表示できる。値はInspector、接続はArchitect、操作説明は最下端Statusへ分離する。通常Groupは別Composition／別Previewへ移動せず、同じStageと時間面で1本に畳むかchildをその場へ展開する。Group自身のM/SはGroup出力の試聴・隔離、展開はchild編集に使い、親操作からchildのM/S状態を複製しない。Automationはbar内の`◆ n`から実際に自動化済みのchannel行だけをAE型の縦一覧へ展開し、複数channelのkeyを同時選択できる。選択後はTimeline右端へ既定dockした独立Key Toolsを`KEYS / LAYERS`で排他的に切り替え、Keystone 3型のmodular操作を行う。BrowserのPreset探索とInspectorのparameter編集を隠さず、将来は共通panel systemで移動・split・tab化できる。未使用channelは同じ面から検索追加し、全parameter行やshortcut暗記を要求しない。Depth Railは現在時刻の`position.z`評価へ追従する独立操作面であり、別のDepth正本や3D modeではない。Easingはkeyではなく隣接key間の動きとして扱い、Preview直下のGraph iconをplayheadが区間内部にある時だけ点灯する。iconから開くGraph Viewで補間種別、value-time graph、Bezier handle、raw値を検査し、適用は現在区間への1 Undoとする。区間番号、key数、時刻範囲、key stripは重ねない。自動化可能というだけの空parameterは並べない。固定Laneを戻さずGroup名を読む方法と回帰審判は[譜面UI構成モデル](ui-score-model.md)を正本とする
- **文脈ヘルプ(最下)**: hover/focus中の対象について、名前・できること・ショートカット・いまできない理由を短く示す(Blenderはこの機能だけの先例)

この物語はモックv2系の構成(3-pane+高密度タイムライン+status bar)と一致する。寸法・色・icon等の具体値は本書の対象外(視覚言語とG0系審判の領分)。

## 5. 根拠の強さ

視覚言語文書と同じ三分法で明示する:

- **製品決定(既にmainの台帳にあるもの)**: 音声=楽曲1本 / BPM手動+グリッド / 区間イージング(グラフビュー非搭載) / 意図単位エフェクト(F-8) / 型付きリンク(式なし) / 高密度・装飾禁止の方向 / 軽さの数値目標化(ガード10) / 業界標準操作を既定に(ガード5)
- **先例(公式一次資料で確認)**: Abletonのbeat-time ruler+グリッド操作 / CapCut Beat Syncの存在と訴求 / FlowのグラフエディタレスなイージングUIの市場成立 / Jakob's Law(NN/g)
- **本書の仮説(反対側レビュー対象)**: 「譜面台」という体験像への統合そのもの / First Beatを動的審判として採ること / 波形基準線の非表示・削除を許さないこと / 画面の物語の役割語

## 6. 散在する正本の地図(2026-07-16時点)

UIコンセプトの関連文書は複数の未マージPRに分かれている。本書はそれらを**置換しない**。層の分担:

| 層 | 問い | 正本 | 所在 |
|---|---|---|---|
| 体験(なぜ) | 何が起きてほしいか | **本書** | 本PR |
| 操作(どう触る) | 外殻・共通文法・診断 | UI操作言語 / 操作単純化モデル | PR #176(未マージ) |
| 視覚(どう見える) | token・意味色・密度・審判 | UI視覚言語 + モックv1/v2 | PR #137 → #176で改訂(未マージ) |
| 発注(何を作る) | タスク・依存・ゲート | M3仕様 + 入場条件 + チケット分解 | main(停止中) + PR #176 |
| 実装証拠 | 成立するか | U0Vスパイク(token生成+egui shell)、タイムラインベンチ、IMEスパイク | PR #184 / #178 / #190は旧Slint案。egui証拠は[採用判断](reviews/2026-07-18-m3-egui-selection.md) |

- 未マージ文書との矛盾が見つかった場合、**操作・視覚の規範は当該文書が勝ち、体験像の意図は本書が勝つ**。どちらの改訂で解消するかをレビューで決める(docs/README「矛盾はバグ」)
- [M2基盤再締結ゲート](reviews/2026-07-15-m2-foundation-reclosure-gate.md)はmainで解除済みだが、M3製品実装は別のM3入場PRまで停止のまま。本書はドキュメントのみであり、入場PRが「入場条件の再翻訳」を行う際の体験側の入力になることを意図する

## 7. 非目標

- 新しい契約・Documentスキーマ・公開API・タスクIDを発明しない
- M3仕様のステータス・タスク表・実装ガードを変更しない(ガードは引き続き有効。本書は裏返しではなく補完)
- 数値KPI(First Beatの秒数・操作数等)を本書で固定しない — 採否と数値はG0系審判の設計時に決める
- 「譜面台」「譜面」等の物語の役割語を、そのまま製品UIの表示名にしない(表示名は柱3の審判を別途通す)

## 8. 反対側レビューで潰す質問

1. **楽曲=定規は、楽曲が決まる前の編集を阻害しないか**: 曲より先に映像の構想がある制作順のユーザーに、基準線の常設・ロックは邪魔にならないか。「楽曲未設定でも編集できるが、基準線領域は空のまま在る」で足りるか
2. **First Beatは初回体験への過剰最適化にならないか**: 60秒のデモ的成功と、5分尺×40レイヤーの本編集の密度要件(柱4)が同じ画面で両立するか。First Beatのために常設情報を削る圧力が生まれないか
3. **BPM手動入力はテンポ可変曲で破綻しないか**: Tempo/Meter独立の採択(2026-07-13)がグリッド描画・スナップのUIまで貫通しているか。テンポチェンジを持つ楽曲のFirst Beatはどうなるか
4. **「譜面台」メタファは輸出可能か**: 日本語圏外の開発者・ユーザー(README/Reddit想定)に対して、metaphorが翻訳で意味を失わないか。文書上の物語に留める限り無害か
5. **波形基準線の非表示禁止は正当か**: 実写素材の音を頼りに編集したい等のエッジケースが本当に無いか(動画内蔵音声は常にミュートの決定と整合するか)
